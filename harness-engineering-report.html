P3 Auxiliary LLM Client v3 重评：部分实现 · v4.1 澄清

📐 v4.1 概念澄清：Hermes 里 Auxiliary LLM Client（side-task router）和 Smart Model Routing（user-turn classifier）是两个独立机制：
  • Auxiliary Client = "compression / search / vision 这类 side task 用哪个 backend"，决策时机=任务发起时
  • Smart Routing = "用户这条 message 走 cheap 还是主 model"，决策时机=用户敲完 message 时
v3 报告把它们隐性归并为"模型层架构"是分类错误。本节只讲 Auxiliary Client；Smart Routing 见 GAP 4（已 CANCELLED）。

Hermes 实现

agent/auxiliary_client.py (2,771 行)

  • 独立辅助 LLM client，专用于 compression / vision / search
  • 6 级 fallback chain: per-task config overrides → primary aux → secondary aux → main model → cheap model → local
  • Credit exhaustion 自动 fallback
  • 新增 resolve_vision_provider_client() 选择视觉模型

Axiomate 现状 v3 校准

v2 标"完全没有"错了。axiomate 已有三级模型体系：

  • config.ts: currentModel / midModel / fastModel 三级 config 字段
  • Resolver 层: getCurrentModel() / getMidModel() / getFastModel() 在 utils/model/model.ts
  • 自动 fallback 链 (query.ts:255-263): main 失败时 current → mid → fast 降级；等价于 hermes "auxiliary fallback"
  • 已有内部 smart routing:
    • findRelevantMemories → midModel
    • awaySummary → fastModel
    • tokenEstimation → fastModel
    • anthropicProvider connection probe → fastModel

与 hermes 的真实差距（远小于 v2 标注）：

  • 3 级 vs 6 级 fallback（对单用户 CLI 足够）
  • 无 per-task override 配置表（硬编码哪个任务用哪个 tier）
  • 无 credit exhaustion tracking（但 errorClassifier 已能分辨 billing/auth 失败）
  • 无独立的 vision provider resolution（单用户场景未遇到）

价值: 功能重叠度 > 70%。Hermes 多出来的部分（per-task 表、6 级、credit tracking）主要服务多用户/高负载场景，对本地 CLI 边际价值低。不是 P2 gap，是 P3 完善项。

P3 Auto-Recall / Pre-Turn Memory Prefetch v4.3

Hermes 实现

plugins/memory/hindsight/__init__.py + agent/memory_manager.py

  • auto_recall 模式: 配置 auto_recall: true 时，每轮 LLM 调用之前用 user message 触发 vector + 知识图检索
  • 注入方式: 检索结果以 <memory-context> fenced block 注入 system prompt（不污染 user/assistant turn）
  • 预算控制: recall_budget: low/mid/high + recall_max_tokens: 4096 + recall_max_input_chars: 800
  • Tag 过滤: recall_tags + recall_tags_match: any/all/any_strict/all_strict
  • 检索后端: 三种模式 — cloud (Hindsight Vectorize.io) / local_embedded (本地 PG + LLM) / local_external (自建 Hindsight 实例)
  • 与 #17 Cross-Session Memory 正交: 那是存储侧（pluggable provider 架构），这是检索策略

Axiomate 现状

  • sessionMemory.ts (450 行) 是被动 file-based auto-memory（MEMORY.md + 单文件 memory）

关键洞察 (v4.4) —— 架构层 reframe：上层建筑、user-RAG、产品形态决定架构

A. Hermes 整套是 user-behavior RAG

把 hermes 拆成 RAG 范式，每一层映射很清楚：

B. 依赖层级 —— 上层建筑要基础

Layer 4  User Modeling          (USER.md / persona / 偏好画像)
         ↑
Layer 3  Active Write           (skill_manage tool + memory tool + 后台 fork-review)
         ↑                       ─── #19 skill auto-distill 在这层
Layer 2  Active Retrieval       (auto_recall 每轮前检索 → context 注入)
         ↑                       ─── #21 auto-recall 是这层
Layer 1  Storage                (SQLite WAL + FTS5 + parent_session_id 链)
                                 ─── #22 title-gen 也在这层

Hermes 因为 Layer 1-2 早就建好（SessionDB + Hindsight），Layer 3 的 skill auto-distill 才有意义：review prompt 让 LLM 决定"存/更/不存"时能 query 已有 skill 看是否重复；auto-recall 让新会话自动 surface 过去 skill，触发"我已经知道这个 → patch 它"的闭环。

Axiomate 直接 port Layer 3（v4.3 推荐的 P1 Skill Auto-Distill）会出现：① 重复造 skill 失控（没 retrieval，每次都"感觉是新模式"）② skill 老化孤儿（没 auto-recall，写完没人触发更新）③ 跨 session 学习断裂（JSONL 没法多 session query）④ memory 写入语义混乱（被动 file-based + 自主写入并发冲突）⑤ 触发时机错配（debounce queue 取 snapshot 别扭）。结论：workaround 叠 workaround，是脏实现。

C. 信息论视角 —— axiomate compact 的访问熵

软件价值 = 降低用户面对系统时的有效熵。但 compact 干的是反向 —— 把 agent 已经替你处理过的失败/学习的 entropy reduction 重新散掉。下次同问题要从原始 chaos 重走。

具体看 axiomate compact：

  • Live 上下文层：原 50 条 messages → 1 条 LLM summary，agent 后续看不到原文（只看转述版本）
  • On-disk JSONL 层：append-only，原文物理上还在 (sessionStorage.ts:1457 注释明确说 "Preserved messages exist in the JSONL with their ORIGINAL pre-compact ...")
  • 但功能上等同于丢：没工具搜、agent 拿不到、--continue 跳过 boundary、cross-session 完全不可能

结论：axiomate 是 物理保留 + 访问熵无穷，几乎是最坏的设计选择 —— 既付了存储成本，又没换可访问性。Hermes 的 parent_session_id chain + FTS 是信息论意义上更正确的实现：compact = "封顶 + 开新 session"（不是原地压），原文以独立 session 永久可查（FTS O(log N) 检索），等价于 物理保留 + 访问熵可接受。这点上 hermes 比 axiomate 设计哲学更彻底，与 Manus "保留失败痕迹" 原则更内洽。

E. 三条路径 —— Axiomate 的身份决定

路径含义工作量身份变化
NO —— 守 task tool 承认 axiomate 是无个性、按需调用、用户主导。/skillify 用户主动版就是上限。#19-22 全部 WONTFIX, identity divergence。转头做 cross-provider fallback / PTC 这类不依赖 user-RAG 的事 0（节省） 无
YES —— 走 companion 路 按 Layer 1→4 完整重塑。SQLite + FTS + chain (~2-3 周) → auto-recall (~2-3 周) → skill_manage + 后台 review (~2 周) → 可选 user profile (~1 周)。Axiomate 从 "Claude Code 衍生 CLI" 变成 "growing-with-user agent"，需要重想跨平台/cron/跨项目身份等问题 7-9 周 身份重塑

F. JSONL MVP 路径 —— 把 MAYBE 路再拆

关键洞察：只暴露 hermes 风格的 session_search(query, [filters]) tool 给 LLM，存储完全可以保持 JSONL。LLM 不知道下面是 SQLite + FTS5 还是 stream-grep + JSONL —— 工具行为一致就行。

规模JSONL 扫描SQLite + FTS5
100 sessions, 10MB50-200ms ✅10-50ms
1000 sessions, 100MB1-3s 🟡50-100ms
10000+ sessions, GB痛 🔴仍 OK

Axiomate 当前规模（单用户、项目级、个位数到几十 session）JSONL 零痛点。SQLite 是"几千 session 后才需要的升级"，不是入场门槛。

把 MAYBE 路径再拆：

  • Phase 1（~1 周）：建 session_search tool，复用现有 JSONL，简单关键词匹配 + 上下文窗口 + 时间排序。不动存储层。已经能解锁 7 个 hermes 触发点中最高杠杆的那个：LLM 自主查历史。
  • Phase 2（条件性，~2-4 周）：JSONL 扫描真痛了（实测 > 2 秒）才迁 SQLite。届时有真实 query pattern 参考，schema 反而设计得更准。
  • Phase 3（条件性，~2 周）：Phase 1 之后 LLM 已经能 query 历史，再决定是否加 skill_manage + 后台 review。届时上层才有意义。

这把原 v4.3 的 "Skill Auto-Distill 1-2 周" 错估，改写为 "Phase 1 ~1 周 MVP，剩余按需"。低风险、可中止、不绑死 axiomate 的产品身份。

推荐结论

  1. Skill Auto-Distill (v4.3 P1) 撤回独立推荐，改标 "P1 conditional on identity decision"。优先做的不是 Layer 3 而是 Layer 1。
  2. 实操起点是 Phase 1 (JSONL + session_search tool, ~1 周)。不需要先回答 NO/YES/MAYBE 的身份问题就能开始 —— Phase 1 本身在三条路径下都有价值（NO 路也能用 search 找过往）。
  3. Phase 1 ship 后再用真实数据评估：用户/agent 实际多频率调 search？skill 蒸馏空间值不值得？这时再决策走哪条路径。

关键洞察 (v4.3) —— 关键词搜索的局限

  • 4 个版本盲点的根因：v1-v4.2 评估 hermes feature 时一直依赖关键词 grep（用 distill / extract_skill / session_to_skill / pattern_extraction / auto_skill 等行业通用词），漏掉了 Skill Auto-Distillation。Hermes 实际命名是 skill_manage + nudge_interval + 后台 fork-agent，关键词完全不重合。同一逻辑还漏了 lifecycle hooks (shell_hooks) / auto-recall (auto_recall in plugin) / title-gen (title_generator) 共 4 个 pattern
  • 修复方法（已用于 v4.3）：先 web search 项目营销文档（官网 / DEV Community / Better Stack 评测）建立"功能名 ↔ 实现名"词汇表，再针对性 grep 源码核实。Web search 揭穿了"hermes 没 skill auto-distillation"的错误结论，Better Stack 文章直接说出"Every 10 turns, Hermes runs an internal review..." 这种实现细节
  • 结论 / 流程改进：未来评估 peer agent 项目时，必须先用 web search + 营销文档建词汇表，再做关键词搜索。仅靠 grep 等于在用自己的命名模型去匹配别人的 —— 假设两个项目命名同源，但事实经常不是
  • 影响：v4.3 一次性补 4 个 pattern，hermes 总数从 18 → 22 (+22%)，axiomate 缺失从 1 → 4 (+300%)。Priority 表新增 1 个 P1 (Skill Auto-Distillation) + 1 个 P2 (Plugin/Hooks) + 2 个 P3。这对项目的实际优先级判断是大幅度调整
  • Lesson 沉淀建议：本条洞察值得沉淀到 axiomate memory 作为 feedback memory（"评估 peer agent 必先 web search 建词汇表"），但本次 v4.3 报告更新不含 memory 修改，需后续单独评估

关键洞察 (v4)

  • Manus 原则改变了路线图的决策框架：之前 v3.2 把 compact 拆成 5 PR、5-7 天工作量推进，假设"所有 hermes 有的 feature 都该 port"。v4 期接受 Manus "失败记录 = learning signal" 后，意识到 PR 2-5 大多数都是负价值而不只是低 ROI。Token 节省的正确路径不是裁剪 history，是扩 context / 提早 compact / tool-level dedup。这条原则有更广的辐射范围（error handling / tool repair / hooks 是否也该用 Manus 视角重审），见 plan 文件的"哲学记录"段
  • PR 1 的 latent defect 是测试设计学的反面教材：v4 的 compactFullPipeline.test.ts 用 forkContextMessages: null 短路了主路径检测，导致测试通过但生产 bug 没暴露。后续 iterativeFilterForkPath.test.ts 改用 mock runForkedAgent 拦截参数验对称应用 —— 这个 mock 模式应作为后续所有 compact pre-pass 测试的标准
  • Convergent evolution 是 gap 真实性的强信号：axiomate d69efd4（parsed_array_string repair）和 hermes 9ff21437（MCP coerce stringified arrays/objects）独立得出同型答案。这种"两个项目不约而同修同一个 bug"是确认问题确实存在的最强证据 —— 强于纯架构分析
  • Hermes 进入"补漏期"：3 天 506 commit 但无新 mode，全是 bug fix / 鲁棒性 / niche provider 适配。axiomate 的 18 mode 对照表静态稳定，未来对照可以从"模式 parity"转到"bug 模式"对比 —— 谁修了哪些 hermes 也踩过的坑
  • Hermes c0385873（summary fallback NameError）侧面验证 PR 3 #8 的判断：hermes 自己的 fallback 路径都还在出 latent NameError —— 我们当时拒绝 port 这个"静态消息 fallback"机制是对的，因为它在 hermes 那边也并非"已被验证的健康代码"
  • v4.1 概念分类的重要性：原报告把 Auxiliary Client 和 Smart Routing 隐性合并成"模型层差距"，导致 axiomate 优先级被误导。挖完代码发现是两个独立机制：side-task backend resolution（hermes 仍重投，auxiliary_client.py 2,771 行 active）和 user-turn cheap-model classifier（hermes 已删 smart_model_routing.py 195 行）。axiomate 应只关注前者方向（已 70% 覆盖，剩余 per-task config / vision / credit-exhaustion 是边际项），后者完全跳过。教训：分类错误会传染优先级。每次评估 hermes feature 时要先确认它在 hermes 当前是否还活着
  • v4.1 Windows 沙箱不再列入 axiomate gap：之前因 axiomate Windows 无 sandbox 列入 gap 是错的。挖完发现 hermes 也是 disable（code_execution_tool.py:53 SANDBOX_AVAILABLE = sys.platform != "win32"）；行业现实是 Windows 缺 unprivileged sandbox primitives。axiomate sandbox-axiomate（bwrap+sandbox-exec）反而比 hermes 的"PTC 进程隔离"更接近真实安全沙箱
