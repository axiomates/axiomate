v4.4 架构层 reframe（同日，2026-04-24）—— 修正 v4.3 推荐 P1 时的视野盲区：
  • Hermes 的 #19-22 不是独立 feature，是同一个上层建筑的不同露头 —— Skill auto-distill (Layer 3) 依赖 auto-recall (Layer 2) 依赖 SessionDB+FTS (Layer 1) 依赖 user-modeling (Layer 4 反向)。Cherry-pick 任何一层 = workaround。
  • Hermes 整套就是 user-behavior RAG。Corpus = user 历史/skill/memory；Retrieval = FTS + 向量检索 + chain；Augmentation = <memory-context> 注入；Generation = LLM 拿增强 context 答。这个视角让架构决策清晰得多。
  • 驱动 hermes 这套的真实需求是产品形态，不是技术偏好：always-on personal companion（Telegram/Slack/Discord/CLI/cron 多平台、跨年关系积累）vs Claude Code/axiomate 的 task-scoped tool。前者必须记得，后者每次任务可重新开始。
  • Axiomate compact 的"信息丢失"是物理保留 + 访问熵无穷：JSONL append-only 物理上原文都在，但没工具能查、agent 拿不到、reload 时跳过 —— 等同于丢，最坏的设计选择（付了存储成本，没换可访问性）。
  • Skill Auto-Distill P1 撤回：v4.3 把它当独立 feature 推荐错了。改标 "P1 conditional on identity decision" —— 取决于 axiomate 是否走 personal companion 演进。详见 Priority 表 + 新增"关键洞察 (v4.4)"段。
  • JSONL MVP 路径：如果只暴露 hermes 风格的 session_search tool 给 LLM，JSONL 完全够用（axiomate 当前规模 50-200ms 扫描）。SQLite 是 "几千 session 后才需要的升级"，不是入场门槛。把"7-9 周架构重塑"拆成"~1 周 MVP + 看效果再说"。

三条路径摆台：① 守 task tool 身份 → #19-22 全部 WONTFIX ② 走 companion 路 → 数月级重塑 ③ MAYBE 路（推荐）→ 项目级 session DB + search tool，不跨项目不跨平台不做 cron。

Manus 原则收官 —— 为什么 PR 2-5 全部不做

Manus 2025-11 "Context Engineering" 文章的核心论点：

AI 犯错的历史记录本身就是 learning signal。压缩掉"失败的尝试"等于在抹去"为什么不那么做"的教训。下一轮 LLM 失去了自我纠错的凭据，会重复同样错误。

这个原则直接否定了 PR 2-5 里所有"通过裁剪历史节省 token"的优化：

PR原方案v4 决定理由
PR 1 Iterative prompt + prev summary 注入 ✅ 已落地 不抹历史，只是更稳的 update vs rewrite。Manus 原则中性。
PR 2 Pre-pass tool-aware 结果剪枝 ❌ 不做 裁掉 tool output 等于裁掉 LLM 自我纠错依据。"npm test → 47 lines"摘要丢的是错误堆栈/重现步骤。
PR 3 compact 走 midModel + 静态 fallback ❌ 不做 #6：主模型做 compact 是合理的（quality > cost）。#8：实地挖代码后发现 axiomate 现有 fail-fast (notification + 3 次熔断 + reactive compact) 比 hermes 静默静态消息健康得多 —— hermes 的"占位 + 删消息"会让模型在无摘要状态下继续瞎聊，净值为负。
PR 4 dedup + truncate + anti-thrash + redaction ❌ 不做 dedup 二元组会丢历史快照（"原先是 user.token"信息没了）；三元组本身命中率 <10%（Read 已被 FILE_UNCHANGED_STUB tool-level 处理）。truncate 巨型 paste 里的 stack trace / 重现步骤是高价值信号。anti-thrash 与现有 3 次熔断重叠。
PR 5 /compact <topic> budget 分配 ❌ 暂不做 focus-topic 会更激进按"相关性"砍内容，更容易丢学习信号。如果未来要做，思路应是"扩相关段 budget"而非"削无关段"。

替代路径：需要省 token 时走 1) 扩 context window（换更大模型）；2) 提早触发 compact（让每次 compact 做得更"浅"）；3) tool-level idempotent stub（如 Read 的 FILE_UNCHANGED_STUB，在 tool 执行阶段消重，不是 compact 后处理）。

影响：compact feature parity 在 v4 从 v3.2 的 ~40% 推到 ~50%（PR 1 关掉 #1 #2），剩余 8 个 gap 出于哲学原因永久放弃。这不是"懒得做"，是"做了反而是负价值"。

🔍 v4.1 后续概念纠错（2026-04-24 同日）

用户深入提问揭示三个早期分类错误，本节集中纠正：

  1. Auxiliary Client ≠ Smart Routing：原 v3 报告隐性把两者归并为"模型层架构差距"。实测后是两个独立机制：
    • Auxiliary Client = side-task router（compression / search / vision 用哪个 backend）—— hermes 仍重投，2,771 行 active
    • Smart Routing = user-turn classifier（用户 message 走 cheap 还是主 model）—— hermes 已删除（424e9f36，2026-04-19）
    见上方 GAP 3 / GAP 4 改写。
  2. Hermes "Code Execution Sandbox" ≠ axiomate sandbox-axiomate：前者是 PTC（Programmatic Tool Calling）的 child process 进程隔离，后者是 OS-level 安全沙箱。两个项目对 Windows 都是 disable —— 不是 axiomate gap，是行业现实。Windows 缺 unprivileged sandbox primitives：
    • Windows Sandbox / Hyper-V Container：分钟级启动 + 要 Pro/Enterprise + 虚拟化
    • Job Objects：能逃逸，不算真隔离
    • WSL2+seccomp：能用但加 WSL 强依赖
    • AppContainer：实现复杂度极高（manifest / capability registration / Windows-only API）
    • Docker Desktop：重依赖 + 性能损耗
    Hermes 和 axiomate 同选 disable + 提示用户。这条不计 gap。
  3. Smart Routing 的状态从"v3 推荐 port" 变成"hermes 自己删了" —— 强烈负信号。axiomate 不要 port。替代方案：/fast slash command（用户主动）。

C. 信息论视角 —— axiomate compact 的访问熵

软件价值 = 降低用户面对系统时的有效熵。但 compact 干的是反向 —— 把 agent 已经替你处理过的失败/学习的 entropy reduction 重新散掉。下次同问题要从原始 chaos 重走。

具体看 axiomate compact：

  • Live 上下文层：原 50 条 messages → 1 条 LLM summary，agent 后续看不到原文（只看转述版本）
  • On-disk JSONL 层：append-only，原文物理上还在 (sessionStorage.ts:1457 注释明确说 "Preserved messages exist in the JSONL with their ORIGINAL pre-compact ...")
  • 但功能上等同于丢：没工具搜、agent 拿不到、--continue 跳过 boundary、cross-session 完全不可能

结论：axiomate 是 物理保留 + 访问熵无穷，几乎是最坏的设计选择 —— 既付了存储成本，又没换可访问性。Hermes 的 parent_session_id chain + FTS 是信息论意义上更正确的实现：compact = "封顶 + 开新 session"（不是原地压），原文以独立 session 永久可查（FTS O(log N) 检索），等价于 物理保留 + 访问熵可接受。这点上 hermes 比 axiomate 设计哲学更彻底，与 Manus "保留失败痕迹" 原则更内洽。

F. JSONL MVP 路径 —— 把 MAYBE 路再拆

关键洞察：只暴露 hermes 风格的 session_search(query, [filters]) tool 给 LLM，存储完全可以保持 JSONL。LLM 不知道下面是 SQLite + FTS5 还是 stream-grep + JSONL —— 工具行为一致就行。

规模JSONL 扫描SQLite + FTS5
100 sessions, 10MB50-200ms ✅10-50ms
1000 sessions, 100MB1-3s 🟡50-100ms
10000+ sessions, GB痛 🔴仍 OK

Axiomate 当前规模（单用户、项目级、个位数到几十 session）JSONL 零痛点。SQLite 是"几千 session 后才需要的升级"，不是入场门槛。

把 MAYBE 路径再拆：

  • Phase 1（~1 周）：建 session_search tool，复用现有 JSONL，简单关键词匹配 + 上下文窗口 + 时间排序。不动存储层。已经能解锁 7 个 hermes 触发点中最高杠杆的那个：LLM 自主查历史。
  • Phase 2（条件性，~2-4 周）：JSONL 扫描真痛了（实测 > 2 秒）才迁 SQLite。届时有真实 query pattern 参考，schema 反而设计得更准。
  • Phase 3（条件性，~2 周）：Phase 1 之后 LLM 已经能 query 历史，再决定是否加 skill_manage + 后台 review。届时上层才有意义。

这把原 v4.3 的 "Skill Auto-Distill 1-2 周" 错估，改写为 "Phase 1 ~1 周 MVP，剩余按需"。低风险、可中止、不绑死 axiomate 的产品身份。

  2. 实操起点是 Phase 1 (JSONL + session_search tool, ~1 周)。不需要先回答 NO/YES/MAYBE 的身份问题就能开始 —— Phase 1 本身在三条路径下都有价值（NO 路也能用 search 找过往）。
