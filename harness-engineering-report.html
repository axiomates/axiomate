2. 缺失模式详解

P1 Structured Error Taxonomy + Recovery Hints

Hermes 实现

agent/error_classifier.py (809 lines)

  • 14 种 FailoverReason: auth, auth_permanent, billing, rate_limit, overloaded, server_error, timeout, context_overflow, payload_too_large, model_not_found, format_error, thinking_signature, long_context_tier, unknown
  • ClassifiedError 携带 4 个 recovery hints:
    • retryable: 是否重试
    • should_compress: 重试前是否压缩 context
    • should_rotate_credential: 是否换 credential
    • should_fallback: 是否降级模型
  • 智能分类: 402 区分 billing vs transient rate_limit (通过 "try again" / "insufficient credits" 关键词); 大 session + disconnect → context_overflow; 400 + >80K tokens → compress
  • 支持 中文错误信息 识别 ("超过最大长度", "上下文长度")

Axiomate 现状

services/api/errors.ts (1173 lines) + withRetry.ts

  • 有 20+ error types 但是 平铺的，没有 recovery hints
  • retry 逻辑直接检查 HTTP status code，没有语义分类
  • 没有 "should_compress" / "should_rotate" 的概念
  • 529 fallback 是 hardcoded（连续 3 次 → 降级模型），不是通用机制

价值: 多模型场景下，不同 provider 的错误语义不同。结构化分类让 retry/fallback/compress 决策精确，减少无效重试和 token 浪费。

P1 LLM-Assisted Context Compression

Hermes 实现

agent/context_compressor.py (37KB)

  • 5 阶段算法:
    1. Pre-pass: 用 placeholder 替换旧 tool results (>200 chars)
    2. Boundary protection: 保护 head (前 3 条) + tail (~20K tokens)
    3. LLM Summarization: 用便宜模型 summarize 中间部分
    4. Tool pair sanitization: 修复孤立的 tool_call/result
    5. Assembly: head + summary + tail 拼接
  • Summary 结构: Goal / Constraints / Progress (Done/In Progress/Blocked) / Key Decisions / Resolved Questions / Pending User Asks / Relevant Files / Remaining Work / Critical Context
  • 迭代更新: 多次压缩时保留并更新 previous summary
  • Budget: summary tokens = min(content_tokens * 0.20, 12K)
  • 失败 cooldown: LLM 失败后 600s 内不再尝试

Axiomate 现状

services/compact/ (compact.ts 1704 lines)

  • 有 4 种 compaction 策略: full, micro, reactive, snip
  • Full compact 调用 API 做 summary 但 不如 hermes 的结构化
  • Micro-compact 是增量的，保留 tool interactions
  • Post-compact 会恢复文件 + skills
  • 缺少: iterative summary update、structured template、tool pair sanitization

价值: 长对话 (>50 轮) 会丢失关键上下文。结构化 summary 保留语义信息，成本很低（用便宜模型）。

P1 Prompt Injection Detection

Hermes 实现

agent/prompt_builder.py (lines 35-73)

  • 20+ 注入模式检测:
    • "ignore previous instructions" / "disregard rules"
    • "do not tell the user" / "hide this from"
    • System prompt override attempts
    • Hidden div CSS exploits
    • Exfiltration curls (API keys in URLs)
    • Secret file reads (.env, credentials, .pgpass)
  • 不可见 Unicode 检测: U+200B-U+202E (零宽空格等), U+FEFF (BOM)
  • 匹配到 → 返回 [BLOCKED: suspicious content]
  • 扫描 context files (.hermes.md) 在注入到 system prompt 之前

Axiomate 现状

  • 完全没有 context file 注入检测
  • CLAUDE.md / .claude/ 文件直接注入 system prompt，无安全扫描

价值: 安全防线。用户 clone 的项目可能包含恶意 CLAUDE.md。实现成本低（纯正则）。

P2 Smart Model Routing (便宜模型分流)

Hermes 实现

agent/smart_model_routing.py

  • Heuristic classifier: 所有条件必须满足才路由到便宜模型:
    • ≤ 160 chars, ≤ 28 words, ≤ 1 newline
    • 不含 code markers (backticks)
    • 不含 URLs
    • 不含复杂关键词 ("debug", "implement", "refactor", "patch", "error", "test", "plan", "delegate" 等)
  • 失败时静默 fallback 到主模型

Axiomate 现状

  • 没有 cheap model routing
  • 所有请求走同一模型配置
  • 需要先支持 auxiliary model 概念

价值: 降低成本 30-50%（简单问题用 Haiku 而非 Opus）。但需要先实现 auxiliary model 基础设施。

P2 Credential Pooling + Rotation

Hermes 实现

agent/credential_pool.py (1,364 lines)

  • PooledCredential: id, label, auth_type (oauth/api_key), priority, source, status tracking
  • 4 种 rotation strategy: fill_first (default), round_robin, random, least_used
  • Exhaustion tracking: last_status (OK/EXHAUSTED), TTL cooldown (1h for 429), provider-supplied reset_at override
  • Recovery flow:
    • 429 rate_limit: retry same once → rotate on second
    • 402 billing: immediate rotation
    • 401 auth: try refresh → rotate if fails
  • OAuth refresh: Anthropic, Codex, Nous 各自的 refresh 逻辑
  • 6+ seeding sources: manual, env vars, OAuth flows, Claude Code creds

Axiomate 现状

  • 单 API key per model config
  • OAuth refresh 存在 (handleOAuth401Error) 但没有 pool
  • 没有 rotation / exhaustion tracking

价值: 对多用户/高负载场景重要。单用户 CLI 场景优先级低。

3. 部分实现模式详解

Context Compression (部分实现)

Axiomate 有 4 种 compact 策略 (full/micro/reactive/snip) 和 post-compact restoration，但缺少 hermes 的 iterative summary update（多次压缩时保留前一次 summary 并更新）和 structured template（Goal/Progress/Pending/Remaining）。

Cross-Session Memory (部分实现)

Axiomate 有 file-based auto-memory (MEMORY.md + individual memory files)，但没有 hermes 的 pluggable provider architecture（built-in + 1 external，lifecycle hooks, failure isolation）。

Error Classification (部分实现)

Axiomate 有 20+ error types 和 detailed parsing，但缺少 hermes 的 recovery hints（retryable/should_compress/should_rotate/should_fallback）。当前 retry 逻辑直接检查 HTTP code 而非语义分类。

6. 建议优先级

优先级模式预估工作量价值理由
P1 Error Taxonomy + Recovery Hints 2-3 天 高 直接提升多模型可靠性; 减少无效重试; 为后续 credential rotation 铺路
P1 Prompt Injection Detection 0.5-1 天 高 安全防线; 实现简单 (纯正则); CLAUDE.md 是高风险注入点
P1 Context Compression 增强 (iterative summary) 2-3 天 高 长对话质量关键瓶颈; Axiomate 已有 compact 基础设施, 只需增加 structured template + iterative update
P2 Auxiliary LLM Client 3-5 天 中 是 smart routing 和 LLM compression 的前置依赖; 需要新增 model config 概念
P2 Smart Model Routing 1-2 天 (依赖 auxiliary client) 中 降低成本; 但需要 auxiliary model 基础设施
P3 Credential Pooling 5-7 天 低 (单用户) Axiomate 目前是单用户 CLI, 多 credential 需求不强

关键洞察

  • Axiomate 在 tool 层已经很强: tool call repair (schema-guided) 和 tool result management 都比 hermes 更成熟
  • 最大差距在 "模型层" 和 "安全层": error recovery hints, LLM-assisted compression, prompt injection detection
  • Hermes 的 auxiliary LLM 是关键架构: 它让 compression 和 smart routing 成为可能 — 是很多高级 harness 模式的基础设施
  • 最低成本高回报: prompt injection detection (0.5 天, 纯正则, 安全影响大)
