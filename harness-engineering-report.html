Tool Call Repair — 7 轮精雕（2 天内，6 commit）

基于 v2 之前的 seam-hygiene 基础（PR 1/2/3，commit 1303734），v3 期深入内核审计并落地以下改进：

  • commit d6a4546: tolerantJsonParser.ts 三个真 bug — 不可达 newline 分支清除；+N / +3.14 接受为数字字面量；canValuePossiblyMatchSchema 的 integer 和 number 分离（避免 "3.14" 占据整数槽）
  • commit d69efd4: 数组字段新增 parsed_array_string repair — {tags: "[\"a\",\"b\"]"} 这种被字符串化的数组能直接被还原
  • commit 9575ca8: tolerantJsonParser.scoring.test.ts — 固化 candidate 打分公式（常数 5/2/3）的 ordering relationships，未来 tune 时有护栏
  • commit bace428: rectangularAssignment.ts（150 行 Kuhn-Munkres/Hungarian 实现 + property test 50 次随机矩阵对比暴力全排列）替换原有的贪心 key→property 匹配；解决"两个 input key 竞争同一 schema property 时被挤掉的那个丢失次优映射"的 latent bug
  • commit 22f0e09: repairNumberAgainstSchema 正则与 canValuePossiblyMatchSchema 对齐（feasibility check 和 coercion 不再脱节）

结果：447 tests 全绿（v2 的 419 → v3 的 447）。schemaGuidedRepair.ts 和 tolerantJsonParser.ts 代码量+测试量增长，功能质变 — 从"能修"变成"修得准 + 说明白 + 止损"。

🐛 v3 期新发现 Gap —— M2.7 duplicate tool_use ID（100% 可复现）

同一轮 smoke 测试，M2.7 via NVIDIA integrate.api.nvidia.com 端点在**每次**涉及工具的调用中都 emit 重复 ID 的 tool_use 块：

[ERROR] ensureToolResultPairing: repaired missing tool_result blocks (3 -> 3 messages)
assistant(tool_uses=[call_6bb0951..., call_6bb0951...])   ← 同一个 ID 两次
user(tool_results=[call_6bb0951..., call_6bb0951...])    ← 对应结果也同 ID

Axiomate 的 ensureToolResultPairing（cli.js:185092）能结构性兜住，但**用户可见的最终输出为空**——下一轮模型无法产出有效回复。这不是 hermes 有 / axiomate 没有的"模式缺失"，而是 axiomate 对一个受支持 provider（NVIDIA）的适配 bug。

优先级：高。1-2 天调查，怀疑是 openaiStreamAdapter.ts 或类似组件把 NVIDIA 流式格式的同一次 tool call 重复累积了两份。详见 memory: project_m27_duplicate_tool_use_id.md。

2. 缺失模式详解

P1 Prompt Injection Detection

Hermes 实现

agent/prompt_builder.py (1,046 行, 注入检测在 L32-73)

  • 20+ 注入模式检测:
    • "ignore previous instructions" / "disregard rules"
    • "do not tell the user" / "hide this from"
    • System prompt override attempts
    • Hidden div CSS exploits
    • Exfiltration curls (API keys in URLs)
    • Secret file reads (.env, credentials, .pgpass)
  • 不可见 Unicode 检测: U+200B-U+202E (零宽空格等), U+FEFF (BOM)
  • 匹配到 → 返回 [BLOCKED: suspicious content]
  • 扫描 context files (.hermes.md) 在注入到 system prompt 之前

Axiomate 现状

  • 完全没有 context file 注入检测
  • AXIOMATE.md / .axiomate/ 文件直接注入 system prompt，无安全扫描
  • 5 天内无变化

价值: 安全防线。用户 clone 的项目可能包含恶意 AXIOMATE.md。实现成本最低（纯正则 + Unicode 范围检查）。

P1 LLM-Assisted Context Compression（Iterative Summary 缺失）

Hermes 实现

agent/context_compressor.py (1,216 行)

  • 5 阶段算法:
    1. Pre-pass: placeholder 替换旧 tool results (>200 chars)
    2. Boundary protection: 保护 head (前 3 条) + tail (~20K tokens)
    3. LLM Summarization: 用便宜模型 summarize 中间部分
    4. Tool pair sanitization: 修复孤立的 tool_call/result
    5. Assembly: head + summary + tail 拼接
  • Summary 结构化模板: Goal / Constraints / Progress (Done/In Progress/Blocked) / Key Decisions / Resolved Questions / Pending User Asks / Relevant Files / Remaining Work / Critical Context
  • 迭代更新: 多次压缩时保留并更新 previous summary
  • 本期新增: _truncate_tool_call_args_json() 防止压缩破坏 tool-call JSON
  • 失败 cooldown: LLM 失败后 600s 内不再尝试

Axiomate 现状

services/compact/compact.ts (1,468 行) + microCompact + postCompactCleanup

  • 4 种 compaction 策略: full / micro / reactive / snip
  • 本期新增: 在 context_overflow 错误时 reactive compact 跳跃式压缩（commit bb7bb8e）
  • Full compact 调用 API summarize 但 无结构化模板
  • Post-compact 恢复文件 + skills
  • 缺少: iterative summary update、structured Goal/Progress/Pending template、tool pair sanitization

价值: 长对话 (>50 轮) 会丢失关键上下文。结构化 summary 保留语义信息，成本低。v3 校准：便宜模型基础设施已就绪（axiomate 三级模型 currentModel / midModel / fastModel 直接可用），不依赖额外 Auxiliary Client。

P3 Auxiliary LLM Client v3 重评：部分实现

Hermes 实现

agent/auxiliary_client.py (2,771 行)

  • 独立辅助 LLM client，专用于 compression / vision / search
  • 6 级 fallback chain: per-task config overrides → primary aux → secondary aux → main model → cheap model → local
  • Credit exhaustion 自动 fallback
  • 新增 resolve_vision_provider_client() 选择视觉模型

Axiomate 现状 v3 校准

v2 标"完全没有"错了。axiomate 已有三级模型体系：

  • config.ts: currentModel / midModel / fastModel 三级 config 字段
  • Resolver 层: getCurrentModel() / getMidModel() / getFastModel() 在 utils/model/model.ts
  • 自动 fallback 链 (query.ts:255-263): main 失败时 current → mid → fast 降级；等价于 hermes "auxiliary fallback"
  • 已有内部 smart routing:
    • findRelevantMemories → midModel
    • awaySummary → fastModel
    • tokenEstimation → fastModel
    • anthropicProvider connection probe → fastModel

与 hermes 的真实差距（远小于 v2 标注）：

  • 3 级 vs 6 级 fallback（对单用户 CLI 足够）
  • 无 per-task override 配置表（硬编码哪个任务用哪个 tier）
  • 无 credit exhaustion tracking（但 errorClassifier 已能分辨 billing/auth 失败）
  • 无独立的 vision provider resolution（单用户场景未遇到）

价值: 功能重叠度 > 70%。Hermes 多出来的部分（per-task 表、6 级、credit tracking）主要服务多用户/高负载场景，对本地 CLI 边际价值低。不是 P2 gap，是 P3 完善项。

P2 Smart Model Routing v3 校准：依赖已满足

Hermes 实现

agent/smart_model_routing.py (195 行)

  • Heuristic classifier: 所有条件必须满足才路由到便宜模型:
    • ≤ 160 chars, ≤ 28 words, ≤ 1 newline
    • 不含 code markers (backticks)
    • 不含 URLs
    • 不含复杂关键词 ("debug", "implement", "refactor", "patch", "error", "test", "plan", "delegate")
  • 失败时静默 fallback 到主模型

Axiomate 现状 v3 校准

  • 基础设施已就绪 — getMidModel() / getFastModel() 可直接调用
  • 没有 **user-turn 级** heuristic classifier 分流（只在内部任务如 memory/summary 硬编码用 cheap tier）
  • v2 说"依赖先实现 Auxiliary LLM Client"错了 — 依赖已满足，直接能做

价值: 降低 token 费 30-50%（"2+2" 不该用 main model 回）。因为依赖已满足，纯粹是加一个 heuristic classifier + 路由判断，1-2 天可完成。

P3 Credential Pooling + Rotation

Hermes 实现

agent/credential_pool.py (1,326 行) — 本期 Codex auth 隔离 (commit b02833f3)

  • PooledCredential: id, label, auth_type (oauth/api_key), priority, source, status tracking
  • 4 rotation strategy: fill_first (default), round_robin, random, least_used
  • Exhaustion tracking: last_status (OK/EXHAUSTED), TTL cooldown (1h for 429), provider-supplied reset_at override
  • Recovery flow: 429 → retry once → rotate; 402 → immediate rotation; 401 → refresh → rotate if fails
  • OAuth refresh: Anthropic, Codex (本期独立), Nous

Axiomate 现状

  • 单 API key per model config
  • OAuth refresh 存在 (handleOAuth401Error) 但没有 pool
  • 没有 rotation / exhaustion tracking

价值: 对多用户 / 高负载场景重要。单用户 CLI 场景优先级低。

3. 部分实现模式详解

Context Compression (部分实现)

Axiomate compact.ts (1,468 行) 有 4 种 compact 策略 (full/micro/reactive/snip) 和 post-compact restoration，本期还在 context_overflow 时新增了 reactive compact 跳跃式压缩。但缺少 hermes 的 iterative summary update 和 structured template（Goal/Progress/Pending/Remaining）。

Cross-Session Memory (部分实现)

Axiomate sessionMemory.ts (450 行) 有 file-based auto-memory (MEMORY.md + individual memory files)，本期 sessionMemoryEnabled 与 extractMemoriesEnabled 进入 /config UI。但没有 hermes memory_manager.py (373 行) 的 pluggable provider architecture（built-in + 1 external，lifecycle hooks on_turn_start/on_pre_compress/on_delegation，failure isolation）。

Subagent / Delegation (部分实现，重新评估)

Axiomate AgentTool/runAgent.ts + forkedAgent.ts: forked agents 与 state isolation。缺少 hermes 的形式化 depth limit (=2)、heartbeat (30s)、credential inheritance 模型、toolset restriction。

Provider Abstraction (部分实现)

Axiomate 有 provider.ts neutral interface (StreamRequest/Response, RequestHooks)，及 anthropicProvider.ts (675 行) + openaiProvider.ts (425 行)。但缺少 hermes 的 3 API modes (chat_completions/anthropic_messages/codex_responses) 和 6-level fallback chain。

6. 建议优先级（v3 校准）

⚠ v3 校准声明： v2 优先级基于"port hermes gap"思路，未区分威胁模型与使用场景。axiomate 是本地单用户 CLI 开发工具，不是 multi-tenant SaaS — 威胁假设和 hermes 不同。以下 v3 优先级按 真实影响力 ÷ 工作量 重排，并补入 v3 实测发现的真 gap。

优先级模式预估工作量价值v3 理由
P1v3 新 M2.7 duplicate tool_use ID 诊断 + 修复 1-2 天 高 100% 可复现的用户可见故障：M2.7 via NVIDIA 一调工具就"没输出"。ensureToolResultPairing 结构性兜住但模型后续无回复。怀疑 openaiStreamAdapter.ts 对 NVIDIA 流式格式的兼容 bug。最高 ROI — 修好等于救活整个 M2.7 支持路径
P1 Iterative Summary Templates (compaction 增强) 2-3 天 高 长对话 (>50 轮) 质量退化是**每天**撞得到的问题，非假想。compact.ts 已有 1,468 行基建，只需 structured Goal/Progress/Pending/Remaining template + iterative update 逻辑。用主模型也能做，不阻塞在 auxiliary client 上

关键洞察 (v3)

  • Tool Call Repair 已是 axiomate 最稳固的领域 — v3 拉大差距: 7 轮精雕后，21 种 JSON repair kind + Hungarian 全局分配 + structured diagnostic hint + 连续失败计数。hermes 的差距从"更强"变成"更强 ++"。此领域进入"收尾 + 维护"状态，除非新模型暴露新 failure mode
  • 实测 > 推测: v2 标 XML-in-text 是潜在 gap，v3 跑 6 次 MiniMax smoke 得 0/6，直接关闭。经验教训：再好的理论假设也要真跑过才算数
  • 真正的 P1 不是 hermes 有的 gap，而是 axiomate 自己的 bug: M2.7 duplicate tool_use ID 在两次实测中 100% 复现 — 比任何"port hermes 模式"都更值得优先修
  • 威胁模型要区分场景: v2 把 Prompt Injection 标 P1 "最低成本最高回报"，v3 校准为 P3 — 本地 CLI 的 attack surface 小于 SaaS。照搬 hermes 优先级等于接管 hermes 的威胁假设
  • v2 的"模型层架构差距"被 v3 大幅修正: axiomate 实际已有 currentModel / midModel / fastModel 三级 + 自动 fallback + 内部任务分流。"剩余差距是依赖链"的 v2 判断错了 — Smart Model Routing 可以直接做，不需要先建 auxiliary client。真正的架构缺口只剩 Iterative Summary Template
  • Hermes 5 天仍在防御加固: JSON 完整性、Codex auth 隔离、cron approval、auto routing、4.7 迁移。hermes 的 18 模式趋于饱和，axiomate 在 v3 期对 Tool Repair 的深度投入已反超此格局

v2 洞察（已被 v3 部分超越，保留供参考）

  • Axiomate 5 天进步显著: Error Classification 从 P1 缺口闭环为 Advanced；同时完成 Independence 清理 (~5,900 LOC 移除 + first-party feature pruning)，harness 与 Anthropic 私有协议彻底解耦
  • Axiomate 在工具层 + 安全层（命令检测）已非常强: tool call repair (1,237 行 schema-guided), tool result management (983 行), bash security (2,427 行) 都比 hermes 更成熟
  • 剩余差距集中在 "模型层": auxiliary LLM client / smart routing / LLM-assisted compression — 这三者形成依赖链 ← v3 校准：依赖链是错的。axiomate 三级模型 + fallback 链已存在，Smart Routing 可独立做
  • Hermes 转向防御加固而非新模式: 5 天内所有重要更新都是 hardening (JSON 完整性、Codex auth 隔离、cron approval、auto routing) 或新模型迁移 (4.7) — 暗示 hermes 的 18 模式已基本饱和，竞争重点转向稳定性
  • 最低成本高回报仍是 prompt injection detection ← v3 校准：该结论过度乐观，详见上方 v3 洞察第 4 条
