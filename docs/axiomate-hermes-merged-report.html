Executive Judgment

结论摘要

Axiomate 已经实现 persistent goals。 代码与文档显示它具备 Ralph-loop 风格 /goal：GoalManager、JSONL goal-state 持久化、turn-end judge、/subgoal 附加准则、Ctrl+C pause、turn budget、parse-failure backstop、footer GoalIndicator、/resume/--continue/compact resume 状态恢复，以及用户文档 docs/user/goals_zhcn.html。这是一等控制原语，不是待补功能。

Hermes 的领先仍然真实，但领先点更集中。 它不是因为工具更多而领先，而是把 long-running agent 做成了平台级 runtime：durable kanban、Curator/background review、MemoryProvider/ProviderProfile/ContextEngine plugin ABI、gateway/delivery/restart、s6/service supervision、security hardening 与跨入口 session continuity。2026-05-10 之后的提交进一步强化的是这些面，而不是单一模型 adapter。

Axiomate 的战略优势也更清晰。 Rust napi、CDP browser bridge、computer-use-mac/win、rtk-axiomate pre-LLM 工具输出压缩、4 后端 swarm、协议中性 provider 抽象、file checkpoint + conversation rewind 是真实工程深度。Axiomate 不应把这些优势稀释成“也做一个 Hermes gateway clone”。

一句话判断： 当前差距不是“缺 goal / 缺 checkpoint / 缺工具”，而是“缺 runtime 治理方法”。Axiomate 下一轮优先级应是把已有 primitive 组织成 durable task runtime：kanban lifecycle、curator review、plugin ABI、runtime resume、guardrails 与 delivery/watchdog，而不是追 Hermes 的全部聊天平台。
Axiomate /goal 已实现 Checkpoint + conversation rewind 仍是优势项 rtk-axiomate 是产品级 token 经济 Hermes 5/10 后增强 platform runtime 最高风险：swarm primitive 未升级为 durable orchestration

Gap Analysis

当前详细差距

读法：下面每条先解释 Hermes 那套系统是干什么的、由哪些模块组成，再写 Axiomate 当前缺哪一层。重点不是“对方有一个文件名我们没有”，而是它是否形成了可复用的 runtime 能力。

1. Durable 多代理：swarm primitive 还不是 kanban runtime

Hermes 的 Kanban 不是“一个看板 UI”，而是多代理长任务的持久化控制平面。
最高优先级

Hermes 这套系统做什么

  • 把长任务拆成可持久化 card，而不是只把 prompt 扔给一个 worker。
  • 记录任务状态、依赖、claim、block、promote、done，让多个 agent 可以围绕同一个 board 协作。
  • worker 出错、幻觉 claim、卡住或需要人工输入时，可以通过 diagnostics / hallucination gate 把状态上抛。

主要组成

  • plugins/kanban/：runtime 插件入口。
  • hermes_cli/kanban*.py：CLI、DB、decompose、diagnostics、swarm、specify 等控制面。
  • TUI/gateway 集成：让 board 状态不是只存在于一个终端 turn 里。

Axiomate 缺哪一层

  • 已有 swarm/backends、workerAgent、teammateMailbox，能启动 worker。
  • 缺 durable board state、task ownership、lease/heartbeat/reclaim、worker-initiated block。
  • 缺把 goal/subgoal 绑定到 worker card 的统一任务语义。

为什么重要：没有 board + ownership，长任务一旦 worker crash、漂移或被用户抢占，就只能靠 transcript 猜状态；有 kanban runtime 后，cron、subagent、goal、resume 都能复用同一套任务锚点。

2. Curator / background review：Axiomate 还没有自我改进闭环

Hermes 的 Curator 是“跑后复盘 + 技能/记忆治理”的后台流水线，不是简单 memory 写入。
高优先级

Hermes 这套系统做什么

  • 在主任务之外 fork 一个 review agent，审查最近会话、memory、skill 使用和失败模式。
  • 输出结构化报告，把建议分成 consolidate、prune、archive、promote 等类别。
  • 让 agent 的经验沉淀进入可维护资产，而不是无边界地堆到上下文里。

主要组成

  • agent/curator.py：维护/整理 skill 与 memory 的 orchestrator。
  • agent/background_review.py：后台 fork review。
  • hermes_cli/curator.py + logs/curator/run.json + REPORT.md：可审计产物与 CLI 控制面。

Axiomate 缺哪一层

  • 有 memdir、skills、SkillImprovementSurvey，但 Survey 是用户问卷，不是 agent 自评。
  • 缺 review rubric、per-run artifact、后台 fork、archive/promote 生命周期。
  • 缺让 goal/checkpoint 结果自动反馈到 skill/memory 的闭环。

为什么重要：没有 Curator，memory/skill 会随使用量增长而变脏；有 Curator 后，agent 能把失败、成功模式和新工具经验变成可治理资产。

3. MemoryProvider / ContextEngine：Axiomate 仍是单实现，Hermes 已 provider 化

Hermes 把 memory 从“本地文件功能”升级成了 runtime provider 接口。
中高优先级

Hermes 这套系统做什么

  • 把“记忆如何检索、如何写入、如何注入系统 prompt、如何随 turn 同步”变成统一接口。
  • 不同后端可以提供 prefetch、sync_turn、on_memory_write、on_pre_compress、tool schemas 等 hook。
  • 通过 session key，让 CLI、gateway、API server 入口共享同一个记忆身份。

主要组成

  • agent/memory_provider.py：provider 抽象。
  • agent/memory_manager.py、context_engine.py、context_compressor.py：调度和上下文注入。
  • plugins/memory/*：ByteRover、Honcho、mem0、OpenViking、RetainDB、Supermemory 等实现。

Axiomate 缺哪一层

  • Axiomate 的 memdir 是好的本地默认实现，但不是 provider ABI。
  • 没有统一的 MemoryProvider/ContextEngine lifecycle 和 session-key 传递。
  • 无法让第三方 memory 后端不改 core 就接入，也无法让 Curator 跨后端跑批。

为什么重要：记忆系统真正难点不是“存几条文本”，而是检索时机、压缩前后同步、跨入口身份、后台整理和工具化访问。Provider 化是这些能力的前置条件。

4. Provider/Transport plugin ABI：Axiomate 抽象不弱，但还没 runtime 可插拔

Hermes 把 provider 差异、transport 协议、fallback 与凭据轮换拆成了可组合 runtime。
中高优先级

Hermes 这套系统做什么

  • Transport ABC 负责不同 API 协议：Anthropic、Chat Completions、Responses、Bedrock、Codex 等。
  • ProviderProfile 负责 provider-specific message shaping、extra body、model fetch、thinking 参数等差异。
  • ordered fallback + credential pool 负责主 provider 失败、额度耗尽、凭据轮换后的连续运行。

主要组成

  • agent/transports/base.py 与多个 transport 实现。
  • plugins/model-providers/*：Anthropic、Bedrock、Gemini、OpenRouter、DeepSeek、Qwen、Copilot 等 profile。
  • agent/credential_pool.py、fallback config、model catalog 与 auth/setup 逻辑。

Axiomate 缺哪一层

  • 已有 protocol-neutral provider 类型、registry、retry/cache middleware，这是好底座。
  • provider 仍主要是源码内实现，不是第三方可插拔 profile。
  • fallback chain 与 credential pool 还不是显式 runtime 状态。

为什么重要：模型接入增长到一定规模后，风险不在“会不会调 API”，而在 provider 特例、凭据耗尽、fallback 后状态一致性和扩展边界。Axiomate 应抄架构，不应先抄 adapter 数量。

5. TUI Session Orchestrator：Hermes 新增的是多会话控制面，不只是 UI

2026-05-17 的 TUI session orchestrator 让 Hermes 可以在 TUI/gateway 内管理多个活跃 agent session。
中高优先级

Hermes 这套系统做什么

  • 把 TUI 从“当前一个 REPL”升级为 session orchestrator：切换、创建、恢复、压缩、状态显示、slash worker。
  • 支持 busy input mode（queue/steer/interrupt）、active session switcher、completion queue 和后台通知。
  • goal continuation、slash commands、tool progress、browser connect 等都能通过 TUI gateway 统一调度。

主要组成

  • tui_gateway/server.py：会话状态、RPC、slash worker、callbacks、completion queue。
  • ui-tui/src/components/activeSessionSwitcher.tsx 等 TUI 组件。
  • tests/test_tui_gateway_server.py 和多组 UI/session lifecycle 测试。

Axiomate 缺哪一层

  • Axiomate 的终端体验强在当前 session、rewind、goal indicator。
  • 缺多会话 orchestrator、session switcher、后台 session completion queue。
  • 缺 queue/steer/interrupt 这类跨会话输入策略的 runtime 抽象。

为什么重要：当 agent 能跑 goal、cron、subagent 后，用户需要同时观察和接管多个工作流。TUI orchestrator 是从“单会话工具”到“本地 control plane”的关键一步。

6. Service supervision / gateway delivery：Hermes 有 daemon 级生命周期

Hermes 5/21 之后的 s6 / ServiceManager 系列提交，把 gateway 从脚本升级为可监督服务。
中高优先级

Hermes 这套系统做什么

  • 管理 gateway/dashboard/subagent 等服务的启动、停止、重启、容器重启后的 reconciliation。
  • 让 messaging gateway 的 delivery、platform adapter、session context 不依赖一个前台 CLI 活着。
  • 在 Docker/container 场景用 s6 作为 PID 1 和 per-profile supervision 层。

主要组成

  • hermes_cli/service_manager.py、host wrappers、S6ServiceManager。
  • docker/s6-rc.d/*、container boot/reconciliation 逻辑。
  • gateway/delivery.py、gateway/session_context.py、platform adapters。

Axiomate 缺哪一层

  • Axiomate 当前主要是 terminal/IDE-first，没有 daemonized gateway。
  • 缺 service lifecycle、process supervision、restart reconciliation、delivery retry/notification 这层。
  • 若不做 messaging gateway，也仍可抽轻量 local background supervisor 给 cron/goal 使用。

为什么重要：runtime durability 不只是在 JSONL 里有状态，还要能在进程、容器、网络入口重启后恢复执行与通知。Axiomate 的状态原语强，但 process-level 管理还没成系统。

7. MCP Catalog / Skills Hub：Hermes 把扩展发现和新鲜度做成产品面

Hermes 2026-05-25/26 的 MCP catalog 与 skills-hub 更新，强化的是“可发现、可选择、可健康检查”的扩展治理。
中优先级

Hermes 这套系统做什么

  • 让 MCP 不只是手写 config，而是有 approved catalog、interactive picker、optional MCP manifests。
  • skills-hub 页面显示 catalog source、freshness badge，并用 watchdog cron/CI 检查 freshness。
  • 把“扩展能不能被找到、能不能安装、是否过期”纳入产品体验。

主要组成

  • hermes_cli/mcp_catalog.py、mcp_picker.py、optional-mcps/*/manifest.yaml。
  • tools/skills_hub.py、skills index workflows、website skills 页面。
  • skill bundle symlink reject、install lock path guard 等安全配套。

Axiomate 缺哪一层

  • Axiomate 有 marketplace、MCP、skills 注入，分发基础在。
  • 缺 approved catalog + picker + freshness/health check 这层治理。
  • 缺把扩展安全校验、来源、版本新鲜度直接暴露给用户的产品面。

为什么重要：当插件/skills 数量增长时，用户体验瓶颈会从“能不能装”变成“该装哪个、是否可信、是否过期”。这是生态治理，不是单个工具能力。

8. Security / guardrails：Hermes 的攻击面更大，但 guardrails 更系统

Hermes 5/10 后大量 hardening 提交会影响评估，尤其是 gateway、plugins、skills 和外部输入。
中高优先级

Hermes 这套系统做什么

  • 把 promptware、恶意 memory、tool result 注入、文件读取、配置/env 泄漏、webhook/auth 都纳入默认防护面。
  • 对大平台 runtime 的高风险边界做 fail-closed：OAuth secrets、dashboard websocket、webhook signatures、skill bundle symlink。
  • 用测试覆盖历史漏洞回归，而不是只靠文档提示。

主要组成

  • tools/threat_patterns.py、agent/tool_dispatch_helpers.py、tool-result delimiters。
  • tools/file_tools.py、tools/skills_hub.py、OAuth/credential store hardening。
  • gateway platform auth、webhook signatures、dashboard origin/host checks。

Axiomate 缺哪一层

  • Axiomate 有 verify skill、fileHistory snapshot、tool error surface。
  • 缺统一 threat pattern 库、tool-result delimiter、memory load-time scan、外部输入协议级 guard。
  • 缺把 Edit/Write/Bash/Web/MCP 的安全策略统一成 guardrail layer。

为什么重要：本地 coding agent 也会读取 untrusted repo、网页、MCP 输出和 shell 日志。guardrails 不是 gateway 专属，Axiomate 的本地强执行反而更需要默认防护。

  • tools/file_operations.py、tools/file_tools.py、tools/fuzzy_match.py。
  • tests/tools/test_line_ending_preservation.py、test_patch_failure_tracking.py。
  • 相关提交：6bd0be30 patch indentation / CRLF / per-file failure escalation。

Axiomate 缺哪一层

  • Axiomate 有 checkpoint/rewind 和工具错误 UI，但编辑工具的 failure taxonomy 还可更细。
  • 缺“每个文件为什么没改成功”的可机器处理结果。
  • 可把失败语义接入 ValidationErrorsList 与自动 verify。

为什么重要：checkpoint 能撤销坏结果，但 patch reliability 能在坏结果落盘前减少失败。二者不是替代关系。

10. Goals 的下一步：不是实现，而是并入 kanban/cron/subagent runtime

Axiomate goal 已落地；现在缺的是让 goal 成为其他 runtime 原语的共同锚点。
已补齐后的下一步

Axiomate 已完成

  • /goal、/subgoal、judge、persist、compact resume、footer status。
  • turn budget、parse-failure config、Ctrl+C pause、continuation reason。
  • 这已经是 first-class control primitive。

Hermes 仍强在哪里

  • goal 与 queue/steer/permission/TUI/gateway 的联动更完整。
  • goal continuation 已在 TUI gateway 路径里处理。
  • 长任务控制原语与 kanban/cron/subagent 更接近同一 runtime。

Axiomate 下一步

  • goal 与 swarm task、cron job、background agent、queue/steer、permission policy 统一。
  • subagent 完成后回写 goal/subgoal verdict 和 evidence。
  • checkpoint/rewind 后 goal 状态与任务 board 状态保持一致。

11. Checkpoint + Rewind：Axiomate 是优势项，但还要升到 runtime 层

Axiomate 当前强于 Hermes 文件回滚轴，并新增 conversation 轴。
优势项

Axiomate 强项

  • File tab + Conversation tab 双轴脱钩。
  • conversation head JSONL 持久化、未来节点 dim、双游标。
  • pre-rewind 安全网、hash-chain、event-aligned stats、1GB/30d 产品化默认值。

Hermes 仍值得学的点

  • checkpoint 与 gateway/restart-pending/process-level resume 的测试和路径。
  • 让恢复能力服务后台 agent、gateway delivery、TUI session，而不是只服务交互 REPL。

下一步边界

  • 不需要回头补 Hermes 只有的 file rollback 体验，Axiomate 已超。
  • 需要把状态机制给 daemon/gateway/cron/subagent 用。
  • 补 runtime resume 测试族：goal、conversation head、file checkpoint、pending task。
Priorities

Axiomate 未来最该补的 10 件事

1. 把 swarm 升级成 durable kanban runtime

  • 先做 task state machine（todo/ready/in-progress/blocked/done）+ worker lease/heartbeat/reclaim。
  • 接 worker-initiated block、claim/ownership、diagnostics、hallucination gate。
  • 复用现有 4 后端 swarm，不重写 worker 执行层。

2. 做 Curator + background review fork

  • 先 read-only rubric report：本轮是否该更新 memory/skill/prompt。
  • 再加 per-run artifact：run.json + REPORT.md。
  • 最后才做 archive/promote 与 skill/memory 自动改写。

3. 抽 MemoryProvider + ContextEngine ABC

  • 把 memdir 变成 reference provider。
  • 定义 session-key 与 cross-project/session resume 的传递规则。
  • 先做接口稳定性，再接外部 vendor。

4. 抽 ProviderProfile / Transport ABC + fallback + credential pool

  • 把现有 anthropic/openai/openaiResponses provider 包装成 profile。
  • 实现 ordered fallback chain 与 credential pool 状态持久化。
  • 避免用“多接模型”替代 runtime 可靠性。

5. 把 file + conversation + goal 状态升到 runtime resume

  • 新增 crash/restart 测试：active goal、paused goal、rewind head、pending tool、cron task。
  • 定义 background task 的 resume contract。
  • 让 checkpoint/rewind 不只服务交互 session，也服务 unattended agent。

6. 下沉 security / guardrails

  • 默认启用 tool-result delimiter、promptware pattern scan、untrusted markdown/XML guard。
  • 补 symlink/traversal/env/secret/OAuth/update bundle 防护。
  • 把失败上抛到 ValidationErrorsList 与 fileHistory snapshot delta。

7. 把 Goal 接进 kanban / cron / subagent / queue / permission

  • 不要再把 goal 当独立命令看；它应成为所有长任务的目标锚点。
  • 让 subagent 接收 goal/subgoal，完成后回写 verdict/context。
  • 增加 queue/steer 语义，但避免与用户抢占规则冲突。

8. 定义 plugin runtime hook surface

  • ProviderProfile、MemoryProvider、ContextEngine、BrowserProvider、ObservabilityProvider 五类先立接口。
  • marketplace/MCP 负责分发；hook surface 负责改 runtime 行为。
  • 现有内置实现全部作为 reference plugin。

9. Cron 闭环补 delivery/watchdog/result sink

  • Axiomate 已有 scheduler 原语；缺结果通知、超时、失败恢复。
  • 先支持 OS notification / webhook / local log sink，不必直接做 16 个 IM 平台。
  • 让 goal + cron 组合成 unattended task loop。

10. 明确 gateway 战略边界

  • 若不进 messaging-bot 赛道：集中放大本地执行、CDP、napi、rtk、IDE/terminal-first。
  • 若要进：必须接受 gateway/delivery/restart/auth/media/platform-account 的整套成本。
  • 不要在战略未定时复制 Hermes 的 web control plane 与 16+ platform adapters。
Copy / Avoid

哪些该直接抄，哪些不必追平

应该直接吸收的方法

  • Kanban 的 lifecycle / ownership / block / diagnostics / recovery UX。
  • Curator 的 rubric、per-run report、auxiliary model、archive/promote。
  • ProviderProfile / MemoryProvider / ContextEngine 这种 runtime ABI。
  • Transport ABC + ordered fallback + credential pool。
  • Security hardening 清单：promptware、symlink、env/secrets、markdown/XML、webhook/auth、tool-result delimiter。
  • Runtime resume 测试族，而不是只测单 session resume。

不必盲目追平的面

  • 16+ messaging platform adapters。它们是产品赛道选择，不是 agent runtime 必修课。
  • Web dashboard/control plane。先决定是否做平台 runtime，再决定前端面。
  • SaaS browser provider 作为主路线。Axiomate 的 CDP+napi 本地路线更稀缺。
  • 复制 30+ model adapters。先补 ABC/fallback/credential pool。
  • 把 rtk-axiomate 换成 conversation-level compression 思路。Axiomate 的优势是工具输出进上下文前就压。
边界判断： 若 Axiomate 的定位是“强本地执行型 coding agent + token 经济优先”，最优路线是复制 Hermes 的治理方法，而不是复制 Hermes 的全部平台入口。
Final Judgment

最终判断

Hermes Agent 当前仍是更成熟的平台 runtime：durable kanban、curator review、plugin ABI、gateway/delivery/restart、security hardening、TUI/session orchestrator 与 service supervision 形成了系统优势。2026-05-10 之后的提交没有推翻这个判断，反而让平台成熟度差距更明确。

Axiomate 的强项集中在 persistent goal、checkpoint/rewind、conversation rewind 持久化、双轴 picker、rtk-axiomate、本地 CDP/napi 执行线。这些是 Hermes 没有正面对位的强项。

因此新版结论是：Axiomate 不是“全面落后”，而是“本地执行和状态回滚强，runtime 治理与平台化不足”。下一步最该补的是 durable orchestration、curator、ABC/plugin hook、runtime resume 与 guardrails；不是重新实现 goal，也不是盲追 Hermes 的全部 gateway 平台。
