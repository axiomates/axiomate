  • Conversation rewind 持久化：head 状态写进 JSONL（一条 type:"head" entry），/exit 后 resume 进同一个 session 落点跟 rewind 时一致。Hermes 与 claude-code 上游均为内存截断、退出即丢，axiomate 在这条上是行业内首发持久化方案。

  • UX 反超：picker newest-first 排序（hermes 无对照）、自适应时间格式（just now / 5m ago / 绝对日期）、reason.ts codec 单一来源管理 commit subject/body 格式（picker / list / chooser 一致）。
  • 4 个 slash 命令：/checkpoints status / list / prune --force / clear，跨 project 视图、TTL 与 size guardrails、orphan workdir reclaim、跨进程 race-safe。底层走 git shadow store（~/.axiomate/checkpoints/store/），与 hermes 同形。

  • 盲移植 Python 心智：Hermes 在 Python 生态有些 hack（lazy import / defer feasibility / sys.meta_path finder）在 Bun 单文件下无意义。
  • 稀释 rtk-axiomate 这条 token 经济线：Hermes 的 rtk init --agent hermes 是用户级集成，Axiomate 的“默认 bundle + auto-pin fork + Bash 深集成”是产品级护城河，不应换成 conversation-level 压缩范式。
若 Axiomate 战略是"强本地执行型 coding agent + token 经济优先"，应优先复制 Hermes 的治理方法（lifecycle/curator/ABC），同时保住并放大 rtk-axiomate 这条 pre-LLM 压缩护城河 + 已全面反超的 file checkpoint + conversation rewind 双轴系统；若要做平台 runtime，则必须接受这是一条更重的工程路线，并明确投入 gateway/delivery/restart 的成本。
Conclusion

最终判断

Hermes Agent 的领先点不是“多几个工具”，而是把 agent 做成了更像 runtime 的系统：durable kanban、curator review、checkpoints v2、provider/memory plugin ABI、跨入口 session continuity——这些是 platform runtime 工程的真正面。

Axiomate 的强项则集中在本地执行：6 个 Rust napi 子包、CDP browser bridge、rtk shell 过滤、4 后端 swarm、协议中性 LLM 抽象与 fileHistory snapshot——这是不该被稀释的护城河。

所以最准确的结论不是"Axiomate 全面落后"，而是：在 local execution 这条战线 Axiomate 不弱、部分项占优；file checkpoint 系统在 2026-05-24 大重构后反超 hermes，conversation rewind 持久化 + 双轴 picker 在 2026-05-25 完成、是行业内首发方案。但若以 runtime 治理与 platform 成熟度衡量，当前最该优先补的是 durable orchestration、persistent goals、curator review、memory provider 化与 transport ABC——这些治理方法，而不是工具数量。
