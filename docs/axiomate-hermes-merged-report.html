Executive Judgment

结论摘要

Axiomate 已经实现 persistent goals。 代码与文档显示它具备 Ralph-loop 风格 /goal：GoalManager、JSONL goal-state 持久化、turn-end judge、/subgoal 附加准则、Ctrl+C pause、turn budget、parse-failure backstop、footer GoalIndicator、/resume/--continue/compact resume 状态恢复，以及用户文档 docs/user/goals_zhcn.html。这是一等控制原语，不是待补功能。

API harness 和 file harness 都不再是 Axiomate 的基础短板。 05-27 之后的代码和文档显示，API 失败恢复已收敛到 observe → decide → execute；file harness 已覆盖 read-before-write、stale guard、registry/lock、file-tool strict atomic write、BOM/CRLF、failure taxonomy、edit/read escalation，并在 2026-06-01 修复 Write/Edit resume reconstruction、atomic helper scope、NotebookEdit mtime fallback、file-harness throw runner 边界、Bash simulated sed guard、registry/path lock alias、path-lock reentry fail-fast 与 killed/failed subagent reminder。随后第二轮烧测又修复了空文件 Edit validation 提前放行和 notebook Read offset/limit 状态误标两个边界问题。当前结论是核心 file harness 完成；不做 file harness telemetry/UI 统计，剩余只是已接受的跨进程/任意 shell/硬链接边界和低优先级开发观察。

Context/memory 的基线必须按 Claude Code 2.1.88 切开。 Axiomate fork 自 Claude Code CLI 2.1.88；本次参考的 C:\public\workspace\claude-code-sourcemap README 说明该仓库从公开 npm 包 @anthropic-ai/claude-code 的 cli.js.map 还原，版本为 2.1.88。Axiomate 的 memdir、extractMemories、SessionMemory、autoDream、compact 等核心 context/memory 系统仍是这套基线的延续，主要差异是品牌、路径和 Anthropic 私有 gate 的删减。因此报告里的“已有什么”应理解为继承了 2.1.88 之前的 Claude Code 系统；2.1.88 之后 Claude Code 不再有可用开源基线，只能通过官方文档、博客和行为描述补差。

Hermes 的领先仍然真实，但领先点转为 runtime 与运维厚度。 它不是因为单次 API retry 更强而领先，而是把 long-running agent 做成平台 runtime：durable kanban、Curator/background review、MemoryProvider/ProviderProfile/ContextEngine plugin ABI、gateway/delivery/restart、s6/service supervision、security hardening、TUI session orchestrator 与跨入口 session continuity。05-27 之后新增的 prompt-size diagnostic、/compress here、atomic file operations、MCP isolation、FTS degradation 和 kanban breaker 修复继续强化这些面。

Axiomate 的下一版不应复制 Hermes 的 RAG memory 插件路线。 官方 Claude Code memory 文档显示 Claude Code 依赖 CLAUDE.md、作用域 memory 文件和递归导入；Managed Agents memory/dreams 文档把 memory stores 挂载成 filesystem 目录，再用 dreams 异步合并重复、替换陈旧和提炼新洞见。对 coding agent 来说，这比默认向量化切碎代码更符合 Axiomate 的本地执行路线。

一句话判断： 当前差距不是“缺 API harness / 缺 file harness / 缺 goal / 缺 checkpoint”，而是“需要从 Claude Code 2.1.88 的继承基线继续补 context/memory 生命周期”。Axiomate 下一轮优先级应是 Claude-style memory + dream + prompt-size/compression + session search，然后做 MCP/process hygiene、guardrails 和轻量 durable task runtime；file harness 已从主线待补项移出。
Axiomate /goal 已实现 API O-D-E + /doctor 已落地 File harness 已烧测完成 Checkpoint + conversation rewind 仍是优势项 rtk-axiomate 是产品级 token 经济 Hermes 5/27 后增强 context/ops hygiene 最高风险：context lifecycle 没有产品级闭环

2. Durable 多代理：swarm primitive 还不是 kanban runtime

Hermes 的 Kanban 不是“一个看板 UI”，而是多代理长任务的持久化控制平面。
中高优先级

Hermes 这套系统做什么

  • 把长任务拆成可持久化 card，而不是只把 prompt 扔给一个 worker。
  • 记录任务状态、依赖、claim、block、promote、done，让多个 agent 可以围绕同一个 board 协作。
  • worker 出错、幻觉 claim、卡住或需要人工输入时，可以通过 diagnostics / hallucination gate 把状态上抛。

主要组成

  • plugins/kanban/：runtime 插件入口。
  • hermes_cli/kanban*.py：CLI、DB、decompose、diagnostics、swarm、specify 等控制面。
  • TUI/gateway 集成：让 board 状态不是只存在于一个终端 turn 里。

Axiomate 缺哪一层

  • 已有 swarm/backends、workerAgent、teammateMailbox，能启动 worker。
  • 缺 durable board state、task ownership、lease/heartbeat/reclaim、worker-initiated block。
  • 缺把 goal/subgoal 绑定到 worker card 的统一任务语义。

为什么重要：没有 board + ownership，长任务一旦 worker crash、漂移或被用户抢占，就只能靠 transcript 猜状态；有 kanban runtime 后，cron、subagent、goal、resume 都能复用同一套任务锚点。
