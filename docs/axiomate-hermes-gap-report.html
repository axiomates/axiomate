Executive Summary

结论摘要

Axiomate 不是一个“能力很少”的 agent 客户端。它已经有较完整的本地 agent runtime： 多模型、MCP、插件市场、session search、auto-memory、worktree、team/swarm primitive、cron、 desktop automation、voice dictation 都在代码里落地。

但 Hermes 的领先点不在于基础 tool 数量，而在于它把这些基础块继续工程化为 长期运行、可恢复、可扩展、可自治、跨入口连续 的系统。 换句话说，Axiomate 更像“强本地 agent 客户端”，Hermes 更像“agent operating surface”。

一句话判断： Axiomate 不是全面落后。如果目标是继续做最强本地电脑操作型 CLI agent，它已经有真实竞争力； 如果目标是和 Hermes 正面竞争“agent 平台”层，则当前最大的缺口是 durable orchestration、 self-improvement、gateway/runtime durability、以及深度插件化。
Axiomate 已具备：MCP / 插件市场 / 会话搜索 / 自动记忆 / worktree / computer-use Hermes 明显领先：Durable multi-agent / Gateway / Curator / Checkpoints v2 当前最大风险：把“本地执行优势”误判成“平台层也已经对齐” 竞争方向建议：从本地强执行升级为可恢复可演化的 agent runtime

What Axiomate Already Has

先澄清：Axiomate 已有的关键能力

本地 agent runtime 已成型

  • agent/src/tools.ts 暴露 Agent、Task、MCP、Cron、SessionSearch、Skill、Worktree 等工具。
  • agent/src/main.tsx 包含 session restore、plugin loading、MCP、telemetry、worktree、swarm 初始化。
  • README 明确支持多模型、多 provider、voice、web search、plugins、MCP。

多代理不是空壳

  • TeamCreateTool 有 team file、leader session、task dir、teammate registry。
  • SendMessageTool 已有 mailbox、broadcast、shutdown request / response 等协作协议。
  • utils/swarm/ 下有 tmux、iTerm、in-process backend。

记忆与检索已有可用实现

  • SessionMemory 在后台抽取当前会话摘要。
  • extractMemories 在会话结束时把 durable learning 写入 auto-memory。
  • SessionSearchTool 支持 recent mode、本地检索和可选 LLM summary。

模型接入不是短板

  • 已有 protocol-neutral provider interface、provider registry、stream/request adapter。
  • 有 verifyConnection、token counting、non-stream fallback、stall detection 与较多单测。
  • 短板主要在 provider surface 的平台化，而不是核心抽象本身。

异常恢复已有体系

  • 已有结构化 error classifier、retry/fallback、thinking disable、max_tokens adaptation。
  • 请求级错误恢复已经有较强工程意识。
  • 真正待补的是跨 session / worker / runtime 的系统级恢复。

Computer-use 是护城河

  • 独立的 browser-bridge-axiomate 与 computer-use-mcp-axiomate 子工程。
  • 有 CDP page refs、UIA/AX enumeration pipeline、SoM、allowlist 与 sub-gates。
  • 这块应继续强化，而不是视作需要向 Hermes 补课。
Detailed Gaps

详细差距分析

1. Durable Multi-Agent Orchestration

Hermes 把多代理做成了“有调度、有租约、有恢复”的系统；Axiomate 目前仍停留在高级原语层。
最高优先级差距

Hermes 已有

  • Durable kanban board
  • Heartbeat / reclaim / zombie detection
  • Per-task retry budget
  • Hallucination recovery
  • Task ownership enforcement on destructive tools

Axiomate 现状

  • 能创建 team、spawn teammate、message passing
  • 有 task dir 和 team file
  • 有 tmux / iTerm / in-process backend
  • 但没有 durable board、租约、自动回收、全局故障治理

为什么重要

  • 这是“多代理演示”与“多代理生产系统”的分水岭。
  • 没有 worker lifecycle 治理，多代理一旦长时间运行就会失控。
  • 这也是 Hermes v0.13.0 最大的工程优势之一。
Hermes 证据： README.md: "Delegates and parallelizes ... Spawn isolated subagents for parallel workstreams" RELEASE_v0.13.0.md: "Multi-agent Kanban — durable multi-profile collaboration board" RELEASE_v0.13.0.md: heartbeat / reclaim / zombie / retry-cap / task ownership / hallucination gate Axiomate 证据： agent/src/tools/TeamCreateTool/TeamCreateTool.ts agent/src/tools/SendMessageTool/SendMessageTool.ts agent/src/utils/swarm/backends/*.ts

2. Persistent Goals, Steering, Queueing

Hermes 已经把“长期目标不丢失”和“运行中纠偏”做成一等能力，Axiomate 暂未看到等价原语。
高优先级

Hermes

  • /goal 让 agent 跨 turn 保持目标锁定。
  • /steer 可以在运行中追加引导。
  • ACP 支持 /queue，让 IDE/外部端对 in-flight session 施加后续动作。

Axiomate

  • 有 task/todo/agent/team primitive。
  • 但未发现“跨 turn 强目标绑定”或“运行中 steer”的一等接口。
  • 目前更依赖当前上下文和 agent prompt discipline。
这类能力看起来不像“新功能”，但它直接影响 agent 的可控性。没有 persistent goal， agent 一旦经过压缩、resume、handoff、多代理分工，就更容易偏航。

3. Self-Improvement Loop 与 Curator

Axiomate 已经有自动记忆，但 Hermes 已经走到自动维护 skill 库、自我整理、自我更新的闭环。
高优先级

Axiomate 已有

  • SessionMemory：周期性更新当前会话记忆文件。
  • extractMemories：会话末尾提炼 durable learning。
  • README 里有 /remember 技能。

Hermes 已有

  • Background review fork
  • Rubric-based review prompt
  • Active-update bias
  • Curator 周期性归档、合并、裁剪 skill
  • Per-run curator report 与 status 排名

差距本质

  • Axiomate 的 memory 更像“沉淀内容”。
  • Hermes 的 curator 更像“沉淀并重构 agent 自身能力”。
  • 后者更接近 self-improving agent 平台叙事。

4. Runtime Durability：Checkpoint、Restart Resume、Long-Running Hygiene

Axiomate 能 resume transcript；Hermes 则把重启恢复和 checkpoint 清理做到了 daemon / gateway 级别。
高优先级

Axiomate

  • sessionStorage.ts 与 sessionRestore.ts 已实现 transcript-based resume。
  • 有 worktree session restore。
  • 适合 CLI 恢复，但更偏前台交互场景。

Hermes

  • Gateway restart 后自动恢复 interrupted session。
  • Checkpoints v2 有 pruning、disk guardrails、shadow repo cleanup。
  • 更适合常驻进程和 unattended 运行。
如果 Axiomate 后续想承载 cron、守护型 agent、后台团队协作或外部入口，单纯 transcript resume 不够， 需要提升到 runtime durability 语义。

5. Gateway 与跨平台会话连续性

这是 Hermes 与 Axiomate 产品形态差异最大的部分。
平台级差距

Hermes 已形成的平台面

  • Telegram、Discord、Slack、WhatsApp、Signal、Email 等统一 gateway。
  • 同一 agent runtime 跨入口连续对话。
  • 平台 allowlist、approval UX、media routing、restart notification。

Axiomate 当前边界

  • 主要是 terminal-first、本地桌面自动化客户端。
  • 没有看到 messaging gateway、跨平台适配层或 session continuity bus。
  • 这不是小功能，而是产品边界不同。

6. Plugin Surface 深度不足

Axiomate 的插件系统更像分发与集成；Hermes 的插件系统更像 runtime surface。
高优先级

Axiomate

  • 有 marketplace、安装、更新、启停、依赖解析、内建插件注册。
  • 插件可提供 skills / hooks / MCP servers。
  • 从 builtinPlugins.ts 与 plugin 操作代码看，管理面较完整。

Hermes

  • 插件可扩 memory provider、context engine、model provider、platform、dashboard、image gen、TTS。
  • 还有 transform hook，例如 transform_tool_result、transform_llm_output。
  • 插件不只“加工具”，还可改 runtime 行为。

差距含义

  • Axiomate 目前偏“插件生态”。
  • Hermes 偏“平台内核可插拔”。
  • 若未来要接第三方 provider/platform，这层会变成关键瓶颈。
这里“更系统”到底指什么： 不是指 Hermes “工具更多”，而是指它把工具定义、可用性判断、toolset 组合、跨 CLI/gateway 入口复用、 插件挂接、MCP 暴露、权限与动态 schema 管理都收敛进了统一的工具系统。Axiomate 也有统一的 Tool 抽象和工具池装配，但 Hermes 的工具系统进一步承担了 平台分发层 的职责，而不只是本地 agent 调用层。
Axiomate 工具系统： - Tool.ts 提供统一 Tool / ToolUseContext / permission / progress 结构 - tools.ts 负责 built-in tool pool + MCP pool 装配 - 更偏“单 runtime 内的统一工具接口” Hermes 工具系统： - tools/registry.py 负责自注册 discovery - model_tools.py 负责 schema 提供、调用编排、缓存与 async bridge - toolsets.py 负责按场景/入口组合工具集合 - 工具系统同时服务 CLI、gateway、batch、ACP、dashboard、plugins - 更偏“跨入口、跨平台、跨插件的工具分发与运行系统”

7. Execution Backends 与远程运行面

Axiomate 强在本机；Hermes 强在“agent 住在哪里都行”。
中高优先级

Axiomate

  • 本地 Bash/PowerShell、worktree、桌面 automation 很强。
  • 但没有看到 docker/ssh/modal/daytona/vercel sandbox 这种统一执行后端抽象。

Hermes

  • 支持 local、Docker、SSH、Singularity、Modal、Daytona、Vercel Sandbox。
  • 这让 gateway、cron、background agents 可以摆脱用户本机。

8. Session Search 与 Memory Infra 的成熟度差距

Axiomate 有可用功能，但 Hermes 在底层索引与 provider 化上更进一步。
中等差距

Axiomate 亮点

  • SessionSearchTool 提供 recent mode 和 search mode。
  • 支持 project-scoped 搜索与可选辅助 LLM 摘要。
  • 对 CLI 用户已经很实用。

Hermes 更强的点

  • SQLite FTS5 + CJK trigram index。
  • 索引 tool_name 与 tool_calls。
  • Memory providers、Honcho user modeling、API session key。

判断

  • 这不是“有没有 session search”的差距。
  • 这是“有没有把 session search 做成 memory platform”的差距。

9. Cron / Unattended Automation 深度不足

Axiomate 已有 schedule 原语，但 Hermes 已经把 cron 做成 agent delivery runtime。
中等差距

Axiomate

  • 工具集中有 CronCreateTool / CronDeleteTool / CronListTool。
  • README 里有 /loop 技能。

Hermes

  • cron 与 gateway 平台投递打通。
  • 支持 no_agent script-only watchdog mode。
  • 更适合运维类无人值守任务。

10. Research / Trajectory Tooling

Hermes 兼顾 agent 产品和 agent research；Axiomate 目前更像终端产品。
中等差距

Hermes

  • Batch trajectory generation
  • Trajectory compression
  • 更适合训练下一代工具调用模型或做 agent 评估

Axiomate

  • 目前没有看到等价的数据生产/压缩基础设施。
  • 如果未来面向内部模型训练，这会成为能力短板。

11. Dashboard / Web Control Plane

Hermes 有 dashboard、analytics、plugin page、profiles；Axiomate 当前没有同等级控制面。
中等差距

这部分对于纯 CLI 用户不是刚需，但一旦要做多入口、常驻服务、队列化任务、插件可视化管理， Web control plane 会从“附加界面”变成“必要运维面”。

12. 写后自动校验与更强 guardrails

Axiomate 有 `/verify`，但 Hermes 把部分校验下沉到了 write/patch 生命周期。
中等差距

Hermes 在 v0.13.0 新增了 post-write delta lint，让某些语法错误在文件写入后立刻暴露。 Axiomate 的验证能力目前更多体现在 skill 层，而不是 runtime 默认 guardrail。

Hermes Momentum

最近三个月最重要的 11 个新重大功能

Top 1. Durable Multi-Agent Kanban

发布时间：2026-05-07 · 来源：v0.13.0
最高价值

这是最近三个月最重要的 Hermes 能力升级。它把多代理从“可 spawn 的 worker”升级成了 “带状态机、带租约、带回收、带失败治理”的持久化看板系统。

  • Heartbeat、reclaim、zombie detection、retry budget 构成了真正的 worker lifecycle 管理。
  • Hallucination recovery 说明 Hermes 不只关注 happy path，还在治理 agent 编造任务状态的问题。
  • 这类能力直接决定多代理能不能做长时间生产任务。

Top 2. /goal 持久目标锁定

发布时间：2026-05-07 · 来源：v0.13.0
高价值

/goal 的意义不是“又多一个命令”，而是把长期目标维持做成 agent runtime 原语。 这能显著缓解上下文压缩、跨 turn、分工协作后的目标漂移。

Top 3. Gateway 重启后自动恢复会话

发布时间：2026-05-07 · 来源：v0.13.0
高价值

这是 Hermes 迈向常驻 agent runtime 的关键特征。只要 agent 不再被限定为前台 CLI， restart recovery 就变成核心可靠性能力。

Top 4. Checkpoints v2

发布时间：2026-05-07 · 来源：v0.13.0
高价值

状态持久化重写，补上 pruning、disk guardrails、orphan shadow repo 清理，属于典型的 harness engineering 升级，不花哨，但非常决定系统上限。

Top 5. Providers Are Now Plugins

发布时间：2026-05-07 · 来源：v0.13.0
平台级

把 inference provider 做成插件化表面，意味着 Hermes 在架构上正式把模型接入层从核心仓库中解耦， 更接近平台内核，而不是静态 provider 集合。

Top 6. Autonomous Curator

发布时间：2026-04-30 · 来源：v0.12.0
高价值

Curator 让 Hermes 开始具备“自己维护自己的技能库”的能力。它不只是存经验，而是定期做 skill grading、 consolidation、pruning 和 report 输出。

Top 7. 升级后的 Self-Improvement Loop

发布时间：2026-04-30 · 来源：v0.12.0
高价值

Hermes 把背景 review fork 从“会存东西”升级成“有 rubric、有 active-update bias、有受限工具集、有干净上下文”的闭环。 这类升级直接提升长期学习质量。

Top 8. 新 Ink TUI + Python JSON-RPC 后端

发布时间：2026-04-23 · 来源：v0.11.0
基础设施级

这是交互面的大重构。价值不只是 UI 更好，而是形成了清晰的前后端分层，为后续 dashboard 复用、 slash command worker、observability overlay 等能力铺路。

Top 9. Transport 抽象层 + Native AWS Bedrock

发布时间：2026-04-23 · 来源：v0.11.0
架构级

Hermes 把 provider transport 从主 agent loop 中抽离，形成独立 transport abstraction。 Native Bedrock 支持只是结果，更重要的是这一步让 provider 扩展与协议演进不再堆在主循环里。

Top 10. 扩展后的 Plugin Surface

发布时间：2026-04-23 · 来源：v0.11.0
平台级

插件开始可以注册命令、拦截 tool、改写 tool result、改写 terminal output、扩 dashboard tab。 这标志着 Hermes 的插件系统从“加内容”走向“改 runtime 行为”。

Top 11. Nous Tool Gateway

发布时间：2026-04-16 · 来源：v0.10.0
产品级

这个功能的价值在于“降低托管工具门槛”。用户不需要额外配置多套 API key， 就能通过 Nous Portal 订阅使用 web search、image generation、TTS 和 browser automation。

Recent Heat

最近三个月 Hermes 社区与发布热点

社区高热主题

  • 多代理架构与协作工作流：高讨论度 feature 方向集中在 orchestration、specialized roles、resilient workflows。
  • 长文本与输出截断：社区明显关心 long-form response truncation、tool/stream continuity。
  • Provider 可用性与认证路径：如 Anthropic subscription auth、MiniMax 529 overload 这类问题评论很多。
  • 外部记忆与持久化：memory provider / external memory 相关需求热度持续存在。
  • TUI 交互质量：光标、闪烁、terminal rendering 这类体验问题也能拿到高讨论。

发布日志热点

  • v0.11.0 到 v0.13.0 的主线非常清晰：transport/provider abstraction → curator/self-improvement → durable orchestration/restart recovery。
  • Hermes 最近不是在“补零散工具”，而是在持续加固 runtime、multi-agent、memory、platform、plugin surfaces。
  • 这与社区热度是相互印证的：用户关心的不只是“能不能做”，而是“长任务稳不稳、跨入口能不能恢复、模型/provider 路由是否可靠”。
这意味着 Hermes 当前的产品与社区重心，已经明显从“单机 CLI agent”转向 可持续运行的 agent runtime / platform。Axiomate 若只从功能数量角度对比， 会低估这种方向性差异。
Roadmap

未来 3 个月最该补的 5 件事

1. 给多代理加上 durable worker lifecycle

  • 把现有 team + mailbox + tasks 升级成带 heartbeat、lease、timeout、retry 的 worker 模型。
  • 引入 task state machine，至少覆盖 queued / running / blocked / failed / completed。
  • 补齐 crash recovery 和 orphan worker reclaim。

2. 做 persistent goal / steer，而不是继续堆 prompt 纪律

  • 增加 session-level goal pinning，让 agent 经历压缩、resume、handoff 后仍保留任务锚点。
  • 支持运行中 steer，允许用户在不打断整轮任务的情况下修正方向。
  • 这是提升 agent 可控性的最短路径。

3. 从 memory extraction 走向 review loop

  • 把 SessionMemory 和 extractMemories 上升为背景 review fork。
  • 先做 skill update 建议，再做 curator 式自动归档/合并。
  • 不要一开始就做庞大 memory provider 生态，先把“会学”做扎实。

4. 下沉写后校验与 runtime guardrails

  • 在 write/edit/patch 后自动做语法、结构或最小 lint 校验。
  • 让常见失败在工具生命周期内暴露，而不是等用户自己发现。
  • 这会显著抬高 agent 可靠性下限。

5. 决定是否做平台，不要半做不做

  • 如果要进入 Hermes 赛道，就要认真做 gateway / restart durability / remote execution。
  • 如果不做，就继续深挖本地 computer-use + browser bridge + coding orchestration，形成明确差异化。
  • 最危险的状态是同时想做两边，但两边都只做到 60 分。
Subsystem Review

四个关键子系统 Review

模型接入

判断：Axiomate 并不弱，成熟度接近 Hermes，但生态扩展面稍弱。
Axiomate 表现强

Axiomate 优点

  • 有 protocol-neutral LLMProvider 抽象。
  • 有 provider registry、request adapter、stream adapter、usage mapper、stall detection。
  • 有 side query、token counter、non-stream fallback、verifyConnection、prompt cache 相关逻辑。
  • 测试覆盖明显优于“脚手架级”实现。

Hermes 更强的点

  • Transport 层与 provider/plugin 生态更完整。
  • provider 种类更多，且已走向 ProviderProfile 插件化。
  • 对 gateway、cron、aux models、dashboard picker 的联动更强。

评价

  • Axiomate 不是“模型接入薄弱”。
  • 它的弱点不在核心抽象，而在 provider surface 不够平台化。
  • 如果只看本地 CLI agent，这块已经相当合格。
Axiomate 关键文件： agent/src/services/api/provider.ts agent/src/services/api/providerRegistry.ts agent/src/services/api/llm.ts agent/src/utils/model/*.ts Hermes 关键文件： agent/transports/*.py providers/base.py plugins/model-providers/*

错误与异常处理

判断：Axiomate 这一块做得比想象中成熟，和 Hermes 的差距不算大。
Axiomate 表现强

Axiomate 已具备

  • 结构化 errorClassifier，区分 rate_limit、overloaded、auth、context_overflow、max_tokens_too_large 等。
  • withRetry 里有 foreground/background 分流、fallback、thinking disable、max_tokens adaptation。
  • 有 API error message 正规化、超时、媒体错误、prompt-too-long 解析。
  • 整体思路是“分类驱动恢复”，不是字符串 if-else 混战。

Hermes 领先点

  • 错误处理已经穿透到 gateway、cron、plugin、transport、kanban worker 生命周期。
  • 更强的是“系统级恢复”，不只是 API 请求级恢复。
这里的差距不在“有没有错误分类器”，而在“异常恢复是否已经覆盖整个平台运行面”。 Axiomate 请求级恢复不错，但系统级恢复还不够深。

Computer Use / Browser

判断：这是 Axiomate 的明显优势区，不是短板。
Axiomate 优势项

Axiomate 强项

  • 独立的 computer-use-mcp-axiomate 与 browser-bridge-axiomate 子工程。
  • CDP browser bridge 使用 snapshot refs，不把网页交互退化成 OS 像素点击。
  • computer-use 侧有 UIA/AX enumeration pipeline、SoM、frontmost gate、allowlist、sub-gates。
  • 实现明显不是“套壳”，而是认真做了人机交互与安全约束层。

Hermes 现有叙事

  • 近三个月发布日志更强调 browser automation 作为工具面，而不是同级别的桌面控制工程。
  • Hermes 的主战场不在本地 computer-use 深水区。

评价

  • 如果用户价值在“本机完成复杂操作”，Axiomate 这里是护城河。
  • 这块不该被当成追 Hermes 的补课项，反而应继续放大优势。

Memory / Skill / 上下文工程

判断：Axiomate 有不错基础，但 Hermes 更系统，差距主要在闭环与平台化。
Axiomate 中等偏强，但 Hermes 更完整

Axiomate 已有

  • 技能加载、frontmatter、hooks、bundled skills、MCP skill builders。
  • AXIOMATE.md + memory files + user/system context 拼装。
  • session memory、extract memories、prompt suggestion、auto compact。
  • 上下文工程已经不是裸 prompt 拼接，而是有缓存、安全与压缩意识。

Hermes 更强

  • FTS5 session store、memory provider plugins、context engine plugins、curator、自学习闭环。
  • 压缩、auxiliary models、review fork、session durability 的耦合更深。
  • 更接近“knowledge + memory runtime”，而不是“记忆功能集合”。

评价

  • Axiomate 这块已经超过很多普通 agent 客户端。
  • 但它还没完成从“上下文工程不错”到“自演化知识系统”的跃迁。
  • 这恰好是 Hermes 近两个月最重要的领先点之一。
四块里，真正该优先补的不是模型接入，也不是 browser/computer-use，而是 memory / skill / context 与 multi-agent orchestration 的系统闭环。
Roadmap

建议给 Axiomate 的三阶段补强路线

Phase 1：把现有本地 agent 变成“更稳定的长任务 runtime”

  • 给 team/swarm 增加 worker heartbeat、lease、retry、failure state。
  • 引入 persistent goal / steer 机制，先只覆盖 CLI 与 subagent。
  • 给关键写操作增加自动语法校验或 post-write verifier。
  • 把 session restore 从 transcript resume 提升到 task/runtime resume。

Phase 2：把 memory 升级成 self-improvement loop

  • 在 SessionMemory / extractMemories 之上增加 review fork。
  • 让 agent 能更新 skill 内容，而不只是写 memory note。
  • 增加 curator-like 周期任务，对 skills/memories 做归档与合并。
  • 建立 per-run report，便于观测 agent 自学习质量。

Phase 3：决定是否进入平台战场

  • 如果目标是 Hermes 同类平台，需新增 gateway、remote execution backend、dashboard control plane。
  • 如果目标仍是本地电脑自动化第一，则不必完整复制 gateway，而应继续强化 computer-use、verification、multi-agent coding、desktop workflows。
  • 此阶段最重要的是产品取舍，不是单纯追 feature parity。
Strategic View

战略判断：哪些差距必须补，哪些不必盲目追平

必须补

  • Durable multi-agent orchestration
  • Persistent goals / steer
  • Self-improvement review loop
  • Runtime durability / restart hygiene
  • 更深的 plugin runtime hook surface

不必机械追平

  • 20 个 messaging platform 适配数量
  • 完整 gateway 平台矩阵
  • 所有远程 execution backend
  • 研究型 trajectory pipeline
如果 Axiomate 的定位继续围绕“本地终端 + 桌面自动化 + coding agent”，最值得学 Hermes 的不是“接更多平台”， 而是它在多代理治理、长期任务恢复、自学习闭环上的工程方法；同时应主动放大 Axiomate 自己在 computer-use / browser / 本地执行面上的优势，而不是把这些误判成次要能力。

Final Take

最终判断

Hermes 的强项不是“有更多工具”，而是把 agent 做成了一个更像操作系统的 runtime：可恢复、可委派、可跨入口、可自我整理。 Axiomate 的强项则更集中在本地电脑操作、CLI 体验、coding/runtime 原语与执行能力。两者并不是完全同类，但如果要比较 harness engineering 成熟度，Hermes 当前明显领先。

因此，最准确的结论不是 “Axiomate 全面落后”，而是： Axiomate 在本地执行型 agent 上有竞争力，甚至在 computer-use / browser 上有优势；但在平台成熟度、长期运行能力、 多代理治理、self-improvement 闭环上明显落后于 Hermes。

Axiomate 最应该补的不是平台数量，也不是继续优先加模型接入或请求级错误处理，而是把现有的多代理、memory、resume、plugin、cron 这些基础能力继续工程化， 做成 durable、long-running、self-improving 的系统，同时继续放大本地 computer-use、browser bridge 与 coding orchestration 的优势。
