Hermes System

Hermes File Harness 由哪些层组成

1. 文件操作内核

  • tools/file_operations.py 定义 ReadResult、WriteResult、PatchResult、SearchResult。
  • _atomic_write 用同目录 temp file + rename，目标是半写入失败不污染原文件。
  • write_file / patch_replace 处理 mode、lint/LSP delta、BOM、line ending、diff 输出。

2. 工具层 guard

  • tools/file_tools.py 负责 path resolution、sensitive path、cross-profile、read dedup、stale warning、patch failure counter。
  • 读文件有大文件字符上限、重复读取阻断、device/proc guard、dedup stub。
  • 写入/patch 前会检查 stale view，并把 resolved path 写回结果，降低 wrong-cwd 误改。

3. Patch 匹配策略

  • tools/fuzzy_match.py 有 exact、line-trimmed、whitespace、indentation、escape、unicode、block-anchor、context-aware 等策略。
  • 非 exact 匹配后会对 new_string 做 reindent，避免模型用错缩进却仍写坏文件。
  • 失败时给 closest sections 和三次失败 escalation，指导模型 re-read 或改用 full write。

4. 并发状态协调

  • tools/file_state.py 是 process-wide registry：per-agent read stamps、global last writer、per-path lock。
  • 用于 sibling subagent 写同一文件后的 stale warning。
  • 补足单 agent 的 readFileState，处理多 agent / 多线程场景。

5. 安全边界

  • agent/file_safety.py 统一 credential/control-plane/project .env 的 read/write deny。
  • 跨 profile 写技能、插件、cron、memories 时 soft block，要求显式确认。
  • read guard 明确说明是 defense-in-depth，不伪装成强安全边界。

6. 测试矩阵

  • test_file_write_safety.py 覆盖 atomic、BOM、mode、temp cleanup、特殊字符。
  • test_line_ending_preservation.py 覆盖 CRLF/LF write 与 patch。
  • test_patch_failure_tracking.py、test_fuzzy_match.py 锁住失败提示和匹配策略。
Axiomate Baseline

Axiomate 当前 File Harness 状态

FileReadTool

  • 支持 text/image/PDF/notebook，多种输出 schema。
  • 基于 readFileState 做同 range 读文件去重，返回 file_unchanged。
  • 重复 unchanged stub 已有 loop guidance：首次静默、第二次提示复用旧 Read、第三次 STOP。
  • 已有 binary extension、UNC、device path、size/token guard。

FileEditTool

  • 允许 partial Read 后的小范围精确 edit；Write 才要求已有文件必须 full Read。
  • mtime 变更后会用内容对比降低 Windows/云同步误报。
  • 通过 readFileSyncWithMetadata 保留 encoding、leading BOM 和 line ending，再用 writeTextContent 落盘。
  • 连续 old_string 找不到或多匹配会升级提示，并记录不含路径的 telemetry。

FileWriteTool

  • 写已有文件前同样要求 read 后再 write，并做 stale 检查。
  • Write 是 full replacement：输出 UTF-8、LF、no BOM，不保留旧文件 BOM/CRLF。
  • live 与 resume 路径只在实际去掉 BOM 或归一 CR/CRLF 时记录 runtime-only toolNormalization；标准 LF/no-BOM 写入没有额外标记。
  • 公共写入 helper 默认 strict；FileWriteTool 不 fallback 到 direct write，config/settings 只有 rename lock opt-in fallback。

Resume Reconstruction

  • queryHelpers.ts 对历史 Write input 只按语义内容 canonicalize，不读当前磁盘，也不重建 disk envelope。
  • 历史 Edit 只在已有 full-known transcript content 时 replay；失败或缺上下文就不 seed state。
  • 不改 JSONL、不改历史 tool call，避免破坏 KV-cache-sensitive transcript。

Registry / Lock

  • fileStateRegistry 记录同进程 context/subagent 的 read/write 顺序。
  • structured writes 有 same-path async mutex，最终 stale check 和 write 在锁内完成。
  • parent/subagent completion reminder 会提示子 agent 改过父 context 已读文件。

Checkpoint / Rewind

  • utils/fileHistory.ts 通过 shadow git store 做 turn-keyed snapshot/rewind。
  • 这能回滚坏修改，但不能替代 patch/write 失败前防护。
  • 文件 harness 的目标应是减少坏写落盘；checkpoint 是第二道恢复线。

测试现状

  • FileHarness 测试目录已覆盖 read/write/edit/notebook/simulated sed。
  • atomic、BOM、CRLF、failure metadata、dedup、registry/lock 都有 targeted tests。
  • queryHelpers.fileStateResume.test.ts 覆盖 Write semantic resume、Edit replay、不从当前磁盘 seed state。
  • Focused FileHarness 测试建议使用 --no-file-parallelism 避免 import mock 并行抖动。

Direct Copy vs Adapt

哪些该抄，哪些不该抄

应该直接吸收的 invariant

  • 写入失败不能产生半文件；temp 必须在目标同目录并清理。
  • 已有文件 mode 需要保留。
  • BOM 不应暴露给模型，但编辑后应一致保留或明确处理。
  • CRLF/LF 不能混写；Edit 至少应保留既有行尾。
  • old_string 连续失败必须给升级提示，不让模型无限试错。
  • read dedup stub 不能被写回真实文件。
  • 测试应覆盖真实 filesystem，而不是只测 pure helper。

不建议直接照搬的部分

  • Hermes 的 ShellFileOperations 跨 terminal backend 抽象。Axiomate 的本地 TS/Bun 文件工具不需要 Python shell 中间层。
  • V4A multi-file patch parser。Axiomate 当前模型表面是 Read/Edit/Write，不应先引入第二套 patch ABI。
  • Hermes profile/cross-profile guard 的具体路径。Axiomate 应映射成 .axiomate/skills、agents、commands、memory、scratchpad、plans 的本地权限体系。
  • Hermes Write 默认保留旧 CRLF 的策略。Axiomate 当前已对齐为：Write 新建 canonicalizes，覆盖已有文件保留 encoding/BOM/line-ending envelope；precise edit 继续 preserves。
Risks

主要冲突点与风险

1. UI 展示可能还偏弱

当前 failure metadata 已经进入 validation/execution/helper 路径，但 UI、/doctor、recovery trace 是否要展示这些结构化原因尚未展开。这里应先看真实调试需要，避免把模型已经能理解的错误文案重复包装。

2. 数据统计要继续低敏

Stage 7B 已经增加 file_edit_failure_escalation telemetry，但只记录 reason、count、level 和 file extension。后续如果统计 stale guard、partial-read reject、atomic failure 或 read-dedup STOP，也应继续避免完整路径和代码内容。

3. 跨进程 registry 不做

当前 fileStateRegistry 和 per-path mutex 是 process-local。Pane/tmux teammates 或另一个 CLI 进程继续依赖 mtime/content stale checks 和 checkpoint/rewind。完整 file-backed registry/lock 协议会引入 stale lock、crash cleanup、路径规范化和性能问题，当前不值得。

4. Registry alias 已修复但边界收窄

同进程 structured registry/path lock 已升级为 canonical key：已有路径走 realpath，未存在尾部路径解析最深已有父目录，Windows 才 casefold。readFileState 仍保持逻辑路径键。同一 async chain 重入同一路径锁会 fail-fast，不会等待自己。硬链接、任意 shell 写和跨进程 registry 仍不纳入。

5. Fuzzy patch 仍不应默认复制

Hermes 的 fuzzy match 很强，但也会让 old_string 不精确时仍能改文件。Axiomate 目前选择 repeated failure hint + telemetry。是否引入 fuzzy patch 应看 file_edit_failure_escalation 数据，而不是默认照搬。

Final Judgment

最终判断

Axiomate file harness 的主体迁移已经完成。正确路线仍然不是把 Hermes 的文件工具整套搬进来，而是把 Hermes 证明有效的工程约束迁入 Axiomate 现有 TypeScript 工具体系；这条路线已经覆盖测试、读后写、stale guard、同进程 registry、path lock、file-tool strict atomic write、BOM/CRLF、failure taxonomy、edit failure escalation、read dedup loop guard，以及 resume/cold-start file-state reconstruction。

已收敛的边界是：encoding_unsupported 保留 taxonomy reason 但不启用硬拒绝，跨进程 registry/locking 不做，JSONL 历史不为格式矫正而重写，config/settings 只对 rename lock 保留受控 direct fallback，file harness safety failure 在工具内部 throw、由 runner 捕获为 is_error tool result；同进程 registry/path lock 已处理 symlink、Windows case alias、same-chain reentry fail-fast 和 killed/failed subagent reminder。剩余工作不应再按“继续抄 Hermes”推进；应先做 HR10 telemetry/privacy 审计，再做 UI 展示、低敏数据统计，以及 telemetry 证明必要后的 edit recovery 打磨。
