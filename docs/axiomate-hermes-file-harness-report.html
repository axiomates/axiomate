Axiomate vs Hermes Agent File Harness 专项报告

本报告只讨论文件读写/编辑可靠性：原子落盘、BOM/CRLF、staleness guard、read dedup、patch failure taxonomy、权限/安全 guard、并发文件状态和测试夹具。结论是：Axiomate 不缺文件工具主线，但缺一层可测试、可解释、可收紧的 file harness；Hermes 的实现应按 invariant 迁移，不应逐行复制 Python 工具栈。

Executive Summary

一句话结论

Axiomate 的 file harness 不是缺工具，而是缺“失败前边界”和“测试化语义”。 现在 FileEditTool 和 FileWriteTool 已经能做本地文件读写，且有读后写 staleness 检查、权限系统、LSP 通知、checkpoint/rewind。真正短板是：公共写入函数虽然尝试 temp+rename，但失败时会 fallback 到非原子写；BOM 语义不够清晰；FileEdit/FileWrite 专项回归测试明显薄；失败结果还不够机器可处理。

Hermes 的优势是把文件操作当成 long-running agent harness 来治理。 它不只实现 write_file/patch，还围绕 read loop、stale view、并发 subagent 写入、敏感路径、跨 profile、BOM/CRLF、fuzzy match、linter/LSP delta、失败提示和 temp cleanup 建了完整测试。

建议优先级： 先做 Axiomate-native 的 file reliability harness：测试矩阵 + strict atomic write + BOM/CRLF 策略 + failure taxonomy。不要先引入 Hermes 的 Python ShellFileOperations、V4A patch parser 或跨后端 shell 抽象；它们解决的是 Hermes 的平台/容器 backend 问题，不是 Axiomate 当前最短路径。

Hermes System

Hermes File Harness 由哪些层组成

1. 文件操作内核

  • tools/file_operations.py 定义 ReadResult、WriteResult、PatchResult、SearchResult。
  • _atomic_write 用同目录 temp file + rename，目标是半写入失败不污染原文件。
  • write_file / patch_replace 处理 mode、lint/LSP delta、BOM、line ending、diff 输出。

2. 工具层 guard

  • tools/file_tools.py 负责 path resolution、sensitive path、cross-profile、read dedup、stale warning、patch failure counter。
  • 读文件有大文件字符上限、重复读取阻断、device/proc guard、dedup stub。
  • 写入/patch 前会检查 stale view，并把 resolved path 写回结果，降低 wrong-cwd 误改。

3. Patch 匹配策略

  • tools/fuzzy_match.py 有 exact、line-trimmed、whitespace、indentation、escape、unicode、block-anchor、context-aware 等策略。
  • 非 exact 匹配后会对 new_string 做 reindent，避免模型用错缩进却仍写坏文件。
  • 失败时给 closest sections 和三次失败 escalation，指导模型 re-read 或改用 full write。

4. 并发状态协调

  • tools/file_state.py 是 process-wide registry：per-agent read stamps、global last writer、per-path lock。
  • 用于 sibling subagent 写同一文件后的 stale warning。
  • 补足单 agent 的 readFileState，处理多 agent / 多线程场景。

5. 安全边界

  • agent/file_safety.py 统一 credential/control-plane/project .env 的 read/write deny。
  • 跨 profile 写技能、插件、cron、memories 时 soft block，要求显式确认。
  • read guard 明确说明是 defense-in-depth，不伪装成强安全边界。

6. 测试矩阵

  • test_file_write_safety.py 覆盖 atomic、BOM、mode、temp cleanup、特殊字符。
  • test_line_ending_preservation.py 覆盖 CRLF/LF write 与 patch。
  • test_patch_failure_tracking.py、test_fuzzy_match.py 锁住失败提示和匹配策略。
Axiomate Baseline

Axiomate 当前已有的 File Harness 基线

FileReadTool

  • 支持 text/image/PDF/notebook，多种输出 schema。
  • 基于 readFileState 做同 range 读文件去重，返回 file_unchanged。
  • 已有 binary extension、UNC、device path、size/token guard。

FileEditTool

  • 要求已读完整文件后才能 edit，防止 partial view 覆盖。
  • mtime 变更后会用内容对比降低 Windows/云同步误报。
  • 通过 readFileSyncWithMetadata 保留 encoding 和 line ending，再用 writeTextContent 落盘。

FileWriteTool

  • 写已有文件前同样要求 read 后再 write，并做 stale 检查。
  • 当前 full write 明确用模型给出的内容，并以 LF 写入，避免旧 CRLF 污染脚本。
  • 调用公共 writeTextContent，底层已有 temp+rename 但语义还不够 strict。

权限与路径安全

  • utils/permissions/filesystem.ts 包含 dangerous files、dangerous dirs、.axiomate 内部路径、scratchpad、memory、plan carve-out。
  • getPathsForPermissionCheck 会检查原路径、symlink target 和最终 resolved path。
  • UNC、Windows ADS/short-name/trailing-dot 等风险已有专门处理。

Checkpoint / Rewind

  • utils/fileHistory.ts 通过 shadow git store 做 turn-keyed snapshot/rewind。
  • 这能回滚坏修改，但不能替代 patch/write 失败前防护。
  • 文件 harness 的目标应是减少坏写落盘；checkpoint 是第二道恢复线。

测试现状

  • checkpoint/fileHistory 测试较厚。
  • FileEdit/FileWrite/BOM/atomic/CRLF 专项测试偏少。
  • 因此第一步不是大改功能，而是把现有语义和目标语义先锁成 tests。
Migration Plan

建议迁移路线

阶段 1：建立 Axiomate-native file harness tests

预计 1-2 天。先不要改太多行为，优先把当前语义和目标语义写成 Vitest。

最高优先级
  • 新增 agent/src/__tests__/unit/tools/FileWriteTool/... 和 FileEditTool 专项目录。
  • 覆盖：temp cleanup、mode preserve、atomic failure leaves original intact、BOM read/edit/write、CRLF edit、Write LF policy、read dedup after write。
  • 把 Hermes 的测试用例转换成 TypeScript/Bun 语境，不复制 Python fixture。

阶段 2：收紧公共写入原语

预计 1-2 天。这里是 file harness 的核心行为改变。

可直接做
  • 把 writeFileSyncAndFlush_DEPRECATED 拆成明确的 atomicWriteTextContent 或等价函数。
  • 同目录创建 temp，写入并 flush，保留已有 mode，rename 替换，失败清理 temp。
  • 删除或默认禁用 non-atomic fallback；fallback 会削弱“失败不落半文件”的核心 invariant。

阶段 3：BOM / line ending 策略定稿

预计 1-2 天。这里有产品语义，不应盲抄 Hermes。

需决策
  • 建议：FileEditTool 保留原 BOM/CRLF；FileWriteTool 对新文件按模型内容写，对已有文件是否保留 CRLF 要明确。
  • 如果维持 Axiomate 现在的 Write LF policy，测试要明确锁住原因：full replacement 是模型显式内容。
  • 无论行尾策略如何，UTF-8 BOM 应有一致行为：模型读取时不见 phantom char，编辑后不要丢或重复 BOM。

阶段 4：失败 taxonomy 与可解释结果

预计 2-4 天。把工具错误从字符串提升成稳定类别。

中高收益
  • 定义错误码：not-read、partial-view、stale-mtime、stale-content、string-not-found、multiple-match、permission-denied、atomic-write-failed、encoding-unsupported。
  • FileEdit/FileWrite 的 validation 和 call path 用同一套分类输出。
  • 后续可接入 /doctor、ValidationErrorsList、tool result rendering、checkpoint delta。

阶段 5：patch failure escalation 与 read dedup loop guard

预计 2-3 天。属于模型自纠错效率优化。

可后置
  • 连续三次对同一文件 old_string 不匹配时，给强提示：停止微调旧字符串，重新 read，扩大上下文，必要时 full write。
  • read dedup stub 连续命中时升级为 BLOCKED，避免弱模型陷入 read→stub→read。
  • 防止模型把 read dedup status 文本写回文件。

阶段 6：并发 subagent file state registry

预计 3-5 天。只有在多 agent 并发编辑变常见后才值得优先。

中期项
  • 借鉴 Hermes FileStateRegistry，记录每个 agent 的 read stamp 和 last writer。
  • 同路径 read-modify-write 用 per-path lock 序列化。
  • 和 Axiomate coordinator/workerAgent 的 agentId、ToolUseContext 合并，而不是新增 Hermes task_id 体系。
Direct Copy vs Adapt

哪些该抄，哪些不该抄

应该直接吸收的 invariant

  • 写入失败不能产生半文件；temp 必须在目标同目录并清理。
  • 已有文件 mode 需要保留。
  • BOM 不应暴露给模型，但编辑后应一致保留或明确处理。
  • CRLF/LF 不能混写；Edit 至少应保留既有行尾。
  • old_string 连续失败必须给升级提示，不让模型无限试错。
  • read dedup stub 不能被写回真实文件。
  • 测试应覆盖真实 filesystem，而不是只测 pure helper。

不建议直接照搬的部分

  • Hermes 的 ShellFileOperations 跨 terminal backend 抽象。Axiomate 的本地 TS/Bun 文件工具不需要 Python shell 中间层。
  • V4A multi-file patch parser。Axiomate 当前模型表面是 Read/Edit/Write，不应先引入第二套 patch ABI。
  • Hermes profile/cross-profile guard 的具体路径。Axiomate 应映射成 .axiomate/skills、agents、commands、memory、scratchpad、plans 的本地权限体系。
  • Hermes Write 默认保留旧 CRLF 的策略。Axiomate 已有 full write LF policy，是否改变需单独决策。
Risks

主要冲突点与风险

1. Write 行尾策略冲突

Axiomate 现在 FileWriteTool 的注释明确：full content replacement 应尊重模型发来的内容，不保留旧文件行尾，并写成 LF。这和 Hermes 的 “覆盖 CRLF 文件时保留 CRLF” 不一致。这个不是简单 bug，必须先决定产品语义。

2. 非原子 fallback 会掩盖真实失败

Axiomate 底层写入已尝试 atomic rename，但失败后 fallback 到 non-atomic write。对 file harness 来说，这会破坏最关键保证：失败要么不落盘，要么明确告知状态不确定。建议收紧。

3. Checkpoint 不能替代写前防护

Axiomate 的 checkpoint/rewind 是优势，但它解决的是坏结果落盘后的恢复；Hermes file harness 解决的是落盘前减少坏结果。两者应该叠加，不应互相替代。

4. 直接复制 Hermes fuzzy patch 可能扩大行为面

Hermes 的 fuzzy match 很强，但也会让 old_string 不精确时仍能改文件。Axiomate 如果引入，应先从 reindent 和 failure hint 开始，而不是一次性引入所有 fuzzy 策略。

Final Judgment

最终判断

Axiomate 下一步聚焦 file harness 是合理的，而且工作量可控。正确做法不是把 Hermes 的文件工具整套搬进来，而是把 Hermes 证明有效的工程约束迁入 Axiomate 现有 TypeScript 工具体系：先建测试，后收紧原子写，再定 BOM/CRLF 策略，最后扩展 failure taxonomy 和并发状态。

最值得马上做的是 750-1,250 LOC 的最小高价值包：“严格原子写 + BOM/CRLF 测试 + FileEdit/FileWrite 基础 failure boundary”。这会直接降低 coding agent 最常见的坏文件风险，并且不会和 Axiomate 的 API harness、checkpoint/rewind、本地权限系统冲突。
