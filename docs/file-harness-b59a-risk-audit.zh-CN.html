Findings

已确认风险与修复状态

R1. Write 的 full-read guard 没有统一使用 fileStateHasFullContent()

已修复

设计决策是：覆盖已有文件的 Write 必须在 full Read 之后才允许。审计时发现代码存在不一致： validateInput 和 call 的入口曾只拒绝 !readTimestamp || readTimestamp.isPartialView， 但 FileState 的 full-content 定义还包括 limit === undefined 和 offset === undefined || offset === 1。 本轮已将 Write 覆盖已有文件的 validation 和 execution guard 统一改为 fileStateHasFullContent()。

证据

  • agent/src/utils/fileStateCache.ts:29-35 定义 full content：不能是 partial view，不能有限制 limit，offset 只能为空或 1。
  • agent/src/tools/FileWriteTool/FileWriteTool.ts:221-233 validation 入口现在用 !fileStateHasFullContent(readTimestamp) 拒绝 partial state。
  • agent/src/tools/FileWriteTool/FileWriteTool.ts:335-342 execution 入口现在用 !fileStateHasFullContent(lastRead) 拒绝 partial state。
  • agent/src/__tests__/unit/tools/FileHarness/fileWrite.behavior.test.ts 新增 validate/call 两个测试，覆盖 “有 limit 但没有 isPartialView” 的状态。

状态：已修复。后续 R5 处理 side-channel read helper 时，再统一 memory/relevant memory 的 partial metadata 与 registrySequence。

R2. resume 重建会把失败的 Read/Write 结果当成成功状态

已修复

extractReadFilesFromMessages 对 Edit 已检查 content.is_error !== true， 审计时发现 Read 和 Write 没有同样的过滤。结果是失败的 Read 可能把错误文本写进 readFileState； 失败的 Write validation/execution 也可能按 tool input 重建为“已经写过/读过”的 full state。 另外，第一遍收集 Write input 使用 input?.content 真值判断，成功写入空文件不会被重建。 本轮已要求 Read/Write 结果非 error 才能重建，并允许 empty-string Write 被记录。

证据

  • agent/src/utils/queryHelpers.ts:404-407 Write input 现在以 typeof input.content === 'string' 记录，空字符串不会被丢弃。
  • agent/src/utils/queryHelpers.ts:451-455 Read 重建现在要求 content.is_error !== true。
  • agent/src/utils/queryHelpers.ts:489-492 Write 重建现在要求 content.is_error !== true。
  • agent/src/__tests__/unit/utils/queryHelpers.fileStateResume.test.ts 新增 failed Read、failed Write、empty Write 三个测试。

状态：已修复。

R3. hook/permission updatedInput 发生在 validate 之后，没有二次验证

已修复

工具执行流程先跑 schema/validation，然后 PreToolUse hook 和 permission 可以替换 processedInput。 审计时发现替换后的 input 会直接进入 tool.call。file harness 的 stale/not-read/sibling 关键防线多数在 call 里重复了， 但并非所有 validation-only 规则都有执行期等价物。对 FileEdit 来说，multiple_match、settings 文件编辑校验、 notebook 必须用 NotebookEdit、空 old_string 语义等都主要在 validate 阶段。 本轮已在两个 updatedInput 入口后重跑 schema + validateInput；失败时返回普通 error tool_result，不进入 tool.call。

证据

  • agent/src/services/tools/toolExecution.ts:834-860 validation 在 hook 之前执行。
  • agent/src/services/tools/toolExecution.ts:941-945 PreToolUse hook 可替换 input。
  • agent/src/services/tools/toolExecution.ts:1145-1147 permission decision 可再次替换 input。
  • agent/src/services/tools/toolExecution.ts 新增 validateUpdatedToolInput()，对 hook/permission updatedInput 重跑 schema + validateInput。
  • agent/src/__tests__/unit/services/tools/toolExecutionFileHarnessError.test.ts 新增 PreToolUse updatedInput 和 permission updatedInput 两个测试，确认 bad updatedInput 不会进入 call。

状态：已修复。后续如发现某些 hook 依赖 intentionally-invalid intermediate input，应改 hook 输出，而不是绕过执行层二次验证。

R4. sibling writer registry 的容量淘汰不是真正 LRU

已修复

lastWriterByPath capped at 4096，淘汰时删除 Map insertion order 的最老 key。 但同一路径重复写入时直接 set 已存在 key 不会刷新插入顺序，所以一个刚被写过的老 key 仍可能被后续第 4097 个不同路径淘汰。 淘汰后 sibling 检测会退化为 mtime/content 检查；如果 mtime 被恢复或文件系统时间粒度不足，就可能漏报。 本轮已在写入已有 registry key 前先 delete 再 set，让 Map insertion order 符合 true LRU 语义。

证据

  • agent/src/utils/fileStateRegistry.ts:47-52 capMap 删除 map.keys().next()。
  • agent/src/utils/fileStateRegistry.ts noteFileWrite 现在先删除旧 key 再写入新 stamp，刷新 writer registry 顺序。
  • agent/src/utils/fileStateRegistry.ts:139-151 stale sibling 判断依赖该 Map 中的 last writer。
  • agent/src/__tests__/unit/utils/fileStateRegistry.test.ts 新增 “recently rewritten paths are kept when the writer registry reaches its cap” 测试，覆盖旧实现会淘汰刚重写老 key 的场景。

状态：已修复。仍承认 4096 cap 本身是 bounded registry 设计；超出容量的非常老路径会退回 mtime/content/checkpoint 边界。

R5. side-channel read state 没有统一记录 registrySequence

已修复

FileReadTool 写入 read state 后会调用 recordFileRead，从而写入 registry sequence。 但 memory/relevant memory/REPL startup 等 side-channel 直接 readFileState.set，没有统一调用 recordFileRead。 当前 registry 对 registrySequence === undefined 的处理是保守地视为 stale；这能避免漏写，但会增加误提醒/误拒绝，也让“模型已经通过上下文看到内容”与 registry 判断脱节。 本轮新增 setObservedFileState()，把 side-channel “已经展示给模型/注入上下文的文件内容” 统一写入 read cache 并登记 registrySequence。

证据

  • agent/src/tools/FileReadTool/FileReadTool.ts:1037-1045 text Read 写 cache 后调用 recordFileRead。
  • agent/src/utils/fileStateRegistry.ts 新增 setObservedFileState()：先写 readFileState，再调用 recordFileRead。
  • agent/src/utils/attachments.ts nested memory 和 relevant memory 现在通过 setObservedFileState() 写入；truncated relevant memory 同时标记 isPartialView。
  • agent/src/screens/REPL.tsx startup memory 现在也通过 setObservedFileState() 写入。
  • agent/src/utils/fileStateRegistry.ts:148-150 undefined sequence 会被视为 stale。
  • agent/src/__tests__/unit/utils/attachmentsFileStateRegistry.test.ts 覆盖 relevant memory 的 registrySequence 和 truncated partial-view 标记。

状态：已修复。语义保持原决策：side-channel full 内容仍可作为已读依据；裁剪/变形内容继续通过 isPartialView 阻止 Write 覆盖。

R6. path lock / sibling path key 仍有跨平台别名边界

已修复

当前 path key 已处理 symlink parent，单进程锁也能按该 key 串行。 审计时发现大小写规则只按 process.platform === 'win32' 做全局折叠； 这把 Windows、macOS、Linux 混成了一个二分规则。新的决策是三平台分开： Windows 永远大小写不敏感，Linux 永远大小写敏感，macOS 按实际卷/目录能力探测并按 device 缓存。 探测失败时保守不折叠，避免误合并大小写敏感卷上的不同文件。

证据

  • agent/src/utils/fileStateRegistry.ts 现在拆分 windowsRegistryPathKey()、macRegistryPathKey()、linuxRegistryPathKey()。
  • Windows 规则：registry key 始终 lower-case，匹配 NTFS/Windows 常见大小写不敏感行为。
  • Linux 规则：registry key 保留大小写，不做折叠。
  • macOS 规则：找最近存在目录，写入临时 probe，检测大小写别名是否存在，并按 stat.dev 缓存结果。
  • agent/src/__tests__/unit/utils/fileStateRegistry.test.ts 新增三平台规则测试，以及 macOS 动态探测测试。

状态：已修复。UNC/双斜杠路径不会触发 macOS probe；无法探测时保守保持大小写敏感，避免把不同文件误合并。

R7. telemetry 内容记录是 opt-in，但边界还没有产品化决策

低 / 决策

当前 telemetry 对工具输入/输出内容的记录不是默认开启：beta tracing 要求 ENABLE_BETA_TRACING_DETAILED=1、BETA_TRACING_ENDPOINT 和非交互会话； addToolContentEvent 还要求 OTEL_LOG_TOOL_CONTENT。 但一旦开启，Read content、Write content、Edit diff、Bash command/output 都可能进入 tracing，最多截断到 60KB。 这不是当前默认行为的 bug，但在 file harness UI/统计之前需要明确数据边界。

证据

  • agent/src/utils/telemetry/betaSessionTracing.ts:63-73 beta tracing 需要 env + endpoint + headless。
  • agent/src/utils/telemetry/sessionTracing.ts:711-724 tool content event 还要求 OTEL_LOG_TOOL_CONTENT。
  • agent/src/services/tools/toolExecution.ts:1250-1289 组装 file_path/content/diff/bash_command/output 后发送 content event。
  • agent/src/utils/telemetry/betaSessionTracing.ts:411-452 beta tool input/result 会写入 span attributes，并做 60KB 截断。

建议：把 file harness 统计默认限定为 reason/code/path extension/size/boolean，不默认记录正文、diff、命令完整输出；需要正文时必须显式 opt-in 并在文档标注。

Path Lock / Sibling Status

path lock 与 sibling 状态专项结论

当前已经做对的部分

  • withFileStatePathLock() 使用 per-path promise tail，能在单进程内串行同一路径的 callback。
  • 锁持有期间用 AsyncLocalStorage 标记 held locks，重入同一路径会抛 path_lock_reentry，避免死锁。
  • finally 中 release gate、更新 depth、删除 tail，callback throw 不会永久占住队列。
  • FileWrite/FileEdit/NotebookEdit/内部 simulated sed 都把最终 stale check 和写入放进同一 critical section。
  • symlink parent alias 已有 registry key 和行为测试覆盖。

仍要承认的边界

  • 锁是进程内的，不是跨进程文件锁；另一个 Axiomate 进程、编辑器、shell 命令仍可能并发写。
  • 任意 Bash/PowerShell 写文件不进入 fileStateRegistry；只有内部 simulated sed 会更新 state。
  • mtime 使用 Math.floor(mtimeMs)，有意降低误报，但同一秒内外部写入仍主要靠 sibling registry/checkpoint 兜底。
  • hard link 未 canonicalize 到 inode；跨进程/外部写入仍不进入进程内 registry。
Non Findings

本轮不视为 bug 的设计边界

边界 原因 仍需注意
任意 shell 写入不进入 registry PowerShell/Bash 可写文件的方式太多，拦截所有写入需要 checkpoint/sandbox/FS monitor，当前决策是不做。 依赖 mtime/content/checkpoint；内部 simulated sed 是特例，因为它由 permission preview 注入并由工具自己执行。
cross-process registry 不做 已有决策：不引入跨进程 registry；Git checkpoint 系统是跨 session 的重型方案。 同一 repo 多进程并行写时，registry 不能互相看见。
path lock reentry 直接失败 这是防死锁策略，不应排队等待自己释放自己。 需要确保上层 catch 为 tool error，而不是进程退出。现有 file harness error path 已有测试。
unsupported encoding 当前不拒绝 Edit 之前决策是保持现行为；Edit 尽量按当前 read/write metadata 处理。 仍应避免在统计里误称 encoding 已完全安全。
