Axiomate user guide

模型配置与路由策略

更新时间：2026-06-02。本文面向第一次接触 Axiomate 的用户，解释 ~/.axiomate.json 里模型相关字段、手动配置方法、/model 命令和 Settings 里的模型 UI。

一句话：models 定义“有哪些具体模型资源”，model.defaultRoute / model.routes 定义主对话走哪条模型链，auxiliary 定义标题生成、目标评审、摘要、hook 等辅助任务走哪条模型链。

模型链就是一次主对话或辅助任务允许使用的一组候选模型：先用 primary，只有统一恢复系统判断“这类错误适合切模型”时，才按 fallbackChain 的顺序尝试后面的模型。

Axiomate 的 API 可靠性由统一的观察 → 决策 → 执行恢复系统负责。路由配置只提供候选模型和策略门槛，不是另一套独立 retry 逻辑。

文件位置

用户级配置文件在 ~/.axiomate.json。Windows 上通常是 C:\Users\<你的用户名>\.axiomate.json。

首次运行或执行 /model add 时，Axiomate 会创建或更新这个文件。你也可以手动编辑它，但保存前要确保 JSON 合法。

当前配置合同：模型配置只使用 models、model.routes、model.defaultRoute 和 auxiliary。模型切换和路由管理使用 /model 命令，不使用 /config 设置项。

先理解三个概念

配置文件里最容易混淆的是模型资源、模型链和 route。用户只需要记住：模型资源负责“怎么调用”，模型链负责“失败时还能尝试谁”，route 负责“把一套主对话用法保存成名字”。

概念 是什么 怎么用 什么时候用 配置位置
模型资源 这个模型怎么调用：协议、endpoint、API key、实际 provider model name、上下文窗口、图片能力。 用 /model add 添加，或手动写进 models。 接入一个新 provider 或新模型时。 models.<model-id>
模型链 一次请求可以依次尝试哪些模型。 在 route 或 auxiliary task 里配置 primary 和 fallbackChain。 希望 API 限流、超时、服务端错误、输出畸形时能自动换候选模型。 primary + fallbackChain
路由 给一条模型链起名字，并附上恢复策略。主对话通过 route 选择不同“用法”。 用 /model route <route-id> 切换，或手动改 model.defaultRoute。 想在“默认主力 / 质量优先 / 速度优先”等主对话模式之间切换。 model.routes.<route-id>

model.defaultRoute 只是指向其中一条 route。比如 "defaultRoute": "deepseek-main" 的意思是：主对话默认使用 deepseek-main 这条模型链，而不是直接写死某一个模型。

模型链不是负载均衡，也不是每次随机选模型。Axiomate 每次请求先走 primary。只有当观察层确认本轮错误属于可恢复错误，决策层又选择了 switch_model，执行层才会切到 fallbackChain 中的下一个模型。

主 route：是什么、怎么用、什么时候用

是什么

主 route 是“主对话用哪套模型策略”的名字。每条 route 都包含一个第一选择模型 primary、一组后备模型 fallbackChain，以及允许哪些错误触发恢复的策略字段。

一条 route 就可以运行。多条 route 是为了给同一批模型预设几种常用模式，例如默认主力、质量优先、速度优先。它们可以引用同一批 models，只是模型顺序不同。

route 是什么 典型模型链 什么时候用
deepseek-main 默认主力路线。 deepseek-v4-pro → gpt-5.4 → Pro/zai-org/GLM-5.1 → Pro/moonshotai/Kimi-K2.6 → Qwen/Qwen3.5-397B-A17B → Pro/MiniMaxAI/MiniMax-M2.5 → Qwen/Qwen3.5-122B-A10B → Qwen/Qwen3-8B 日常主对话。先用最常用的主力模型，只有可恢复错误需要切模型时再按质量顺序兜底。
quality-main 质量优先路线。 gpt-5.4 → deepseek-v4-pro → Pro/zai-org/GLM-5.1 → Pro/moonshotai/Kimi-K2.6 → Qwen/Qwen3.5-397B-A17B → Pro/MiniMaxAI/MiniMax-M2.5 → Qwen/Qwen3.5-122B-A10B → Qwen/Qwen3-8B 复杂重构、长推理、重要输出。明确希望先用最强或最贵模型时切到这条 route。
fast-main 速度/成本优先路线。 Qwen/Qwen3.5-122B-A10B → Pro/MiniMaxAI/MiniMax-M2.5 → Qwen/Qwen3.5-397B-A17B → Pro/moonshotai/Kimi-K2.6 → deepseek-v4-pro → Qwen/Qwen3-8B 简单问答、短文本处理、临时快速任务。先用较快的大模型，失败时再升到更强候选。

model.defaultRoute 决定默认使用哪条主 route。例如 "defaultRoute": "deepseek-main" 表示平时先走默认主力链。切换 route 的本质就是把 model.defaultRoute 改成另一条 route 名。

{
  "model": {
    "defaultRoute": "deepseek-main",
    "routes": {
      "deepseek-main": {
        "primary": "deepseek-v4-pro",
        "fallbackChain": ["gpt-5.4", "Pro/zai-org/GLM-5.1", "Pro/moonshotai/Kimi-K2.6", "Qwen/Qwen3.5-397B-A17B", "Pro/MiniMaxAI/MiniMax-M2.5", "Qwen/Qwen3.5-122B-A10B", "Qwen/Qwen3-8B"],
        "recoveryProfile": "main-agent",
        "allowActions": ["retry_same_model", "adapt_request", "switch_model"],
        "switchModelOn": ["rate_limit", "overloaded", "timeout", "connection", "server_error", "malformed_response", "responses_null_output", "model_not_found", "content_policy_blocked"]
      },
      "quality-main": {
        "primary": "gpt-5.4",
        "fallbackChain": ["deepseek-v4-pro", "Pro/zai-org/GLM-5.1", "Pro/moonshotai/Kimi-K2.6", "Qwen/Qwen3.5-397B-A17B", "Pro/MiniMaxAI/MiniMax-M2.5", "Qwen/Qwen3.5-122B-A10B", "Qwen/Qwen3-8B"],
        "recoveryProfile": "main-agent",
        "allowActions": ["retry_same_model", "adapt_request", "switch_model"],
        "switchModelOn": ["rate_limit", "overloaded", "timeout", "connection", "server_error", "malformed_response", "responses_null_output", "model_not_found", "content_policy_blocked"]
      },
      "fast-main": {
        "primary": "Qwen/Qwen3.5-122B-A10B",
        "fallbackChain": ["Pro/MiniMaxAI/MiniMax-M2.5", "Qwen/Qwen3.5-397B-A17B", "Pro/moonshotai/Kimi-K2.6", "deepseek-v4-pro", "Qwen/Qwen3-8B"],
        "recoveryProfile": "main-agent",
        "allowActions": ["retry_same_model", "adapt_request", "switch_model"],
        "switchModelOn": ["rate_limit", "overloaded", "timeout", "connection", "server_error", "malformed_response", "responses_null_output", "model_not_found", "content_policy_blocked"]
      }
    }
  }
}

有三条 route 时，Axiomate 不会三条同时跑。默认只跑 model.defaultRoute 指向的那条。其他 route 是可切换的预设，用于不同任务取向。

怎么用

你想做什么 推荐命令 手动改 JSON 时改哪里
查看当前主对话实际用哪条 route /model show 看 model.defaultRoute 的值。
列出所有可选 route /model route list 看 model.routes 下面有哪些 key。
查看某条 route 的模型链 /model route show quality-main 看 model.routes.quality-main.primary 和 fallbackChain。
把默认主对话切到质量优先 route /model route quality-main 或 /model default quality-main 把 model.defaultRoute 改成 "quality-main"。
把默认主对话切回默认主力 route /model route deepseek-main 把 model.defaultRoute 改成 "deepseek-main"。
只换当前默认 route 的第一选择模型 /model use <model-id> 或打开 /model 选择器 改 model.routes.<defaultRoute>.primary。这不是切换 route。

命令方式会直接写入 ~/.axiomate.json，下次启动仍然生效。手动编辑时，model.defaultRoute 必须引用 model.routes 中已经存在的 route 名；每个 route 里的 primary 和 fallbackChain 必须引用 models 中已经存在的模型 id。

Settings 里的 Model 不是 route 切换器。Settings 或 /model 选择器主要用于换当前默认 route 的 primary。要在 deepseek-main、quality-main、fast-main 之间切换，请使用 /model route <route-id>，或手动修改 model.defaultRoute。

这类选择不是临时修改。Settings → Config 的 Model、直接打开 /model 选择器、以及 /model use <model-id> 都会持久化写入当前 model.defaultRoute 指向的那条 route，把这条 route 的 primary 改成所选模型。下次启动仍会使用这个新的 primary。

什么时候应该用多条 route

情况 建议 原因
只接了一个模型，或很少切换任务类型 保留一条 route 就够。 配置更简单，主对话一直走同一条模型链。
日常任务和复杂任务差异很大 保留 default-main / quality-main 两条 route。 平时用默认路线，需要高质量时用 /model route quality-main 切过去。
经常做简单、低成本、低延迟任务 增加一条 fast-main route。 让小模型做第一选择，只有可恢复错误时再尝试更强模型。
只是想给当前默认路线换第一选择模型 用 /model 选择器或 /model use <model-id>。 这会改当前 route 的 primary，不需要新建或切换 route。
某个 provider 经常限流、超时、服务端错误 在当前 route 的 fallbackChain 里加入其他 provider 的模型。 fallback 是可靠性机制；只有恢复系统判断适合切模型时才会使用。

模型资源：是什么、怎么用、什么时候用

是什么

模型资源是 models 里的一个条目，描述“如何调用某个 provider 的某个模型”。它不决定主对话默认用谁，也不决定 fallback 顺序；这些由 route 和 auxiliary task 决定。

例如 models["main-qwen"] 是 Axiomate 内部引用的模型 id，route 里写的也是这个 id；而 models["main-qwen"].model 是真正发给 provider API 的模型名。

怎么用

你想做什么 推荐方式 配置会发生什么
接入一个新模型 /model add 新增一个 models.<model-id>，并可选择是否加入主 route。
修改 API key、endpoint、协议、上下文窗口 /model edit <model-id> 或手动编辑 models.<model-id> 只修改模型资源本身，不会自动改变默认 route。
让主对话开始用这个模型 /model use <model-id> 把当前默认 route 的 primary 改成这个模型。
把这个模型作为后备模型 /model fallback add <model-id> 把它加入当前默认 route 的 fallbackChain。
只给标题、摘要、目标评审等辅助任务使用 /model aux set <task> <model-id> 把对应辅助任务的 primary 改成这个模型。

什么时候需要改模型资源

情况 应该改哪里
API key 失效、换 provider endpoint、模型名写错 改 models.<model-id>.apiKey、baseUrl 或 model。
模型不支持图片，却被用于图片输入 把 supportsImages 设为 false，并把需要图片输入的主 route 指向支持图片的模型。
provider 需要特殊 reasoning / thinking 参数 改 thinking，必要时设置 vendor 或 extraParams。
上下文或输出 token 预算不对 改 contextWindow 或 maxOutputTokens。

字段参考

字段 必填 含义
model 是 发送给 provider API 的模型名。它可以和 models 的 key 一样，也可以不同。
name 否 模型选择器里显示的名字。省略时显示 models 的 key。
description 否 模型选择器里的说明。省略时显示协议和 endpoint。
protocol 是 openai-chat、openai-responses 或 anthropic。决定请求/流式响应的 wire format。
vendor 否 供应商模板选择，用来把中立的 thinking 配置翻译成 provider 需要的字段，并处理网关级兼容差异。auto 或省略表示按 protocol + baseUrl 推断网关；已知 host 包括 DeepSeek、阿里云、SiliconFlow、GLM / BigModel。none 表示禁用 vendor 推断，只使用裸协议层。其他值表示固定使用某个自定义或内置 vendor template。
modelTemplate 否 模型层模板选择。auto 或省略表示按模型名 + 已解析 vendor + protocol 智能匹配；匹配到的内置模板可以自动应用 DeepSeek V4+ reasoning 回放、GLM-5.2 reasoning effort 等模型特定行为。none 表示禁用模型层，用于自动匹配误中的情况。其他值表示固定使用某个自定义或内置 model template，显式固定会做兼容性校验。
baseUrl 是 API endpoint，例如 https://api.openai.com/v1 或 OpenRouter / SiliconFlow / vLLM / ollama 的兼容地址。
apiKey 是 API key。即使本地 ollama 不校验 key，也需要填一个占位字符串。
contextWindow 否 上下文窗口 token 数。用于截断、压缩和预算估算。
maxOutputTokens 否 这个模型资源的默认单次响应最大输出 token。辅助任务还可以在 auxiliary.<task>.maxOutputTokens 上进一步收窄请求预算。
supportsImages 否 模型是否支持图片输入。省略时默认 false，初始化 / /model add wizard 也默认选 No。支持视觉输入的模型应显式设为 true；纯文本模型可省略或设为 false。
thinking 否 中立 reasoning 配置，例如 { "enabled": true, "effort": "high" } 或 { "enabled": true, "budget": 8192 }。
extraParams 否 每次请求都原样合并到 body 的 provider 私有参数。
usageMapping 否 OpenAI Chat 兼容响应中 cache token 字段路径映射。Responses 和 Anthropic 通常不需要。
userAgent 否 覆盖 HTTP User-Agent。某些 Responses 兼容网关会按 UA 放行。
repairToolCalls 否 兼容开关：对容易输出畸形 tool-call JSON 的模型启用修复。
stallTimeoutMs 否 覆盖流式响应卡顿告警阈值。设为 0 可关闭该模型的卡顿告警。

Vendor / Model Template 的 reasoning 回放字段

Axiomate 的 template 分三层合并：protocol → vendor → model。后面的层覆盖前面的层，所以当 vendor template 和 model template 都声明同一个字段时，model template 最终生效。

字段 可写位置 推荐位置 含义
autoRoundTripReasoningContent templates、modelTemplates modelTemplates 是否在工具调用后的下一轮请求里回传上一轮 assistant 的 reasoning / thinking。DeepSeek V4+ 这类模型通常需要它。
reasoningRoundTripFormat templates、modelTemplates templates 或 modelTemplates reasoning 回传的历史消息形状。目前仅支持 reasoning_content，用于官方 / 默认 DeepSeek V4+ 的多轮工具调用回放。

vendor 和 modelTemplate 都使用同一组三态语义：auto / 省略表示让 Axiomate 推断或智能匹配；none 表示显式跳过这一层；其他值表示固定到某个具名模板。自动匹配 model template 时会同时按模型名、已解析 vendor 和 protocol 做门控，所以 openai-chat-deepseek-v4p、GLM-5.2 overlay 等内置模板可以无需手动固定就生效。只有确认自动匹配误中时才用 none；需要稳定固定某个模板时再写具名模板。显式固定会做兼容性校验，而 auto 和 none 是保留哨兵值，不是模板名。当前 adapter 只支持 reasoning_content 这一种 reasoning 回放形状；relay 特有形状需要先确认行为并扩展 adapter，不能仅靠自定义 modelTemplates 表达。

  "models": {
    "deepseek-v4-pro": {
      "model": "deepseek-v4-pro",
      "name": "deepseek-v4-pro",
      "protocol": "openai-chat",
      "baseUrl": "https://api.deepseek.com",
      "apiKey": "sk-...",
      "modelTemplate": "openai-chat-deepseek-v4p",
      "contextWindow": 1000000,
      "maxOutputTokens": 384000,
      "supportsImages": false,
      "thinking": { "enabled": true, "effort": "high" }
    }
  }
}

三种协议怎么选

协议 适用场景 典型 baseUrl
openai-chat OpenAI Chat Completions 兼容接口。OpenRouter、SiliconFlow、DeepSeek、vLLM、ollama 等通常用这个。 https://openrouter.ai/api/v1
openai-responses OpenAI Responses API 或兼容网关。适合需要保留 reasoning item 和 Responses tool-call 语义的模型。 https://api.openai.com/v1
anthropic Anthropic Messages wire format。用于真正按 Anthropic 协议暴露的 provider。 https://api.anthropic.com

辅助任务：是什么、怎么用、什么时候用

是什么

auxiliary 是给“不是主对话、但需要 LLM 的小任务”准备的模型策略。标题生成、输入建议、搜索摘要、目标评审、hook、token counting 都属于这个层。它们不应该默认消耗最贵的主模型，也不应该在失败时把 API 错误文本当成正常结果。

怎么用

你想做什么 推荐命令 手动改 JSON 时改哪里
查看所有辅助任务策略 /model aux list 看 auxiliary 下面有哪些 task。
查看某个辅助任务用什么模型链 /model aux show goalJudge 看 auxiliary.goalJudge.primary 和 fallbackChain。
让标题生成用便宜快速模型 /model aux set sessionTitle <model-id> 改 auxiliary.sessionTitle.primary。
让输入建议用便宜快速模型 /model aux set promptSuggestion <model-id> 改 auxiliary.promptSuggestion.primary。
给目标评审增加后备模型 /model aux fallback add goalJudge <model-id> 把模型 id 加进 auxiliary.goalJudge.fallbackChain。
设置辅助任务失败时怎么处理 /model aux policy goalJudge failure fail_open 改 auxiliary.goalJudge.failure。
设置输入建议最多输出多少 token /model aux policy promptSuggestion maxOutputTokens 96 改 auxiliary.promptSuggestion.maxOutputTokens。

什么时候需要改辅助任务

情况 建议 原因
标题生成、摘要、日期解析太慢或太贵 把对应 task 的 primary 换成便宜快速模型。 这些任务不需要主对话同等级模型。
输入框里的下一步建议太慢、太贵，或在 /doctor 里反复出现空输出 把 auxiliary.promptSuggestion.primary 换成便宜快速模型，失败策略保持 return_null。 这是后台提示，不应该拖慢主回复，也不应该因为空结果反复重试。
/goal 判断太松、太严或格式不稳定 调整 auxiliary.goalJudge，选择更稳定遵循 JSON 的模型。 目标评审需要稳定结构化输出。
辅助任务失败时不应该打断主流程 设置 failure 为 return_null、return_original 或 fail_open。 标题、摘要、搜索辅助通常可以失败降级。
hook agent 或关键验证失败时必须暴露错误 设置 failure 为 propagate_error 或 fail_closed。 这类任务失败可能影响安全性或正确性，不能静默吞掉。

字段参考

字段 只在 auxiliary 中额外出现吗 含义
primary / fallbackChain / allowActions / switchModelOn / recoveryProfile 否 和主 route 同语义。
failure 是 所有尝试失败后的最终处置。可选：fail_open、fail_closed、return_null、return_original、return_empty、propagate_error。
timeoutMs 是 该任务单次尝试超时。省略时使用内置 timeout policy。
maxOutputTokens 是 该任务自己的输出上限。它不改变 models[id].maxOutputTokens 表示的模型能力，只限制这个辅助任务一次请求最多要多少输出。
extraBody 是 预留的任务级 provider 参数入口。日常配置优先用 models[id].extraParams；只有明确知道某个辅助执行路径会读取它时再使用。

常见辅助任务

任务 用途 常见模型选择
sessionTitle 生成会话标题。 便宜、快速、稳定的文本模型。
promptSuggestion 生成输入框里淡色显示的下一步建议。 便宜快速的小模型，例如 Qwen3 8B；通常无 fallback，失败 return_null，并用 maxOutputTokens 限制短输出。
tokenCounting 辅助 token 估算/计数路径。 便宜模型；失败通常 return_null。
sessionSearchSummary /resume 或 SessionSearch 命中后的摘要。 便宜摘要模型；失败可 return_original。
memdirRelevance 从 memory frontmatter 中选相关文件。 严格输出 JSON 的中等质量模型。
goalJudge /goal 每轮判断目标是否完成。 严格遵循 JSON 输出的模型，成本最好低于主模型。
hookPrompt / hookAgent prompt hook 或完整 agent hook。 hookAgent 通常需要更高质量和 propagate_error。
skillImprovement 技能改进建议。 便宜但能遵循格式的模型。
mcpDateTimeParse MCP 表单里的自然语言日期时间解析。 快速文本模型。
shellPrefix 提取 shell 命令前缀，用于权限/展示。 快速文本模型。
permissionExplainer 生成权限请求解释。 快速文本模型。

恢复语义：观察、决策、执行

观察

每次 API 尝试都会记录协议、provider、模型、route、auxiliary task、attempt index、初次错误或 retry 错误，并把原始 400、52x、timeout、空输出等归类成语义原因。

决策

决策层读取恢复规则表、上一轮错误、当前错误、route policy 和剩余候选模型，决定 retry_same_model、adapt_request、switch_model 或停止。

执行

执行层按决策修改请求或切到下一个候选。失败结果会回写到同一个 recovery session，供下一轮决策继续使用。

例如第一轮遇到 400，第二轮调整请求后遇到 52x，这不是一个“覆盖掉旧错误”的状态，而是同一轮恢复会话里的两个 attempt 观察记录。决策层能看到“之前是请求形状问题，现在是服务端/网关问题”，再决定是否换模型或停止。

API 自动恢复全貌

是什么：Axiomate 把 API 失败先归类成语义原因，再由统一恢复规则表选择动作。当前规则表显式覆盖 13 类常见可变形错误，恢复动作集合包含 21 类动作。这个数字会随版本增加，用户需要理解的是分类和预算，而不是记住每条内部规则。

怎么用：普通用户只需要配置模型链和策略门槛。主对话默认最多 10 次恢复重试，也就是最多 11 次 attempt；可以用环境变量 AXIOMATE_CODE_MAX_RETRIES 调整。辅助任务按语义分配预算：背景直连辅助请求不做本地恢复重试，验证和快速辅助任务通常 1 次恢复重试，前台交互型或质量优先辅助任务通常 2 次恢复重试。provider 返回 Retry-After 时会尊重，但单次等待会封顶，避免卡住太久。

什么时候会发生：请求失败、流中断、Responses 空输出、模型不支持某些字段、图片过大、上下文过长、服务端 52x、限流、超时、连接断开、模型不可用等情况都会进入观察层。决策层可能选择同模型等待重试、修改请求后重试、请求压缩、非流式降级、刷新连接、切换模型或停止。

失败现象 常见观察原因 可能动作
provider 报不支持 temperature、stop、tool_choice 等字段 unsupported_parameter 省略对应字段后重发。
OpenAI Responses 拒绝历史 reasoning 加密内容 invalid_encrypted_content 移除不可验证的 reasoning replay 后重发。
图片尺寸或大小超过 provider 限制 image_too_large 按图片恢复 profile 重写图片 payload 后重发。
上下文或 payload 太大 context_overflow / payload_too_large 请求压缩，或先降低输出预算、关闭 thinking。
限流、过载、超时、连接断开、服务端 52x rate_limit / overloaded / timeout / connection / server_error 等待后重试；预算耗尽且策略允许时切到 fallbackChain。
模型不存在、内容策略阻止、响应结构为空或畸形 model_not_found / content_policy_blocked / responses_null_output / malformed_response 按规则重试、适配请求或切换模型；内容策略拒绝不会原样重试同一请求，不可恢复时停止并记录 trace。
provider 或聚合器账号策略阻止 provider_policy_blocked 这是账号、隐私或数据策略层面的确定性拦截。Axiomate 会停止并提示检查 provider/account policy、base URL、model 或 route，不会把它当作自动切模型原因。
provider 明确拒绝 streaming，或 streaming endpoint 返回 404 但不是模型不存在 streaming_unsupported / stream_endpoint_not_found 只在这些窄语义下切到 non-streaming fallback；空流、本地 stream 解析错误和普通 52x 会先走流式重试，不会直接降级。

allowActions 决定一条 route 或辅助任务允许哪些动作类型，switchModelOn 只决定哪些语义错误允许切模型。它们只是决策系统的输入，不会绕过观察和规则表直接 fallback。provider_policy_blocked 是诊断和配置修复原因，不是可写入 switchModelOn 的自动切模型原因。

content_policy_blocked 表示 provider 对本轮 prompt 做出了安全/内容过滤拒绝。Axiomate 不会把它当作 transient retry 原样重试；只有当 allowActions 包含 switch_model、switchModelOn 包含 content_policy_blocked，且 fallbackChain 有不同模型时，才会切到候选模型。

每一轮失败都会重新观察错误原因，并从当前 primary / fallbackChain / policy 生成一份独立的 fallback 可用性快照。也就是说，上一次是 rate_limit 并允许切模型，不会污染下一次 model_not_found 或 server_error 的判断。/doctor 中看到的 actionAllowed / reasonAllowed 是这轮请求的诊断快照，不是需要手动写进配置的字段。

/doctor API Providers：怎么看 API 失败

是什么：/doctor 里的 API Providers 区域会把当前会话中最近的 API 恢复 trace 投影成可读卡片。它不是配置入口，也不会主动修复请求；它用于回答“刚才哪个 provider / model / route 出了什么问题，Axiomate 观察到了什么，决策系统做了什么，下一步应该检查哪里”。

怎么用：当主对话、辅助任务、模型验证或流式输出出现失败、明显变慢、切换模型、非流式降级、空输出、52x、限流、模型不存在等现象时，输入 /doctor。如果当前会话没有 API 恢复记录，这个区域不会显示。当前 UI 最多显示 5 张最近卡片；更早的卡片会用隐藏数量提示。

后台任务怎么读：prompt_suggestion、标题生成、权限解释等后台辅助任务失败时，卡片通常是 [Info] 或轻量 [Warning]。例如 scope: auxiliary:promptSuggestion 表示输入框淡色建议失败或被取消，不代表主对话回复失败；它应该检查 auxiliary.promptSuggestion，不是检查当前主 route。

能力探测怎么读：scope: capability:count_tokens 表示 provider 原生 token counting 探测失败。它不是主对话请求失败；Axiomate 会继续尝试 auxiliary.tokenCounting 或本地估算。只有 scope: auxiliary:tokenCounting 才表示用于 token 统计的辅助模型链本身失败。

记录范围：这些记录只保存在当前进程内存里，最多保留最近 200 条安全 trace 事件。/clear、切换或恢复其他会话会清空它们。trace 不落盘，不保存 prompt、图片内容、工具结果、API key、Authorization header 或 provider 原始响应正文；只保留协议、模型、route/task、错误分类、HTTP 状态、少量安全响应头、request id、时间和恢复动作。

卡片字段 含义 怎么判断
[Error] / [Warning] / [Info] 和标题 这张卡片的严重程度和最终状态摘要。 Error 通常需要改配置、换模型或处理账号/网络；Warning 表示发生了恢复动作；Info 通常表示请求已被适配或局部恢复。
scope 失败发生在哪个范围：主 route、某个 auxiliary task、模型验证或其他 API 操作。 route:deepseek-main 说明看 model.routes.deepseek-main；auxiliary:goalJudge 或 auxiliary:promptSuggestion 说明看对应 auxiliary task。
impact 这次失败影响的用户流程，例如主响应流、辅助推理、token 计数、模型验证或非流式降级。 主响应流失败会影响正在聊天；辅助任务失败可能只影响标题、摘要、goal 判断或 hook。
model 本轮恢复中经过的模型路径。 a -> b 表示决策系统已经从 a 切到 b；只有一个模型说明没有实际切模型，可能只是重试或改请求。
observed 观察层归类出的语义错误原因，可能附带 HTTP 状态和 provider request id。 先看这里判断是账号、计费、限流、超时、模型名、协议兼容、上下文过长还是 provider 服务端错误。
recovery 决策和执行结果的短摘要。 这里会说明已恢复、已切模型、已改请求、被策略阻止、预算耗尽或转交给压缩等情况。
stopped 如果恢复停止，这里解释停止原因。 常见原因包括不可重试、重试预算耗尽、没有可用 fallback 模型、route policy 不允许切模型或请求被取消。
next 面向用户的下一步建议。 它会指向具体配置字段，例如 models["..."].apiKey、models["..."].baseUrl、model.routes["..."].fallbackChain 或 switchModelOn。
timeline 同一恢复会话里的 attempt 序列。 格式是 #attempt/max reason HTTP xxx -> action/outcome +mutation。UI 默认显示最后 3 个事件；内部使用 sequence 排序，所以同一毫秒内的多次观察也能保持顺序。
request / timing / advanced / policy 可选诊断字段：request id、超时/耗时、协议、operation、route/task、命中规则、安全响应头和策略快照。 policy 里的 actionAllowed / reasonAllowed 是本轮诊断结果，不是配置字段。配置仍然写 allowActions 和 switchModelOn。

/doctor 里的恢复状态

状态 含义 用户应该关注什么
recovered 请求经过重试或适配后成功。 一般不需要处理；如果频繁出现，检查 provider 稳定性和 route 顺序。
switched_model 决策系统执行了 fallback_model，实际切到了 fallback 模型。 确认 fallback 模型质量、成本、上下文窗口和工具/图片能力是否符合这个 route 的预期。
switched_request_mode 流式请求失败后切到非流式请求。 这是请求模式降级，不是切模型。若频繁出现，优先检查 provider 的 streaming 兼容性。
adapted_request Axiomate 修改了请求形状，例如去掉不支持的字段、降低输出预算、关闭 thinking、重写图片 payload。 通常说明 provider 或模型不完全支持当前请求参数；可检查 vendor、template、extraParams 和模型能力配置。
adaptation_failed 已经尝试改请求，但后续仍失败。 看 timeline 中第一次错误和最后一次错误是否不同；可能需要改协议、换 provider 或调整 route policy。
delegated_recovery 恢复动作被交给外层机制，例如会话压缩或主 route 循环继续处理。 这不是“丢失错误”。它表示本层已经记录观察和决策，并把执行交给更外层的恢复循环。
blocked_by_policy 错误本来可以考虑切模型，但当前 route 或 auxiliary policy 不允许。 检查 allowActions 是否包含 switch_model，switchModelOn 是否包含这类错误，以及 fallbackChain 是否有可用候选。
salvaged 流式输出中断，但已有部分输出被安全保留。 检查最终回复是否完整；若经常发生，优先看网络、provider streaming 和超时策略。
exhausted 可重试错误耗尽了当前预算。 需要检查 provider 状态、提高预算、添加 fallback 模型，或减少上下文和 payload。
aborted 请求被用户或系统取消。 如果是主动取消，无需处理；如果不是，检查是否有超时、退出或会话切换。
failed 请求失败且没有可执行的恢复动作。 按 observed 和 next 检查密钥、计费、模型名、baseUrl、协议和账号权限。
retrying 当前 trace 显示请求仍在恢复过程中。 等待当前请求结束后再看最终卡片；如果长时间停留，检查超时和网络。

读卡片的顺序：先看 scope 定位配置块，再看 observed 确认错误类别，然后看 timeline 判断恢复过程，最后按 next 修改 models、model.routes 或 auxiliary。不要直接照抄 policy 里的 actionAllowed / reasonAllowed；它们只是这次决策的结果快照。

/model 命令

/model 是当前模型配置的主要 UI。它能添加模型、编辑单个模型 entry、切主 route、管理 fallback 和辅助任务策略。

命令 作用
/model 打开模型选择器。选择模型会持久化更新当前默认 route 的 primary。
/model show 显示当前主模型 route、fallback chain 和会话 override 信息。
/model add 交互式添加 provider/model：协议、baseUrl、apiKey、model id、context window、图片支持、vendor template、thinking、User-Agent，并验证连接后写入 ~/.axiomate.json。
/model edit <model-id> 用编辑器修改 models[model-id]。保存时会校验 JSON、模型字段、route 和 auxiliary 引用。
/model use <model-id> 把当前默认 route 的 primary 持久化改成指定模型。
/model route list 列出所有主对话 route。默认 route 前会有标记。
/model route show [route-id] 显示某条 route 的 chain、profile、allowActions、switchModelOn。
/model route <route-id> 或 /model default <route-id> 把 model.defaultRoute 切到指定 route。
/model route create <id> <model-id> 创建一条新主 route。
/model route delete <id> 删除非默认 route。默认 route 不能直接删除。
/model route rename <from> <to> 重命名 route，并保持默认 route 引用一致。
/model route policy <id> allowActions|switchModelOn|recoveryProfile <value> 修改 route 的恢复策略字段。数组字段用逗号分隔。
/model fallback list|add|remove <model-id> 查看或修改当前默认 route 的 fallback chain。
/model aux list / /model aux show <task> 查看辅助任务模型链和策略。
/model aux set <task> <model-id> 设置辅助任务的 primary。
/model aux fallback list|add|remove <task> <model-id> 查看或修改辅助任务 fallback chain。
/model aux policy <task> failure|timeoutMs|maxOutputTokens|allowActions|switchModelOn|recoveryProfile <value> 修改辅助任务最终失败处置、超时或恢复策略字段。

/template vendor list|show|new|delete 用于管理 provider/vendor template，修改顶层 templates；/template model list|show|new|delete 用于管理 model template，修改顶层 modelTemplates。这些命令不修改 route。

Settings 里的模型 UI

Settings → Config 里的 Model 项使用同一个模型选择器。它适合快速切换当前主 route 的 primary 和调整 picker 中的 effort 选择。

这里的选择会写入 ~/.axiomate.json：Axiomate 先读取当前 model.defaultRoute，再把 model.routes[defaultRoute].primary 改成你选择的模型。它和 /model 选择器、/model use <model-id> 是同一种持久化语义，不是只在当前会话生效的临时 override。

Settings 里的 Model 不是完整 route 编辑器。它不会创建 route、编辑 fallback chain 或设置 auxiliary policy。完整路由和辅助任务管理请使用 /model route ...、/model fallback ...、/model aux ... 或手动编辑 ~/.axiomate.json。

/model add 的 onboarding 向导会在添加第二个及之后模型时询问用途：设为主 route primary、加入主 route fallback，或只写入 models 暂不加入路由。

手动配置流程

  1. 打开 ~/.axiomate.json。
  2. 在 models 中添加一个新 key，例如 "custom-main"。这个 key 是 Axiomate 内部引用名，route 和 auxiliary 里写的都是它。
  3. 填好 model、protocol、baseUrl、apiKey，再按模型实际能力补 contextWindow、maxOutputTokens、supportsImages、thinking、extraParams 等字段。
  4. 把这个 key 写进 model.routes.<route-id>.primary、fallbackChain 或 auxiliary.<task>.primary。
  5. 保存后重新打开 Axiomate，或用 /model show、/model route list、/model aux list 检查。

会话级 override

有些入口会临时指定主模型，例如启动参数或当前会话里的模型选择。这类 override 是内部 MainModelOverride 语义：可以指向默认 route、命名 route，或单模型 route。

会话 override 的目标是“这次会话用什么”，不是“改写全局策略”。它不会隐式重写 model.defaultRoute、route primary 或 auxiliary policy。普通 Settings → Config 的 Model、/model 选择器和 /model use 则是持久化修改当前默认 route primary。

常见检查

现象 检查项
启动时报模型不存在 确认 model.routes.*.primary、fallbackChain、auxiliary.*.primary 都引用 models 中存在的 key。
图片请求失败 纯文本模型把 supportsImages 设为 false；需要处理图片输入的主 route 应指向支持图片的模型。
目标评审成本太高 把 auxiliary.goalJudge.primary 指向便宜、稳定、严格 JSON 的模型。
fallback 没有发生 确认 route/aux policy 允许 switch_model，错误原因包含在 switchModelOn，并且 fallbackChain 里有不同于当前模型的候选。某些错误会优先 adapt_request、请求压缩、非流式 fallback 或停止。
Settings 里找不到 route 管理 Settings 目前只提供 Model 快速选择；route 和 auxiliary 管理在 /model 命令里。

建议第一次配置时优先用 /model add，确认能连接后再手动精修 route、fallback 和 auxiliary policy。
