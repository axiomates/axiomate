Checkpoints — Automatic File Snapshots and Rewind

On every turn of your conversation, Axiomate quietly snapshots the working directory into a separate shadow git repository. If you regret a change, /rewind can return the working directory to any previous turn and roll the conversation back to that moment as well. The whole system is almost transparent in daily use, but it gives you a fallback when it matters.

1How it works

Checkpoints are stored in an independent git repository under ~/.axiomate/checkpoints/store/. This repository is completely separate from your project's own git repository. Before Axiomate is about to modify files, it writes a commit to this shadow repository to record the current state.

Key design points

Shadow repository

It does not pollute your project's .git/. Your normal git log and git status never show these snapshots.

One ref per project

Different working directories write to different refs/axiomate/<hash> refs, so they stay isolated.

Structured commit subject

The format is axiomate:<messageId>:<label>, which can be traced back to a conversation turn.

Respects .gitignore

When scanning the working directory, ignored folders such as node_modules/ are not captured.

2Automatic behavior No manual step

When snapshots are created automatically

Before each model call to a file-editing tool (Write / Edit / NotebookEdit), Axiomate writes a commit to the shadow repository. This is enabled by default and requires no setup.

When snapshots are skipped automatically

ConditionBehavior
The current directory contains more than 50,000 filesSkip this snapshot because the cost is too high.
A single file is larger than 10 MBThat file is left out of the index; other files are captured normally.
The current directory is /, ~, or a drive rootReject the whole snapshot to avoid accidental broad capture.
Nothing changed since the previous snapshotDo not write an empty commit.
git is not installedThe checkpoint subsystem becomes a no-op and does not raise an error.

When cleanup runs automatically

After you have been idle for at least 1 minute and the process has been running for at least 10 minutes, Axiomate runs one background prune. Cleanup includes:

  • Orphans: project refs for working directories you manually deleted.
  • Stale refs: project refs that have not been touched for more than 14 days.
  • Size cap trimming: when all snapshots together exceed 500 MB, the oldest ones are removed first.
  • Session keep refs: if a ref about to be deleted still has a commit referenced by a session from the last 28 days, that commit is anchored under refs/axiomate/_keep/ until the corresponding session expires.

Prune runs at most once every 24 hours, controlled by .last_prune. It does not run repeatedly. All prune operations are fail-soft: errors are logged, not thrown.

3User commands Run explicitly

/rewind — Return to a conversation turn

Enter /rewind in the REPL to open an interactive picker that lists every turn in the current session. After you select a turn:

  • The conversation state returns to that turn, and later messages disappear.
  • The working directory is restored to match the snapshot exactly. Files in the snapshot are checked out; files that are on disk but weren't in the snapshot tree are removed.
Note: rewind restores the full working directory to match the target snapshot — including files Axiomate's tools never touched. Manual edits to those files are overwritten. To make this safe, Axiomate automatically takes a "pre-rewind" safety snapshot right before the restore, labeled pre-rewind:<targetMsgId>. If the rewind destroyed something you wanted to keep, open /rewind again, select that pre-rewind row, and rewind to it — your previous state comes back.

Empty-workdir behavior

If you start Axiomate in a brand-new empty directory and then ask it to create the first file, that creation gets anchored to a synthetic root snapshot (an empty-tree commit). Rewinding to that root removes any file Axiomate created since.

Picker view: code-restore anchors vs all turns

The picker has two views. The default view lists only code-restore anchors — turns where the AI itself ran a file-modifying tool (Edit / Write / NotebookEdit) and a snapshot was written for that turn. Every row in the default view can restore code, restore conversation, or both. There is no ⚠ No code restore noise here by design.

Press Tab to switch to the all-turns view, which adds every other selectable user turn (read-only chats, slash commands, prose-only prompts). Those rows can only do conversation-only rewind — useful for "cut the dialogue back to the moment before I ran /clear" or "drop the last question I asked." Press Tab again to return to anchors-only. The footer line tells you which view you are in and how many entries the toggle exposes:

  • Default: Tab to show all 8 turns (5 conversation-only)
  • After Tab: Tab to show only code-restore anchors (5 hidden)

The view toggle is per-mount — closing and re-opening /rewind resets to the default (anchors-only). Filtering is purely a view-layer thing; the underlying message list is not modified, and any visible row rewinds to the correct point regardless of which view is active. In a brand-new session that has not produced any code anchors yet, the picker automatically falls back to the all-turns view so conversation rewind is still reachable.

Command-line version

The same subcommands can also be run from the axiomate command line without entering the TUI: 

axiomate checkpoints status
axiomate checkpoints status --rows 50
axiomate checkpoints list
axiomate checkpoints list --rows 50
axiomate checkpoints prune --force
axiomate checkpoints prune --retention-days 30 --max-size-mb 1024
axiomate checkpoints prune --keep-orphans --force
axiomate checkpoints clear --force

Each subcommand rejects unexpected positional arguments with too many arguments rather than silently dropping them. clear requires --force on the CLI side because it bypasses the interactive confirmation that the slash version has.

Row count for status and list

Both status and list render a tabular summary whose row count is capped at 30 by default. To show more (or fewer) rows, pass an integer in 1..500. Slash commands take it as a positional arg (/checkpoints status 50, /checkpoints list 50); the CLI uses --rows N to match other CLI flags (axiomate checkpoints status --rows 50). Out-of-range or non-integer values are rejected.

To make the override permanent, set checkpointsStatusRows in Settings (the Checkpoints rows shown entry cycles through 10/20/30/50/100/200/500), via /config checkpointsStatusRows N, or by hand-editing ~/.axiomate.json. The per-call override always wins over the config value.

--keep-orphans — Preserve refs for temporarily-missing working directories

By default, prune deletes refs whose working directory has vanished from disk. If a working directory is only temporarily gone — an external drive is unplugged, an in-flight rename you plan to undo, or a re-clone you intend to do later — pass --keep-orphans (CLI) or keep-orphans (slash): this run skips the orphan pass while stale and size-cap still execute. /checkpoints prune keep-orphans works inside the REPL as well. The number of refs the safety valve protected appears as Orphan refs skipped: N in the output.

The Snapshot metrics block reports a rolling window of up to the last 100 snapshot attempts (the last N count grows to 100 as you accumulate snapshots). The block is suppressed entirely until the first snapshot is recorded, and the p50 / p95 values show — until at least two ok snapshots have been measured.

5Failure modes Fallback behavior

The checkpoint subsystem's core contract is fail-open: a snapshot failure must never block the main workflow. That means:

  • git is not installed → the whole subsystem becomes a no-op and does not raise an error.
  • A git command fails transiently, such as EBUSY or timeout → this snapshot is skipped, and the next turn retries automatically.
  • The working directory has more than 50,000 files → the snapshot is skipped to avoid a stuck git add.
  • The disk is full or permissions are insufficient → the error goes to logForDebugging, and the conversation continues.

Cross-working-directory edge cases

If you /resume an old session while your current working directory differs from the original one, axiomate shows a one-line system message before /rewind:

  • reachable: the snapshot is in the current ref, so rewind can proceed normally.
  • reachable-other-worktree: the snapshot exists, but belongs to another working directory. You are prompted to cd there.
  • unreachable: the snapshot has already been deleted by prune, so rewind fails.

Orphan warning during cleanup

/checkpoints status may end with a line like this: 

Note: 3 snapshots from 1 orphan workdir will be discarded on next prune.

If you see this line and the commits for that working directory are still useful, for example because an external drive was just unplugged, restore the working directory to the original path. The next prune will recognize it as live again.

6FAQ

Will this pollute my git repository?

No. All commits are written to ~/.axiomate/checkpoints/store/, completely separate from your project's own .git/. git status, git log, and git push cannot see these commits.

How much disk space does it use?

The hard limit is 500 MB, and it is configurable. git performs object deduplication and zlib compression, so editing the same file 100 times only stores a few deltas in practice. Normal daily use is usually around 100-300 MB.

Can checkpoints sync across machines?

No. Checkpoints are local snapshots. They are not designed with cross-machine sync, encryption, or signing. On another machine, treat the state as having no checkpoint history.

Can I export or import checkpoints?

There is no checkpoints export or import yet. The shadow repository itself is a standard bare git repository, so git-savvy users can inspect it directly with git --git-dir=~/.axiomate/checkpoints/store ....

How do I turn it off completely?

Set fileCheckpointingEnabled to false in Settings, or set the environment variable AXIOMATE_CODE_DISABLE_FILE_CHECKPOINTING=1. Snapshots that already exist are not cleaned up automatically; run /checkpoints clear manually if you want to remove them.

/rewind says the commit cannot be found. What should I do?

Most likely, that turn's commit was already pruned because it was older than 14 days and no current session referenced it. This is expected behavior. For important milestones, keep a separate copy of the relevant working directory output instead of relying only on checkpoints.
