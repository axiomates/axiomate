Checkpoints — Fájlpillanatképek és beszélgetés-visszajátszás

Az Axiomate /rewind parancsa két független tengelyen dolgozik. File tengely: minden fájlmódosító eszköz hívása előtt a háttérben pillanatképet készít a munkakönyvtárról egy külön git shadow repóba; bármelyik korábbi turn állapotára visszaállíthatod a lemezt. Conversation tengely: a JSONL session-naplóra építve visszavisz egy adott Q/A párbeszédhez, és az input mezőt is feltölti a következő prompttal, hogy újra tudd csinálni azt a turn-t. A két tengely 100%-ban szét van választva — a fájlok nem piszkálják a beszélgetést és fordítva. Az egész rendszer szinte észrevehetetlen — hétköznapi használat mellett alig érezni, de a kritikus pillanatokban megment.

1Hogyan működik

A checkpoints egy önálló git repó a ~/.axiomate/checkpoints/store/ könyvtárban, amely teljesen el van különítve a saját projekted git repójától. Az Axiomate minden fájlmódosítás előtt egy commit-ot ír ebbe a shadow repóba, rögzítve az aktuális állapotot.

Kulcsfontosságú tervezési döntések

Shadow repo

Nem szennyezi a projekted saját .git/ könyvtárát. A git log, git status egyáltalán nem mutatja ezeket a pillanatképeket.

Projektenként egy ref

A különböző munkakönyvtárak különböző refs/axiomate/<hash> ref-ekbe íródnak, nem zavarják egymást.

Strukturált commit subject

Formátum: axiomate:<messageId>:<label>, visszakövethető a beszélgetés turn-jére.

Tiszteletben tartja a .gitignore-t

A munkakönyvtár bejárásakor a figyelmen kívül hagyott könyvtárak — mint a node_modules/ — nem kerülnek be a pillanatképbe.

Előnyök a hasonló rendszerekkel szemben

Az Axiomate file checkpointing-ja a hermes-agent pre-tool munkakönyvtár pillanatkép modelljére épül — ami már önmagában is stabil. Erre az alapra az axiomate több olyan dolgot is ráépített, amit a hermes nem tud:

Kéttengelyes rewind + dupla kurzor

A Hermes csak fájl-visszaállítást csinál. Az axiomate hozzáadott egy független beszélgetés-visszajátszási tengelyt — a picker-ben Tab-bal átválthatsz a Conversation fülre, visszatérhetsz bármelyik Q/A párhoz, és az input mező automatikusan visszatölti a következő prompt-ot, hogy újra tudd futtatni. A File és a Conversation teljesen szét van választva. A selection kurzor (▶) és a head kurzor ((current)) fogalmilag el vannak választva — a vizuális kurzorod nem egyenlő a session állapotával. Áttekinthetőbb, mint a hermes "kurzor = állapot" egyesített modellje.

Beszélgetés rewind perzisztencia + jövőbeli node-ok láthatók

A Conversation rewind head állapota beíródik a JSONL-be (egy type:"head" bejegyzés). /exit után a UI-ban resume-olva ugyanoda kerülsz vissza, ahol a rewind után voltál — ellentétben a hagyományos memóriában tartott csonkolással, ami kilépéskor elvész. Rewind után a head-nél újabb turn-ök a picker-ben továbbra is láthatók halvány szürkével (az adat a JSONL-ben megmaradt), és bármikor visszaválaszthatók. A "rewind után vissza lehet lépni előre" az axiomate gondolkodásmódjának egyik alapköve.

Pre-rewind biztonsági háló minden rewind-nál

A Hermes rewind-ja egyirányú művelet — ha visszaléptél, a későbbi állapotok elvesznek. Az Axiomate minden rewind előtt készít egy biztonsági pillanatképet az aktuális munkakönyvtárról; ha meggondoltad magad, válaszd ki a picker-ben a ↶ sort a rewind visszavonásához. Korlátlanul egymásba ágyazható — a rewind a rewindon belül sem veszít el adatot.

Egymásba ágyazott rewind hash láncként jelenik meg

Több egymás utáni rewind után a picker ezeket a ↶ sorokat ↶ Rewind to <target-hash> [own-hash] formában jeleníti meg — egy pillantással láthatod, "melyik rewind-ból indult ki ez a másik." A hierarchia szövegesen explicit, nem fejben kell tartani.

Event-aligned CHANGES oszlop (a git log szemléletével)

A picker, list és status CHANGES oszlopa azt írja le, hogy az adott turn mit írt (anchor[i] vs anchor[i-1]), ugyanúgy, ahogy a git log-ot olvasod. A Hermes eredeti commit-vs-parent modellje off-by-one hibás: egy sor CHANGES oszlopa valójában a következő sor turn-jéhez tartozik, ami elsőre zavaró.

Magasabb alapértelmezett limitek (modern fejlesztői gépekhez)

Alapértelmezett korlátok: 5000 pillanatkép (a /config alatt állítható) / 60 nap megőrzés / 5 GB tároló / 50 MB fájlonként / 200 000 fájl munkakönyvtáranként / 5 000 sor maximum. A Hermes alapértelmezetten 7 nap megőrzést használ, ami sprint-határokon könnyen adatvesztéshez vezet; az axiomate 60 napos megőrzése + 5 GB tárolója illeszkedik a modern fejlesztési ciklusokhoz, beleértve a szabadságokat és a sprint-váltásokat is.

UX előny: newest-first + adaptív időbélyegek

A picker alapértelmezetten a legújabbat mutatja felül (a hermes-ben nincs ilyen), így a legutóbbi művelet azonnal látható megnyitáskor. Az időbélyegek adaptívak: épp most → just now, percek → 5m ago, napok → abszolút dátum.

Teljes /checkpoints CLI

A list mellett az axiomate kínál status / prune / clear alparancsokat és parancssori változatot is. A --keep-orphans biztonsági szelep megakadályozza a pillanatképek elvesztését, ha a külső lemez le van választva.

2Automatikus viselkedés Nem igényel kézi beavatkozást

Mikor készül automatikusan pillanatkép

Minden alkalommal, amikor a modell fájlmódosító eszközt (Write / Edit / NotebookEdit) készül meghívni, az Axiomate egy commit-ot ír a shadow repóba. Mindez alapértelmezetten be van kapcsolva, konfiguráció nélkül.

Mikor ugrik át automatikusan

Mikor fut automatikus tisztítás

Az Axiomate akkor futtat a háttérben egy prune műveletet, ha legalább 1 perce inaktív vagy ÉS az indulás óta eltelt legalább 10 perc. A tisztítás a következőket takarítja:

  • Orphan (árva): a kézzel törölt munkakönyvtárak ref-jei törlésre kerülnek.
  • Stale (elavult): 60 napnál régebben nem érintett projekt ref-ek.
  • Size-cap (méretkorlát): ha az összes pillanatkép mérete meghaladja az 5 GB-ot, a legrégebbiekkel kezdve töröl.
  • Session-anchoring (keep-ref): ha egy törlésre ítélt ref-en van olyan commit, amire az utolsó 28 napban valamelyik session hivatkozik, az anchor-olásra kerül a refs/axiomate/_keep/ alá, és addig megmarad, amíg a hivatkozó session is le nem jár.

A tisztítás legfeljebb 24 óránként egyszer fut (ezt egy .last_prune jelölőfájl vezérli), nem ismétlődik feleslegesen. Minden prune fail-soft: hibák esetén csak naplóz, nem dob kivételt.

3Felhasználói parancsok Kézi meghívás

/rewind — Visszatérés egy turn elé

A REPL-ben beírod a /rewind parancsot, és megjelenik egy interaktív picker, amely felsorolja az aktuális session turn-jeit és a korábbi pillanatképeket. Minden sor címkéje Before "<prompt>" formátumú — egy sor kiválasztása azt jelenti, hogy visszaállítod az állapotot arra a pontra, mielőtt az a turn lefutott volna.

Két fül: File vs Conversation

A picker alapértelmezetten a File fülön nyílik meg. Tab-bal átválthatsz a Conversation fülre. A két fül két független alrendszer: más forrásból származik a listájuk, más a sorok jelentése, más a chooser művelete, és más a perzisztens tárolójuk. Az alábbiakban mindkét fül saját szekcióban kerül bemutatásra.

File fül — Fájl-visszaállítás

Mentális modell: minden anchor egy pre-tool pillanatkép

Az Axiomate minden eszközhívás előtt készít pillanatképet (a hermes-agenttel megegyezően). Így az anchor tree-je azt írja le, "hogyan nézett ki a lemez, mielőtt az a turn lefutott volna". A picker sorok explicit Before "<prompt>" formátumot használnak, hogy elkerüljék az anchor és a "git log commit" összekeverését. Minden sor jobb oldalán a stats (file +X -Y) azt írják le, hogy mi történt a lemezen az előző anchor és ezen anchor között, ami a git log commit-vs-parent szemléletét követi. Az adatforrás a shadow repo (~/.axiomate/checkpoints/store/), független a JSONL session-naplótól.

Csak azok a turn-ök jelennek meg, amelyek ténylegesen módosítottak fájlokat a lemezen (olyan turn-ök, ahol az AI Edit / Write / NotebookEdit hívást végzett, vagy Bash-en keresztül változtatott fájlokat). A tisztán chat-jellegű / csak olvasható válaszok nem jelennek meg.

Rendezés: newest-first

A picker a /checkpoints list parancshoz hasonlóan a legújabbat mutatja felül: 

(current)
↶ Rewind to 041da07 [71c90d4]    test.txt +1 -1   just now
↶ Rewind to before "v2 改成 v3" [041da07]   test.txt +1 -1   just now
Before v2 改成 v3                test.txt +1 -1   2m ago
Before v1 改成 v2                test.txt +1 -1   3m ago
Before 创建 test.txt 内容 v1    test.txt +1 -0   4m ago

A kurzor alapértelmezetten a második soron áll (a legutóbbi visszaállítható célpont). A (current) nem választható ki — ez csak elrendezési referenciapont, a "most". g az első választható sorra ugrik (átugorva a (current)-et), G a végére ugrik. j / k fel-le mozgás.

Chooser: mit lehet tenni a sor kiválasztása után

  • Van diff: Restore file / Never mind. Visszaállítja a lemezt arra az állapotra, amilyen ezen anchor előtt volt, a chat változatlan marad.
  • Nincs diff: csak Never mind (az anchor.tree megegyezik a jelenlegi lemezzel, a visszaállítás no-op).

Pre-rewind biztonsági háló (safety net)

Minden Restore file művelet előtt az Axiomate automatikusan készít egy pre-rewind biztonsági pillanatképet (a subject formátuma: axiomate:<synth-uuid>:pre-rewind:<hash>). Ennek a tree-je a rewind művelet végrehajtása előtti lemez- állapot. Ha a rewind felülírta a fontos változtatásaidat, nyisd meg újra a /rewind-et, válaszd ki azt a ↶ Rewind to before "<X>" sort → Restore file, és visszakapod az eredeti állapotot. A biztonsági háló korlátlanul egymásba ágyazható: egy rewind művelet visszavonása is készít egy újabb biztonsági pillanatképet.

A picker sorok típusai

Minden ↶ sor végén egy [7-karakteres rövid hash] áll — ez az adott anchor saját git short id-ja. A sima turn sorok végén nincs hash (a felhasználó prompt-ja önmagában azonosítja a sort). Minden anchor-ral rendelkező sor jobb oldalán stats + idő látható.

  • Before "创建 test.txt 内容 v1" — egy normál turn anchor-ja. A File fülön a Before előtag emlékeztet, hogy ez a pillanatkép az " lefutása előtti" állapot.
  • ↶ Before "<prompt>" [<hash>] — olyan turn anchor-ja, amely már nincs az aktuális chain-en. Például ha a conversation-t már rewind-olták, de a turn által hagyott fájlállapot még visszaállítható.
  • ↶ Rewind to before "<target>" [<hash>] — egy rewind művelet pre-rewind biztonsági hálója. Kiválasztása visszavonja azt a rewind-et. Az idézőjelben lévő szöveg annak a turn-nek a prompt-ja, ahová a rewind vissza akart térni.
  • ↶ Rewind to <target-hash> [<own-hash>] — egy egymásba ágyazott rewind biztonsági hálója (egy rewind művelet visszavonásának visszavonása). A csupasz 7-karakteres hash annak a célnak a short id-ja, amire hivatkozik; a szögletes zárójeles hash ennek a sornak a saját short id-ja. Vizuálisan: csupasz = "kire hivatkozom", zárójeles = "ki vagyok én", így a hash-ek mentén vissza lehet követni a teljes rewind láncot.
  • ↶ Off-branch anchor [<hash>] — beszélgetési kontextus nélküli anchor (örökölt session-ből vagy külső közvetlen commit-ból a store-ba).
  • (current) — az aktuális lemez-állapot helyőrzője, nem választható ki.

A sorok időbélyegzése

A stats oszloptól jobbra egységes időformátum látható, a /checkpoints list szabályai szerint: just now (<1 perc) / 5m ago (<1 óra) / 3h ago (<24 óra) / 2026-05-24 09:42 (≥24 óra).

A CHANGES oszlop jelentése

Minden sor stats értéke azt írja le, hogy az adott turn maga mit írt (nem az előző sort). Algoritmus:

  • A legújabb sor: anchor.tree vs aktuális lemez ("a legújabb turn lefutása után így néz ki a lemez"). Ha az axiomate-on kívül kézzel módosítottál fájlokat, ennek a sornak a stats-ja követi a változást — az anchor.tree és az aktuális lemez közötti összes különbséget mutatja.
  • A többi sor: anchor.tree vs az előző (újabb) anchor tree-je. Ez ekvivalens azzal, hogy "a turn lefutása után ez lett a következő anchor állapota", frozen, nem változik.
  • file +X -Y / N files +X -Y egy/több fájl változás. (no diff) a turn nem módosított fájlokat (pl. tisztán chat / csak olvasható válasz).

Visszajelző üzenetek

A Restore file befejezése után a transcript-ben megjelenik egy sor: 

● File rewound to "v2 改成 v3". Use /rewind → "↶ Rewind" row to undo.

A Restore conversation nem ad visszajelzést — a csonkolt beszélgetési lánc önmagáért beszél (a levágott turn-ök eltűnnek a képernyőről, a vágási pont prompt-ja visszakerül az input mezőbe).

Üres könyvtár viselkedése

Ha egy teljesen üres könyvtárban indítod el az Axiomate-ot, és az létrehozza az első fájlt, annak a turn-nek az anchor-ja egy szintetikus gyökér-pillanatkép (üres tree). Erre az anchor-ra való visszaállítás törli az Axiomate által létrehozott összes fájlt.

Slash parancsok nem jelennek meg a picker-ben

A /checkpoints status, /help, /clear és hasonlók nem minősülnek turn-nek — ezek a REPL-ben futtatott eszközhívások, a picker automatikusan kiszűri őket. A "rewind a /checkpoints status elé" és a "rewind az azt megelőző valódi turn elé" ugyanaz az állapot.

Conversation fül — Beszélgetés visszajátszás

Mentális modell: minden sor = egy Q/A párbeszéd

Az aktuális chain-en lévő összes valódi user prompt-ot mutatja. Minden sor = egy teljes kérdés-válasz pár (user prompt + AI válasz). A (current) címke jelzi, hogy melyik párbeszéd "tetején" állsz — vagyis amelyik turn-re a head mutat. Az adatforrás a JSONL session-napló (~/.axiomate/projects/.../*.jsonl), független a lemezen lévő fájloktól.

Példa sor-megjelenítésre (hi → ok → good küldése után, rewind nélkül): 

good (current)
ok
hi

Fordított időrendi sorrend, a legújabb Q/A felül. A kurzor automatikusan a (current) sorra áll, ↑/↓ mozgatja felfelé/lefelé. Nincs olyan Before "<prompt>" előtag, mint a File fülön — itt a sor szövege maga a prompt.

Múlt / jövő: a jövőbeli sorok halványak

Ha a rewind egy korábbi turn utánra mozgatta a head-et, a head-nél újabb turn-ök a picker-ben továbbra is láthatók (az adatuk a JSONL-ben megvan), de halvány szürkével jelennek meg — jelezve, hogy ezek a "jövőben" vannak, már eltűntek a chat-ből, de bármikor visszaválaszthatók és újra futtathatók. 

good          ← halvány szürke (jövő)
ok (current)  ← alapértelmezett szín (jelen)
hi            ← alapértelmezett szín (múlt)

Amíg ugyanabban a session-ben nem futtatsz /clear-t vagy /branch-et, a jövőbeli node-ok nem vesznek el, akárhányszor is rewind-olsz.

Két kurzor: selection és head

A Conversation fülön egyszerre két kurzor van — ne keverd össze őket:

  • Selection kurzor (bal oldali ▶) — a ↑/↓ billentyűkkel mozgatott vizuális kurzor, ez határozza meg, hogy Enter lenyomásakor melyik sor kerül kiválasztásra. A picker megnyitásakor automatikusan a head sorra áll.
  • Head kurzor (sor végi (current) címke) — a session aktuális állapotmutatója, "melyik párbeszéd tetején állok most". A head nem mozdul a ↑/↓ hatására, csak a Restore conversation művelet után változik.

A két kurzor átfedhet: a picker megnyitásakor a selection is a head soron van, így azon a soron egyszerre látszik a ▶ és a (current). ↑/↓ mozgatás után a selection máshová kerül, a head továbbra is a (current) soron marad — ez a két információ független egymástól.

A rewind szemantikája: csonkolás eddig a párbeszédig + input mező visszatöltése

A picker-ben válassz ki egy X sort, nyomj Enter-t, a chooser felajánlja a Restore conversation lehetőséget. Végrehajtás után:

  • Chat terület: csonkolás az X párbeszéd befejezéséig (X user prompt-ját és AI válaszát is beleértve); az X utáni turn-ök eltűnnek a képernyőről (az adat a JSONL-ben megmarad).
  • Input mező: automatikusan visszatölti az X utáni következő user prompt tartalmát — így "módosíthatod ezt a kérdést" és újra elküldheted. Ha X a chain vége (nincs utána következő), az input mező kiürül.
  • Head rekord: egy bejegyzés íródik a JSONL-be, jelölve, hogy most az X párbeszéd tetején állsz.

A (current) sor kiválasztása: no-op (már itt vagy, nincs értelme újra végrehajtani). Az Enter nem csinál semmit, a picker nyitva marad.

Perzisztencia: /exit után a resume ugyanoda hoz vissza

A Conversation rewind head-je egy type: "head" bejegyzésként íródik a JSONL-be. /exit + újraindítás + UI resume ugyanabba a session-be után pontosan ott folytatódik, ahol a rewind után voltál. Újra megnyitva a /rewind-et, a (current) címke továbbra is az X soron van — az állapot nem veszett el.

Amit a picker nem mutat

A következő elemek automatikusan ki vannak szűrve a zaj elkerülése érdekében:

  • tool_use / tool_result (a JSONL-ben type=user, de a content eszközvégrehajtási eredmény, nem valódi prompt)
  • Resume-kor injektált Continue from where you left off. folytatósor (isMeta=true)
  • /clear / /help és hasonló slash parancsok input/output wrapper-ei
  • Compact summary helyőrzők, transcript-only metaadatok
  • Tisztán attachment / readonly "nem-prompt" üzenetek

axiomate checkpoints prune --force
axiomate checkpoints prune --retention-days 90 --max-size-mb 10000
axiomate checkpoints prune --keep-orphans --force
axiomate checkpoints clear --force

Minden alparancs elutasítja a felesleges pozíciós paramétereket (too many arguments üzenettel), nem hagyja őket szó nélkül. A clear parancssori verzióban kötelező a --force, mivel itt nincs meg a slash változat interaktív megerősítése.

A status / list sorainak számának szabályozása

A status és list által renderelt táblázat alapértelmezetten legfeljebb 100 sort mutat. Ha többet (vagy kevesebbet) szeretnél látni, adj meg egy 1..5000 közötti egész számot. Slash parancsnál pozíciós paraméterként (/checkpoints status 50, /checkpoints list 50); parancssorban --rows N formában, a többi CLI flag-gel összhangban (axiomate checkpoints status --rows 50). A nem egész vagy határon kívüli értékek elutasításra kerülnek.

Ha tartósan szeretnéd felülírni az értéket, állítsd be a checkpointsStatusRows mezőt a Settings-ben (Checkpoints rows shown néven, a 30/50/100/200/500/1000/2000/5000 értékek között ciklikusan léptethető), vagy használd a /config checkpointsStatusRows N parancsot, esetleg szerkeszd közvetlenül a ~/.axiomate.json fájlt. A hívásonkénti pozíciós paraméter (slash parancs) vagy --rows N (CLI) mindig elsőbbséget élvez a config értékkel szemben.

--keep-orphans — Ideiglenesen hiányzó munkakönyvtár ref-ek megőrzése

Alapértelmezetten a prune törli azokat a ref-eket, amelyek munkakönyvtára már nincs a lemezen. De ha csak ideiglenesen húztad ki a külső lemezt, átneveztél egy útvonalat és később visszaállítanád, vagy tervezed, hogy később újra klónozod ugyanoda, adj hozzá --keep-orphans (CLI) vagy keep-orphans (slash parancs) kapcsolót: ez a prune futás kihagyja az orphan köröket, a stale és size-cap normál módon lefut. A /checkpoints prune keep-orphans a REPL-ben is működik. Az így átugrott ref-ek száma Orphan refs skipped: N formában jelenik meg a kimenetben.

A Snapshot metrics blokk az utolsó legfeljebb 100 pillanatkép-kísérlet gördülő ablakát mutatja (last N értéke a kísérletekkel együtt 100-ig nő). Az ablak adatai a ~/.axiomate/checkpoints/metrics.jsonl fájlban vannak, újraindítás után is megmaradnak. Az első pillanatkép előtt ez a blokk egyáltalán nem jelenik meg; a p50 / p95 értékekhez legalább 2 ok minta kell, egyébként — jelenik meg.

/checkpoints list

Checkpoints for C:\public\test:

  WHEN      ID       CHANGES         TURN
  just now  bd10d75  test.txt +1 -1  Before "v1 改成 v2"
  just now  49f18dc  test.txt +1 -0  Before "创建 test.txt，内容 v1"

Roll back to a turn with /rewind (interactive selector).

Oszlopok jelentése:

  • WHEN — az anchor időbélyegzője, adaptív formátumban (<1p = just now, <1ó = Nm ago, <24ó = Nh ago, ≥24ó = 2026-05-24 09:42).
  • ID — git short hash (7 karakter). Közvetlenül használható külső git parancsokkal, pl. git --git-dir=~/.axiomate/checkpoints/store show 8da1461. A teljes hash + message UUID a --debug naplóban látható.
  • CHANGES — ugyanaz a forrás, mint a picker-ben: "mit írt ez a turn". A legújabb sor anchor-vs-disk, a többi sor a szomszédos anchor-ok közötti diff. file +X -Y / N files +X -Y egy/több fájl változás, (no diff) a turn nem módosított fájlokat.
  • TURN — a sor címkéje, amely megmondja, melyik turn pre-tool pillanatképéről van szó. Formátuma: Before "<prompt>" vagy Undo rewind to before "<target>" (pre-rewind biztonsági háló).

Minden anchor megjelenik. A visszaállításhoz válaszd ki a /rewind-ben a megfelelő sort → Restore file.

5Hibakezelés Védőháló

A checkpoint alrendszer alapvető szerződése a fail-open — egyetlen sikertelen pillanatkép sem blokkolhatja a fő folyamatot. Ez azt jelenti:

  • Nincs git a rendszeren → a teljes alrendszer no-op, nem dob hibát.
  • Git parancs átmeneti hibája (EBUSY, timeout) → ez a kör kimarad, a következő turn-ben automatikusan újrapróbálkozik.
  • A munkakönyvtárban több mint 200 000 fájl van → kihagyja, elkerülve a git add beragadását.
  • Lemez megtelt / jogosultsági hiba → a hiba a logForDebugging naplóba kerül, a beszélgetés folytatódik.

Több munkakönyvtárat érintő határesetek

Ha egy régi session-t /resume-olsz, de a jelenlegi munkakönyvtár eltér attól, ahol a session készült, a /rewind előtt az Axiomate egy rendszerüzenetben figyelmeztet:

  • reachable: a pillanatkép az aktuális ref-ben van, a rewind normálisan működik.
  • reachable-other-worktree: a pillanatkép létezik, de egy másik munkakönyvtárhoz tartozik. A rendszer javasolja, hogy cd-vel válts át oda.
  • unreachable: a pillanatképet a prune már törölte, a rewind sikertelen lesz.

Orphan figyelmeztetés tisztításkor

A /checkpoints status végén megjelenhet egy ilyen sor: 

Note: 3 snapshots from 1 orphan workdir will be discarded on next prune.

Ha ezt látod, és az orphan munkakönyvtár commit-jaira még szükséged van (például mert kihúztad a külső merevlemezt), állítsd vissza a munkakönyvtárat az eredeti elérési útra — a következő prune már live állapotúként fogja kezelni.

6GYIK

Beszennnyezi a saját git repómat?

Nem. Minden commit a ~/.axiomate/checkpoints/store/ könyvtárba íródik, amely teljesen független a projekted saját .git/ könyvtárától. A git status, git log, git push egyáltalán nem látja ezeket a commit-okat.

Mennyi lemezterületet foglal?

A hard limit 5 GB (konfigurálható). A git maga végez objektum- deduplikációt és zlib tömörítést, így ha ugyanazt a fájlt 100-szor módosítod, valójában csak néhány delta kerül tárolásra. A mindennapi használat általában néhány száz MB és 1-2 GB között mozog.

Szinkronizálható gépek között?

Nem. A checkpoints helyi pillanatkép-rendszer, nem tartalmaz gépek közötti szinkronizációs, titkosítási vagy aláírási mechanizmust. Másik gépre váltva olyan, mintha előzmények nélkül kezdenéd.

Exportálható / importálható?

Jelenleg nincs checkpoints export / import parancs. Maga a shadow repo egy szabványos bare git repó, a git-hez értő felhasználók közvetlenül használhatják a git --git-dir=~/.axiomate/checkpoints/store ... parancsot a vizsgálatához.

Hogyan kapcsolható ki teljesen?

A Settings-ben állítsd a fileCheckpointingEnabled értéket false-ra, vagy használd a AXIOMATE_CODE_DISABLE_FILE_CHECKPOINTING=1 környezeti változót. A már felgyűlt pillanatképek nem törlődnek automatikusan, kézzel kell futtatni a /checkpoints clear parancsot.

A /rewind hibát dob, hogy a commit nem található — mit tegyek?

Valószínűleg az adott turn commit-ját a prune már törölte (több mint 60 napos és az aktuális session nem hivatkozott rá). Ez várható viselkedés. Ha kritikus csomópontról van szó, készíts külön biztonsági mentést a munkakönyvtár kimenetéről, ne csak a checkpoint-ra hagyatkozz.
