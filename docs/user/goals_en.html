Goals — Persistent Goals Across Turns

Ordinary conversations are one question, one answer — you say something, the AI responds, and it ends. Goal elevates your objective into a cross-turn "long-task anchor": you set a goal with /goal <description>, the AI starts working, and after each turn a lightweight "judge model" automatically evaluates whether the goal has been achieved — if not, it feeds a continuation prompt to keep the AI going; if done, it stops and notifies you. This continues until completion, cancellation, or budget exhaustion. At any point you can preempt with a message, add criteria via /subgoal, or pause with Ctrl+C — and after /resume, --continue, or /goal resume, the goal state is preserved intact. What it solves: AI forgetting the original objective partway through a long task.

1How It Works

Goal is the classic "Ralph loop" paradigm — breaking a long objective into a series of auto-continued small turns so the AI builds on the results of each turn. Axiomate's /goal writes the goal into the current session's persistent state; after each assistant turn, the judge runs. If not done, it queues the next continuation message. It stops when done, paused, or when the budget is exhausted.

What Happens Each Turn

Round N The AI processes the previous user message normally (which may be your original goal or a continuation prompt from the previous round) — calling tools, editing files, and generating a response.
Turn end Axiomate extracts the AI's final text of the round, bundles it with your goal and all subgoals, and sends them to the judge model using auxiliary.goalJudge route policy, falling back to model.defaultRoute only when that auxiliary policy is absent, asking for: {"done": <bool>, "reason": "..."}.
Verdict: done Goal state flips to done, the screen shows ✓ Goal achieved: <reason>, and the loop ends.
Verdict: continue Turn count +1. If the budget hasn't been exceeded (default 20 turns), a continuation prompt [Continuing toward your standing goal] Goal: <original> … is pushed back into the message queue; the REPL picks it up on the next idle tick as if it were a normal user message, starting the next round.
Budget exhausted When turn count reaches the cap, auto-pause with: ⏸ Goal paused — 20/20 turns used. Use /goal resume to keep going, or /goal clear to stop.

Key Design Decisions

Normal user messages

Continuation prompts go through the exact same message queue as your hand-typed messages. The system prompt is untouched, the tool set is untouched — the prompt cache stays valid.

Judge is a side task

The judge uses the auxiliary.goalJudge task policy: primary model, fallback chain, timeout, failure disposition, and recovery gates. If that policy is absent, it falls back to the main route.

Persisted in JSONL

Goal state is written into the session log as type:'goal-state' entries and automatically restored on /resume. Like file checkpoints, it follows an append-only + last-wins paradigm.

Fail-open

Judge errors (timeout, API failure, parse failure) always return continue — a judge failure will never block the conversation. Only after N consecutive parse failures does it auto-pause (N defaults to 10, configurable via /config).

User priority

If you type a new message mid-loop, it preempts the continuation prompt — that turn processes your message, and the judge runs again after your message is handled (which might just finish the goal itself).

Clean Ctrl+C exit

Interrupting a turn immediately pauses (no judge, no requeue) and records the state as user-interrupted (Ctrl+C). /goal resume restores active; judging resumes after the next normal message.

Here is where Goal appears on screen — these glyphs are the system's "identification marks" and are worth remembering:

GlyphWhenMeaning
⊙ After /goal <text> / /goal status Goal is set / in progress. The status bar Goal line also shows ⊙ Goal 3/20: write fib.
↻ Start of each continue turn / continuation prompt in transcript Loop continues. Screen shows ↻ Continuing toward goal (3/20): <reason>; in the transcript, the auto-fed user message is prefixed with ↻ so you can tell "this wasn't typed by me — it was auto-fed by the goal loop."

Status Bar Goal Line

Whenever a goal is active or paused, a dedicated Goal line appears below the input area: 

⊙ Goal 3/20: write a fibonacci function and verify with fib(10)
⏸ Goal paused: refactor auth — turn budget exhausted (20/20)

This line is for status display only. Confirmations and hints for /goal, /subgoal, and similar commands go into the message stream and no longer occupy the status bar notification area.

3User Commands

FormEffectAliases
/goal or /goal status Show current goal state (active / paused / no goal) + turn progress + subgoal count; if subgoals exist, their contents are listed as well. /goal list, /goal ls
/goal <description> Set a new goal, replacing any previous goal, and immediately use the description as the first user message to start the first round (no need to send again). —
/goal pause Explicitly pause. State is preserved and visible via /goal status; no auto-continuation on the next turn. —
/goal resume Restore from paused to active. The current implementation retains the used turn count and does not reset the budget; the command only restores the state and does not immediately start the next round. Send any normal message (e.g., continue) and the goal loop will resume judging at the end of that turn. —
/goal clear Clear the goal. The status bar Goal line disappears and the loop stops completely. /goal stop, /goal done
Tip: After /goal <description>, you do not need to press Enter again. The command automatically queues the description as the first user message. Command confirmation text goes into the message stream; the status bar Goal line only shows the current active / paused state.

4/subgoal — Adding Criteria Mid-Loop

You set /goal implement a fib function, and after a couple of rounds you notice the AI skipped unit tests. No need to clear and re-create — use /subgoal to add criteria: 

/subgoal there must be a fib.test.ts covering edge cases when done
/subgoal output the actual value of fib(10) to console

The next judging round switches to the with-subgoals template: 

Goal:
<original goal>

Additional criteria the user added mid-loop (all must also be
satisfied for the goal to be DONE):
- 1. there must be a fib.test.ts covering edge cases when done
- 2. output the actual value of fib(10) to console

Agent's most recent response:
...

Decision: For each numbered criterion above, find concrete evidence
in the agent's response that the criterion is satisfied. ...

The judge is forced to find evidence for each subgoal individually — it won't accept a blanket "all done" from the AI.

FormEffect
/subgoalList the current goal state and all subgoals (using 1-based numbering like [1], [2]).
/subgoal list or /subgoal lsExplicit list alias for /subgoal.
/subgoal <text>Append a criterion.
/subgoal remove <n>Delete a criterion by its 1-based index.
/subgoal rm <n>Alias for /subgoal remove <n>.
/subgoal clearClear all subgoals (does not affect the main goal).

Subgoals do not start a new round on their own — they take effect on the next judging pass after the current turn finishes. There is no "immediately notify the AI about the new criterion" semantic, because Axiomate does not interrupt an in-progress turn. So subgoals are good for "patching gaps," not for "emergency stops." For an emergency stop, use /goal pause.

5Judge Model Configuration

The judge model is the cost-sensitive point of the Goal system: it runs every turn. Configure it as an auxiliary task so the main conversation model does not have to judge every round.

PrioritySourceNotes
1 auxiliary.goalJudge.primary First model attempted for goal judging. Use a cheap model that follows JSON instructions reliably.
2 auxiliary.goalJudge.fallbackChain Ordered fallback candidates available when the recovery decision permits switch_model.
3 model.defaultRoute Fallback only when auxiliary.goalJudge is absent; uses the main route chain.

Recommended Config (edit ~/.axiomate.json)

{
  "models": {
    "main": { "...": "..." },
    "judge": { "...": "..." }
  },
  "model": {
    "defaultRoute": "default",
    "routes": {
      "default": { "primary": "main", "fallbackChain": [] }
    }
  },
  "auxiliary": {
    "goalJudge": {
      "primary": "judge",
      "fallbackChain": ["main"],
      "recoveryProfile": "auxiliary-judge",
      "failure": "fail_open",
      "timeoutMs": 30000
    }
  },
  "goalsMaxTurns": 20,
  "goalsParseFailureLimit": 10,
  "goalsContinuationIncludeReason": true
}

goalsMaxTurns and goalsParseFailureLimit can be set via /config with preset values. goalsContinuationIncludeReason can also be toggled via /config; it is on by default.

Goals use the same route/task policy model configuration as the rest of Axiomate. The judge is one of Axiomate's auxiliary tasks. Configure auxiliary.goalJudge once and the recovery runner will use its model chain and policy gates.

When main-route fallback happens: If auxiliary.goalJudge is absent, the judge falls back to model.defaultRoute. Configure auxiliary.goalJudge to keep judging cheap and predictable.
Note on reasoning models: Models that emit <think>...</think> before JSON can be used; Axiomate strips those blocks before parsing. They are usually too slow and token-heavy for this task.

What the Judge Prompt Looks Like

The fixed system prompt sent to the judge model looks roughly like this: 

You are a strict judge evaluating whether an autonomous agent has
achieved a user's stated goal. ...

A goal is DONE only when:
- The response explicitly confirms the goal was completed, OR
- The response clearly shows the final deliverable was produced, OR
- The response explains the goal is unachievable / blocked / needs
  user input (treat this as DONE with reason describing the block).

Otherwise the goal is NOT done — CONTINUE.

Reply ONLY with a single JSON object on one line:
{"done": <true|false>, "reason": "<one-sentence rationale>"}

"Blocked / needs user input" also counts as DONE — this is a safeguard: the judge won't let the loop waste tokens while the AI sits waiting for your answer.

6System Interactions

/resume — Automatic Recovery

Goal state is written into the JSONL session log. When you load a previous session via axiomate --continue, axiomate --resume, or /resume <sessionId>, the goal automatically surfaces — you'll see the status bar Goal line showing ⊙ Goal 3/20: … as soon as you enter the session. Whether the state is active or paused is exactly as you left it.

/rewind — Goal Persists After Rewind

/rewind rolls back the conversation chain or file state — goal state does not participate in the rewind chain (it is pointer-style metadata, not part of buildConversationChain). In other words: rewinding to a conversation position from several turns ago does not touch the goal — the goal follows the sessionId, not the conversation chain.

/branch / --fork-session — Goal Is Not Inherited on Fork

When forking a new session, the old session's goal-state entries are not copied to the new session — the fork filter only carries transcript messages, so the new session starts with no goal. This is intentional: a fork is typically for opening a parallel experiment line, and carrying the old goal would be inappropriate.

Preemption — Your Message vs. Continuation Prompt

What happens if you type and send a new message while the goal loop is running? Two cases:

  • The message is a slash command (e.g., /subgoal …): Does not count as a "real message" — does not preempt the goal. After the slash command is processed, the goal continues.
  • The message is plain text: Preempts. Axiomate's stopHook checks the message queue before judging; if it finds a real, non-slash message, it defers — your message goes first, and after it is processed, judging resumes at the end of the next turn.

Judge Verdicts Do Not Pollute Conversation Context

Prompts like ↻ Continuing toward goal (3/20): reason are marked with isMeta: true — you can see them, but they do not enter the next round's model prompt. This avoids the problem of "judge prose disrupting the prompt cache." However, the continuation prompt itself is a real user message and does enter the conversation history (otherwise the AI wouldn't know to continue).

7Failure Modes

The danger of a Ralph loop is that "the judge model breaks and you never notice, while tokens keep burning." Axiomate has several built-in brakes:

FaultBehaviorRecovery
Judge API timeout / network error Treated as continue, not counted as a parse failure. Loop continues. Automatic recovery, no action needed.
Judge returns non-JSON / empty reply Treated as continue, consecutiveParseFailures +1. Auto-pauses after N cumulative failures (see next row; N defaults to 10). Process is transparent.
Judge returns garbage for N consecutive rounds (N defaults to 10, configurable via /config: 3/5/10/20/50/0) State flips to paused with: ⏸ Goal paused — the judge model (N turns) isn't returning the required JSON verdict. Set auxiliary.goalJudge in ~/.axiomate.json to a stricter model. Then /goal resume to continue. Edit ~/.axiomate.json to use an instruct model, /goal resume to restore, then send a normal message to continue.
Turn budget exhausted (default 20) State flips to paused with: ⏸ Goal paused — 20/20 turns used. /goal resume to restore active, then send any normal message to continue; or /goal clear to wrap up.
User Ctrl+C during a turn Immediately pauses (no judge, no continuation), state recorded as user-interrupted (Ctrl+C). /goal resume to restore, then send a normal message to continue.
AI empty response (no text produced in the final turn) Judge is skipped, not counted against the turn budget (to avoid being drained by API hiccups). Automatic recovery, no action needed.
Judge falls back to the main route because auxiliary.goalJudge is absent The loop still runs, but judging may be more expensive or less predictable. Edit ~/.axiomate.json to add auxiliary.goalJudge.
The real hard failure — goal drift. If you set a vaguely worded goal (e.g., /goal make this project better), the judge will keep saying continue forever while the AI changes things endlessly. The turn budget is a backstop, but 20 Opus calls is not cheap either. Recommendation — write completion criteria into your goal description, e.g., /goal implement a fib function and pass all test cases in fib.test.ts, rather than /goal write a fib function.

8FAQ

Is Goal enabled by default?

Goal does not consume any resources on its own — if you never type /goal, the entire system stays silent. No toggle needed.

Can I use it without setting auxiliary.goalJudge?

Yes. If auxiliary.goalJudge is absent, judging falls back to the main route. It works, but a dedicated judge route is usually cheaper and more predictable.

Can I do other things while a goal is running?

Yes. Axiomate's message queue design is "user first" — you can type and send messages at any time. The next turn processes your message, and goal judging runs after your turn finishes. You can also use read-only slash commands like /checkpoints status, /rewind, and /goal status to check on things while the goal runs.

The 20-turn budget is too small / too large — how do I change it?

In /config, select "Goal turn budget (per /goal)" — presets: 5 / 10 / 20 / 50 / 100 / 200 / 500 / 0. 0 = unlimited budget, relying only on the judge's done verdict, the parse-failure threshold, and manual pause as backstops; the status line and the status bar Goal line will show N/∞. You can also directly edit the goalsMaxTurns field in ~/.axiomate.json with any integer.

Be cautious when raising the number — pair it with auxiliary.goalJudge. Expensive-model x 100 rounds is real money (roughly ~$20+). When setting 500 or 0, consider using a cheaper main model too.

What counts as a turn?

A turn is not the same as a tool call. One turn = the AI completes a full response (which may involve 0-N tool calls in between) + a final stop. The judging point is the stop hook — the same unit of work triggered when you type and send a message. Goal judging runs once at the end of each turn. So 500 turns is not 500 tool calls — it's the AI completing 500 discrete pieces of work. The magnitude is very different.

Can I make the goal run without showing in the transcript?

Not at the moment. Continuation prompts are real user messages — they enter the transcript, consume tokens, and are brought back by /resume. This is intentional: it lets you fully audit what the Ralph loop did. Only the judge verdicts (↻ / ✓ / ⏸ and the like) are isMeta: true and do not enter the transcript chain.

Can I use a custom prompt template for /goal?

Not currently. The continuation prompt and judge prompt use fixed templates; by default, Axiomate also passes the previous round's judge reason into the next continuation prompt (can be disabled via goalsContinuationIncludeReason). If you need domain-specific judging (e.g., "code must pass typecheck to count as done"), the better approach is to write that criterion into the goal description and let the judge model evaluate against your words.

Is this the same as Axiomate's /skills loop?

No. /loop runs a prompt on a schedule (cron-style), without judging; Goal runs event-driven until a semantic condition is met. /loop is good for "babysit a PR every 5 minutes"; Goal is for "do a complete task, then stop."

How is goal persistence implemented? Does it survive an Axiomate process restart?

Yes. Goal state is written as type:'goal-state' entries, append-only, into the session JSONL (~/.axiomate/projects/<hash>/<sessionId>.jsonl), in the same file as your conversation history. Restarting Axiomate or using --continue automatically reads the latest goal-state entry and restores it.

Is there a "global goal list" to view all goals across all projects?

Not at the moment. Goal is per-session with no cross-session index. To check goal state across all projects, you would need to grep for goal-state entries in all JSONL files under ~/.axiomate/projects/ yourself.
