Axiomate user guide

Model Configuration and Routing Strategy

Last updated: 2026-06-02. This guide is for first-time Axiomate users and explains the model-related fields in ~/.axiomate.json, manual configuration methods, the /model command, and the model UI in Settings.

In a nutshell: models defines "which specific model resources are available," model.defaultRoute / model.routes defines which model chain the main conversation uses, and auxiliary defines which model chain auxiliary tasks such as title generation, goal judging, summarization, and hooks use.

A model chain is the set of candidate models allowed for a single main conversation or auxiliary task: the primary model is used first, and only when the unified recovery system determines that "this type of error is suitable for a model switch" does it try subsequent models in fallbackChain order.

Axiomate's API reliability is handled by a unified observe-decide-execute recovery system. Routing configuration only provides candidate models and policy thresholds — it is not a separate, independent retry logic.

File Location

The user-level configuration file is at ~/.axiomate.json. On Windows it is typically C:\Users\<your-username>\.axiomate.json.

On first run or when executing /model add, Axiomate creates or updates this file. You can also edit it manually, but ensure the JSON is valid before saving.

Current configuration contract: Model configuration only uses models, model.routes, model.defaultRoute, and auxiliary. Model switching and routing management use the /model command, not /config settings.

Understand Three Core Concepts First

The most confusing parts of the config file are model resources, model chains, and routes. Just remember: model resources handle "how to call," model chains handle "who else to try on failure," and routes handle "saving a main conversation usage pattern under a name."

Concept What it is How to use When to use Config location
Model Resource How to call this model: protocol, endpoint, API key, actual provider model name, context window, image capabilities. Add with /model add, or manually write into models. When onboarding a new provider or new model. models.<model-id>
Model Chain Which models a single request can try in sequence. Configure primary and fallbackChain inside a route or auxiliary task. When you want automatic fallback to a candidate model on rate limits, timeouts, server errors, or malformed output. primary + fallbackChain
Route A named model chain with an attached recovery policy. Main conversations select different "usage patterns" via routes. Switch with /model route <route-id>, or manually change model.defaultRoute. When you want to switch between modes like "default primary," "quality-first," or "speed-first" for the main conversation. model.routes.<route-id>

model.defaultRoute simply points to one of the routes. For example, "defaultRoute": "deepseek-main" means: the main conversation defaults to the deepseek-main model chain, rather than being hardcoded to a single model.

A model chain is not load balancing, nor does it randomly pick a model each time. Axiomate always starts each request with primary. Only when the observation layer confirms the current error is recoverable and the decision layer selects switch_model does the execution layer move to the next model in fallbackChain.

Main Route: What, How, and When

What it is

A main route is the name for "which model strategy the main conversation uses." Each route contains a first-choice model primary, a set of backup models fallbackChain, and policy fields that specify which errors are allowed to trigger recovery.

A single route is enough to operate. Multiple routes let you preset several common patterns for the same set of models, such as default primary, quality-first, and speed-first. They can reference the same models entries, just with different model ordering.

Route What it is Typical model chain When to use
deepseek-main Default primary route. deepseek-v4-pro → gpt-5.4 → Pro/zai-org/GLM-5.1 → Pro/moonshotai/Kimi-K2.6 → Qwen/Qwen3.5-397B-A17B → Pro/MiniMaxAI/MiniMax-M2.5 → Qwen/Qwen3.5-122B-A10B → Qwen/Qwen3-8B Daily main conversations. Uses the most commonly used primary model first, falling back in quality order only when a recoverable error requires a model switch.
quality-main Quality-first route. gpt-5.4 → deepseek-v4-pro → Pro/zai-org/GLM-5.1 → Pro/moonshotai/Kimi-K2.6 → Qwen/Qwen3.5-397B-A17B → Pro/MiniMaxAI/MiniMax-M2.5 → Qwen/Qwen3.5-122B-A10B → Qwen/Qwen3-8B Complex refactoring, long reasoning, important outputs. Switch to this route when you explicitly want to use the strongest or most expensive model first.
fast-main Speed / cost-first route. Qwen/Qwen3.5-122B-A10B → Pro/MiniMaxAI/MiniMax-M2.5 → Qwen/Qwen3.5-397B-A17B → Pro/moonshotai/Kimi-K2.6 → deepseek-v4-pro → Qwen/Qwen3-8B Simple Q&A, short text processing, quick ad-hoc tasks. Uses a faster large model first, escalating to stronger candidates on failure.

model.defaultRoute determines which main route is used by default. For example, "defaultRoute": "deepseek-main" means the default primary chain is used normally. Switching routes is essentially changing model.defaultRoute to a different route name.

When there are three routes, Axiomate does not run all three concurrently. By default, only the route pointed to by model.defaultRoute is used. The other routes are switchable presets for different task orientations.

How to use

What you want to do Recommended command Where to edit JSON manually
See which route the main conversation is actually using /model show Check the value of model.defaultRoute.
List all available routes /model route list Check which keys exist under model.routes.
View a route's model chain /model route show quality-main Check model.routes.quality-main.primary and fallbackChain.
Switch the default main conversation to the quality-first route /model route quality-main or /model default quality-main Change model.defaultRoute to "quality-main".
Switch the default main conversation back to the default primary route /model route deepseek-main Change model.defaultRoute to "deepseek-main".
Only change the first-choice model of the current default route /model use <model-id> or open the /model picker Change model.routes.<defaultRoute>.primary. This does not switch routes.

The command method writes directly to ~/.axiomate.json and persists across restarts. When editing manually, model.defaultRoute must reference an existing route name in model.routes; the primary and fallbackChain in each route must reference model IDs that already exist in models.

The Model setting in Settings is not a route switcher. The Settings or /model picker is primarily for changing the primary of the current default route. To switch between deepseek-main, quality-main, and fast-main, use /model route <route-id> or manually edit model.defaultRoute.

This type of selection is not a temporary override. Settings → Config Model, opening the /model picker directly, and /model use <model-id> all persist by writing to the route pointed to by the current model.defaultRoute, changing that route's primary to the selected model. The new primary will still be used on the next launch.

When you should use multiple routes

Situation Recommendation Reason
Only one model connected, or rarely switch task types One route is enough. Simpler configuration; the main conversation always uses the same model chain.
Daily tasks and complex tasks differ significantly Keep two routes: default-main / quality-main. Use the default route normally, and /model route quality-main to switch when you need higher quality.
Frequently run simple, low-cost, low-latency tasks Add a fast-main route. Let a small model be the first choice, only trying stronger models on recoverable errors.
Just want to change the first-choice model for the current default route Use the /model picker or /model use <model-id>. This changes the current route's primary; no need to create or switch routes.
A particular provider is frequently rate-limited, timing out, or returning server errors Add models from other providers to the current route's fallbackChain. Fallback is a reliability mechanism; it is only used when the recovery system determines a model switch is appropriate.

Model Resource: What, How, and When

What it is

A model resource is an entry in models that describes "how to call a specific model from a specific provider." It does not decide which model the main conversation uses by default, nor does it decide fallback order; those are determined by routes and auxiliary tasks.

For example, models["main-qwen"] is the model ID referenced internally by Axiomate, and routes also use this ID; whereas models["main-qwen"].model is the actual model name sent to the provider API.

How to use

What you want to do Recommended method What happens in the configuration
Onboard a new model /model add Adds a new models.<model-id>, and optionally includes it in the main route.
Change API key, endpoint, protocol, or context window /model edit <model-id> or manually edit models.<model-id> Only modifies the model resource itself; does not automatically change the default route.
Make the main conversation start using this model /model use <model-id> Changes the current default route's primary to this model.
Add this model as a fallback model /model fallback add <model-id> Adds it to the current default route's fallbackChain.

When you need to modify model resources

Situation What to modify
API key expired, provider endpoint changed, model name misspelled Edit models.<model-id>.apiKey, baseUrl, or model.
Model does not support images but is used for image input Set supportsImages to false, and point main routes that need image input to models that support images.

Field Reference

Field Required Meaning
model Yes The model name sent to the provider API. It may be the same as the models key or different.
name No The name displayed in the model picker. If omitted, the models key is shown.
description No Description shown in the model picker. If omitted, the protocol and endpoint are displayed.
protocol Yes openai-chat, openai-responses, or anthropic. Determines the wire format for requests and streaming responses.
vendor No Vendor template name, used to translate the neutral thinking configuration into provider-specific fields and handle gateway-level compatibility differences. Can usually be omitted; Axiomate infers it from the protocol and baseUrl. Set explicitly if the model is exposed through a special relay gateway.
modelTemplate No Explicit model-layer template name, used to overlay model / relay-specific behaviors. The onboarding wizard recommends a value based on model name, vendor, and baseUrl, but at runtime a model template is only applied if this field is configured; leaving it unset means no model-layer overrides are applied. baseUrl Yes API endpoint, e.g. https://api.openai.com/v1 or the compatible address for OpenRouter / SiliconFlow / vLLM / ollama.
apiKey Yes API key. Even for local ollama which does not validate keys, a placeholder string is required.
contextWindow No Context window token count. Used for truncation, compression, and budget estimation.
maxOutputTokens No Default maximum output tokens per single response for this model resource. Auxiliary tasks can further narrow the request budget via auxiliary.<task>.maxOutputTokens.
supportsImages No Whether the model supports image input. Defaults to true; set to false for text-only models.
thinking No Neutral reasoning configuration, e.g. { "enabled": true, "effort": "high" } or { "enabled": true, "budget": 8192 }.
extraParams No Provider-private parameters merged as-is into the request body on every call.
usageMapping No Cache token field path mapping for OpenAI Chat-compatible responses. Usually not needed for Responses and Anthropic.
userAgent No Override the HTTP User-Agent. Some Responses-compatible gateways require a specific UA to pass through.

stallTimeoutMs No Override the streaming response stall warning threshold. Set to 0 to disable stall warnings for this model.

Axiomate's template system merges in three layers: protocol → vendor → model. Later layers override earlier ones, so when both a vendor template and a model template declare the same field, the model template takes effect.

Auxiliary Tasks: What, How, and When

What it is

auxiliary provides model strategies for "small tasks that need an LLM but are not the main conversation." Title generation, input suggestions, search summaries, goal judging, hooks, and token counting all belong to this layer. They should not consume the most expensive main model by default, nor should they treat API error text as normal results on failure.

How to use

What you want to do Recommended command Where to edit JSON manually
View all auxiliary task policies /model aux list Check which tasks exist under auxiliary.
View the model chain for a specific auxiliary task /model aux show goalJudge Check auxiliary.goalJudge.primary and fallbackChain.
Make title generation use a cheap, fast model /model aux set sessionTitle <model-id> Edit auxiliary.sessionTitle.primary.
Make input suggestions use a cheap, fast model /model aux set promptSuggestion <model-id> Edit auxiliary.promptSuggestion.primary.
Add a fallback model for goal judging /model aux fallback add goalJudge <model-id> Add the model ID to auxiliary.goalJudge.fallbackChain.
Set how an auxiliary task handles failure /model aux policy goalJudge failure fail_open Edit auxiliary.goalJudge.failure.
Set the maximum output tokens for input suggestions /model aux policy promptSuggestion maxOutputTokens 96 Edit auxiliary.promptSuggestion.maxOutputTokens.

When you need to modify auxiliary tasks

Situation Recommendation Reason
Title generation, summarization, or date parsing is too slow or too expensive Change the corresponding task's primary to a cheap, fast model. These tasks do not need a model of the same tier as the main conversation.
Next-suggestion in the input box is too slow, too expensive, or repeatedly shows empty output in /doctor Change auxiliary.promptSuggestion.primary to a cheap, fast model and keep the failure policy as return_null. This is a background hint; it should not slow down the main response or retry endlessly due to empty results.
/goal judgment is too lax, too strict, or has unstable formatting Adjust auxiliary.goalJudge and choose a model that follows JSON more reliably. Goal judging requires stable structured output.
Auxiliary task failure should not interrupt the main flow Set failure to return_null, return_original, or fail_open. Title, summary, and search helpers can usually degrade gracefully on failure.
Hook agent or critical validation failures must surface errors Set failure to propagate_error or fail_closed. Failure in these tasks may affect security or correctness and must not be silently swallowed.

Recovery Semantics: Observe, Decide, Execute

Observe

Every API attempt records the protocol, provider, model, route, auxiliary task, attempt index, initial or retry error, and classifies raw 400s, 52xs, timeouts, empty output, etc. into semantic reasons.

Decide

The decision layer reads the recovery rule table, the previous error, the current error, route policy, and remaining candidate models, then decides to retry_same_model, adapt_request, switch_model, or stop.

Execute

The execution layer modifies the request or switches to the next candidate as decided. Failure results are written back to the same recovery session for the next round of decision-making.

For example, if the first attempt hits a 400 and the second attempt hits a 52x after adjusting the request, this is not a state "overwriting the old error," but rather two attempt observations within the same recovery session. The decision layer can see "previously it was a request shape issue, now it is a server/gateway issue," and then decide whether to switch models or stop.

API Automatic Recovery Overview

What it is: Axiomate first classifies API failures into semantic reasons, then selects actions from the unified recovery rule table. The current rule table explicitly covers 13 common recoverable error categories, and the recovery action set contains 21 action types. This number will grow with releases; users need to understand the classifications and budgets, not memorize every internal rule.

How to use: Regular users only need to configure model chains and policy thresholds. Main conversations default to a maximum of 10 recovery retries, i.e. up to 11 attempts; this can be adjusted with the environment variable AXIOMATE_CODE_MAX_RETRIES. Auxiliary tasks are allocated budgets by semantics: background direct-connect auxiliary requests have no local recovery retries, verification and fast auxiliary tasks typically have 1 recovery retry, and foreground interactive or quality-first auxiliary tasks typically have 2 recovery retries. Provider Retry-After headers are respected, but individual wait times are capped to avoid stalling for too long.

When it happens: Request failures, stream interruptions, Responses empty output, unsupported model fields, oversized images, excessive context, server 52x, rate limits, timeouts, disconnected connections, and unavailable models all enter the observation layer. The decision layer may choose to wait and retry the same model, retry after modifying the request, compress the request, downgrade to non-streaming, refresh the connection, switch models, or stop.

Failure symptom Common observed reason Possible actions
Provider reports unsupported temperature, stop, tool_choice, or similar fields unsupported_parameter Resend after omitting the offending fields.
OpenAI Responses rejects historical encrypted reasoning content invalid_encrypted_content Resend after removing unverifiable reasoning replay.
Image dimensions or size exceed provider limits image_too_large Rewrite image payload according to the image recovery profile, then resend.
Context or payload too large context_overflow / payload_too_large Compress request, or first reduce output budget and disable thinking.
Rate limit, overload, timeout, disconnected connection, server 52x rate_limit / overloaded / timeout / connection / server_error Wait then retry; switch to fallbackChain when the budget is exhausted and the policy allows it.
Model not found, content policy blocked, response structure empty or malformed model_not_found / content_policy_blocked / responses_null_output / malformed_response Retry, adapt the request, or switch models according to the rules; content policy refusals do not retry the same request as-is, and stop with a trace when unrecoverable.
Provider or aggregator account policy blocked provider_policy_blocked This is a deterministic block at the account, privacy, or data-policy level. Axiomate stops and prompts you to check provider/account policy, base URL, model, or route; it will not treat this as an automatic model-switch reason.
Provider explicitly rejects streaming, or streaming endpoint returns 404 but not model-not-found streaming_unsupported / stream_endpoint_not_found Only in these narrow semantics does it fall back to non-streaming; empty streams, local stream parse errors, and ordinary 52x will first go through streaming retry and will not directly downgrade.

allowActions determines which action types a route or auxiliary task permits, and switchModelOn only determines which semantic errors allow a model switch. They are merely inputs to the decision system and do not bypass the observation and rule table to fall back directly. provider_policy_blocked is a diagnostic and configuration-fix reason, not an automatic model-switch reason writable into switchModelOn.

content_policy_blocked means the provider issued a safety / content-filtering refusal for the current prompt. Axiomate does not treat it as a transient retry and will not resend the same request as-is; it only switches to a candidate model when allowActions includes switch_model, switchModelOn includes content_policy_blocked, and fallbackChain has a different model available.

Each failure round re-observes the error reason and generates an independent fallback-eligibility snapshot from the current primary / fallbackChain / policy. That is, a previous rate_limit with switch_model allowed does not pollute the judgment of a subsequent model_not_found or server_error. The actionAllowed / reasonAllowed seen in /doctor are diagnostic snapshots for that request round, not fields you need to manually write into configuration.

/doctor API Providers: How to Read API Failures

What it is: The API Providers section in /doctor projects the most recent API recovery traces from the current session into readable cards. It is not a configuration entry point and does not actively repair requests; it is for answering "which provider / model / route had what problem just now, what did Axiomate observe, what did the decision system do, and where should I look next."

How to use: Run /doctor when the main conversation, auxiliary tasks, model validation, or streaming output shows failures, noticeable slowness, model switches, non-streaming downgrades, empty output, 52x, rate limits, or model-not-found. If the current session has no API recovery records, this section does not appear. The current UI displays up to 5 most recent cards; earlier cards are indicated by a hidden count hint.

How to read background task cards: When background auxiliary tasks like prompt_suggestion, title generation, or permission explanation fail, the cards are typically [Info] or lightweight [Warning]. For example, scope: auxiliary:promptSuggestion indicates the input box dimmed suggestion failed or was cancelled, not that the main conversation reply failed; you should check auxiliary.promptSuggestion, not the current main route.

How to read capability probe cards: scope: capability:count_tokens indicates the provider native token counting probe failed. It is not a main conversation request failure; Axiomate will continue trying auxiliary.tokenCounting or local estimation. Only scope: auxiliary:tokenCounting means the auxiliary model chain used for token counting itself failed.

Record scope: These records are only kept in the current process memory, up to a maximum of 200 recent safe trace events. /clear, switching, or restoring other sessions clears them. Traces are not written to disk and do not store prompts, image content, tool results, API keys, Authorization headers, or raw provider response bodies; they only retain protocol, model, route/task, error classification, HTTP status, a small set of safe response headers, request ID, timestamp, and recovery actions.

Card field Meaning How to interpret
[Error] / [Warning] / [Info] and title This card's severity level and final state summary. Error typically requires configuration changes, model switches, or account/network handling; Warning means a recovery action occurred; Info typically means the request was adapted or partially recovered.
scope Where the failure occurred: main route, a specific auxiliary task, model validation, or other API operations. route:deepseek-main means check model.routes.deepseek-main; auxiliary:goalJudge or auxiliary:promptSuggestion means check the corresponding auxiliary task.
impact The user flow affected by this failure, e.g. main response stream, auxiliary reasoning, token counting, model validation, or non-streaming downgrade. Main response stream failure affects the ongoing chat; auxiliary task failure may only affect title, summary, goal judgment, or hooks.
model The model path traversed during this recovery round. a -> b means the decision system already switched from a to b; a single model means no actual model switch occurred, likely just a retry or request modification.
observed The semantic error reason classified by the observation layer, possibly with HTTP status and provider request ID. Look here first to determine if it is an account, billing, rate limit, timeout, model name, protocol compatibility, context overflow, or provider server error.
recovery Short summary of the decision and execution result. This explains whether recovery succeeded, a model was switched, the request was adapted, it was blocked by policy, the budget was exhausted, or it was delegated to compression, etc.
stopped If recovery stopped, this explains the stop reason. Common reasons include non-retryable error, retry budget exhausted, no available fallback model, route policy disallowing model switch, or request cancelled.
next User-facing next-step suggestion. It points to specific configuration fields, e.g. models["..."].apiKey, models["..."].baseUrl, model.routes["..."].fallbackChain, or switchModelOn.
timeline The attempt sequence within the same recovery session. Format is #attempt/max reason HTTP xxx -> action/outcome +mutation. The UI defaults to showing the last 3 events; internally sorted by sequence, so multiple observations within the same millisecond maintain order.
request / timing / advanced / policy Optional diagnostic fields: request ID, timeout/duration, protocol, operation, route/task, matched rules, safe response headers, and policy snapshot. actionAllowed / reasonAllowed in policy are this round's diagnostic results, not configuration fields. Configuration is still written in allowActions and switchModelOn.

Recovery States in /doctor

State Meaning What the user should focus on
recovered The request succeeded after retry or adaptation. Generally no action needed; if it occurs frequently, check provider stability and route ordering.
switched_model The decision system executed fallback_model and actually switched to a fallback model. Confirm that the fallback model's quality, cost, context window, and tool/image capabilities meet this route's expectations.
switched_request_mode The streaming request failed and was switched to a non-streaming request. This is a request mode downgrade, not a model switch. If frequent, prioritize checking the provider's streaming compatibility.
adapted_request Axiomate modified the request shape, e.g. removed unsupported fields, reduced output budget, disabled thinking, or rewrote image payload. Usually indicates the provider or model does not fully support the current request parameters; check vendor, template, extraParams, and model capability configuration.
adaptation_failed The request was already adapted but subsequent attempts still failed. Check whether the first error and last error in timeline differ; may need to change protocol, switch provider, or adjust route policy.
delegated_recovery The recovery action was handed off to an outer mechanism, e.g. session compaction or the main route loop continuing to process. This is not a "lost error." It means this layer recorded the observation and decision and handed execution to a higher-level recovery loop.
blocked_by_policy The error could have considered a model switch, but the current route or auxiliary policy does not allow it. Check whether allowActions includes switch_model, whether switchModelOn includes this error type, and whether fallbackChain has available candidates.
salvaged The streaming output was interrupted, but partial output was safely preserved. Check whether the final reply is complete; if this happens often, prioritize checking the network, provider streaming, and timeout policy.
exhausted A recoverable error exhausted the current budget. Check provider status, increase budget, add fallback models, or reduce context and payload.
aborted The request was cancelled by the user or system. If intentionally cancelled, no action needed; if not, check for timeouts, exits, or session switches.
failed The request failed and no executable recovery action remains. Follow observed and next to check keys, billing, model name, baseUrl, protocol, and account permissions.
retrying The current trace shows the request is still in the recovery process. Wait for the current request to finish before checking the final card; if it stays for a long time, check timeouts and network.

Order for reading cards: First check scope to locate the configuration block, then check observed to confirm the error category, then check timeline to understand the recovery process, and finally follow next to modify models, model.routes, or auxiliary. Do not directly copy-paste actionAllowed / reasonAllowed from policy; they are only result snapshots of that particular decision.

Model UI in Settings

The Model item in Settings → Config uses the same model picker. It is suitable for quickly switching the current main route's primary and adjusting the effort selection in the picker.

The selection here writes to ~/.axiomate.json: Axiomate reads the current model.defaultRoute, then changes model.routes[defaultRoute].primary to the model you selected. It shares the same persistent semantics as the /model picker and /model use <model-id> — it is not a temporary override that only applies to the current session.

The Model setting in Settings is not a full route editor. It does not create routes, edit fallback chains, or set auxiliary policies. For full routing and auxiliary task management, use /model route ..., /model fallback ..., /model aux ..., or manually edit ~/.axiomate.json.

The /model add onboarding wizard will ask about usage when adding a second or subsequent model: set as main route primary, add to main route fallback, or only write into models without joining routing.

Manual Configuration Process

  1. Open ~/.axiomate.json.
  2. Add a new key under models, e.g. "custom-main". This key is the Axiomate internal reference name used by routes and auxiliary.
  3. Fill in model, protocol, baseUrl, apiKey, then supplement contextWindow, maxOutputTokens, supportsImages, thinking, extraParams, and other fields according to the model's actual capabilities.
  4. Write this key into model.routes.<route-id>.primary, fallbackChain, or auxiliary.<task>.primary.
  5. After saving, reopen Axiomate, or verify with /model show, /model route list, /model aux list.

Session-Level Override

Some entry points temporarily specify a main model, such as launch arguments or model selection during the current session. This type of override follows the internal MainModelOverride semantics: it can point to the default route, a named route, or a single-model route.

The goal of a session override is "what model to use for this session," not "rewrite the global policy." It does not implicitly rewrite model.defaultRoute, route primary, or auxiliary policy. The regular Settings → Config Model, /model picker, and /model use, by contrast, are persistent modifications to the current default route primary.

Common Checks

Symptom Check item
Model not found error on startup Confirm that model.routes.*.primary, fallbackChain, and auxiliary.*.primary all reference keys that exist in models.
Image request fails Set supportsImages to false for text-only models; main routes that need to handle image input should point to models that support images.
Goal judging cost is too high Point auxiliary.goalJudge.primary to a cheap, stable, strict-JSON model.
Fallback did not happen Confirm the route/aux policy allows switch_model, the error reason is included in switchModelOn, and fallbackChain has candidates different from the current model. Some errors will prefer adapt_request, request compression, non-streaming fallback, or stopping first.
Cannot find route management in Settings Settings currently only provides quick Model selection; route and auxiliary management are in the /model commands.

We recommend using /model add for initial setup, confirming connectivity, and then fine-tuning routes, fallback, and auxiliary policy manually.
