MCP Server Configuration and Usage

Last updated: 2026-06-02. This guide explains how to connect external MCP servers in Axiomate, manage configurations, handle authentication, and troubleshoot common issues.

In a nutshell: MCP servers are tool providers external to Axiomate. Once you add a stdio / HTTP / SSE server to your configuration, Axiomate connects to it on startup, reads its exposed tools, prompts, and resources, and adds the tools to the available tool pool in the form mcp__server__tool.

The most common entry point is axiomate mcp add to add a configuration, then use /mcp inside a conversation to view status, authentication, enable/disable, and reconnect.

Quick Start

Axiomate supports three MCP transports for regular users: stdio, http, and sse. If --transport is not provided, it defaults to stdio.

Add a Local stdio Server

axiomate mcp add my-server -- npx my-mcp-server

A stdio server is launched by Axiomate as a child process and communicates with Axiomate via standard input/output. The -- after the command tells the CLI that everything that follows is the server command and arguments, not Axiomate's own flags.

Add an HTTP Server

axiomate mcp add --transport http sentry https://mcp.sentry.dev/mcp

Add an SSE Server

axiomate mcp add --transport sse search https://example.com/sse

With Environment Variables or Headers

axiomate mcp add -e API_KEY=xxx my-server -- node server.js
axiomate mcp add --transport http api https://example.com/mcp -H "Authorization: Bearer xxx"

Note: If you pass a URL directly to axiomate mcp add name https://... without specifying --transport http or --transport sse, Axiomate will treat it as a stdio server. This is usually not what you want.

Configuration Format

The core structure of MCP configuration is a mcpServers object. Each key is a server name, and each value is the connection configuration.

stdio

{
  "mcpServers": {
    "filesystem": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "."],
      "env": {
        "API_KEY": "${MCP_API_KEY}"
      }
    }
  }
}

HTTP

{
  "mcpServers": {
    "remote-api": {
      "type": "http",
      "url": "https://example.com/mcp",
      "headers": {
        "Authorization": "Bearer ${MCP_TOKEN}"
      }
    }
  }
}

SSE

{
  "mcpServers": {
    "events": {
      "type": "sse",
      "url": "https://example.com/sse",
      "headers": {
        "X-Api-Key": "${MCP_API_KEY:-dev-key}"
      }
    }
  }
}
Field Applicable Transport Description
type All stdio can be omitted; use http or sse for remote servers. The JSON layer also supports ws, but the regular mcp add command does not currently expose a WebSocket addition option.
command / args stdio The local command and arguments Axiomate will launch.
env stdio Environment variables passed to the child process. Values support ${VAR} and ${VAR:-default} expansion.
url http / sse / ws Remote MCP endpoint.
headers http / sse / ws Static request headers. Values support environment variable expansion.
headersHelper http / sse / ws Runs a local helper to dynamically generate headers, suitable for short-lived tokens or internal company credentials.
oauth http / sse Configures MCP OAuth related parameters, such as clientId, callbackPort, and authServerMetadataUrl.

Using npx on Windows: If the project configuration directly uses "command": "npx", Axiomate will emit a warning. A safer approach is to wrap it with cmd /c:

{
  "type": "stdio",
  "command": "cmd",
  "args": ["/c", "npx", "-y", "my-mcp-server"]
}

Configuration Scopes

axiomate mcp add defaults to the local scope. You can use -s to explicitly specify where to write.

Scope Command Example Purpose
local axiomate mcp add -s local ... Only you within the current project. Suitable for private tokens, local paths, and temporary servers.
user axiomate mcp add -s user ... Writes to the user's global configuration, available across all projects.
project axiomate mcp add -s project ... Writes to the current directory's .mcp.json, suitable for team sharing, but requires approval before connections are made.

If a server with the same name appears in multiple scopes, later configurations override earlier ones during aggregation. The typical priority order is: plugin < user < approved project config < local config. When an enterprise-managed MCP configuration exists, it takes exclusive control and does not merge with regular user configurations.

Runtime and Management

After adding a configuration, restart Axiomate or start a new session, and Axiomate will automatically load the MCP configuration and connect to available servers. Once connected, the tools exposed by the server will be added to the model's available tool pool.

In-Conversation Management

Command Function
/mcp Opens the MCP management interface to view server status, tool count, prompts, resources, and authentication status.
/mcp reconnect <server-name> Manually reconnect a specified server.
/mcp disable <server-name> Disable a specified server. The disabled state is written to the current project configuration.
/mcp enable <server-name> Re-enable a specified server and attempt to connect.
/mcp disable / /mcp enable When no server name is provided, applies to all currently displayed MCP servers.

CLI Management

Command Function
axiomate mcp list List configured servers and perform connection health checks.
axiomate mcp get <name> View a server's configuration scope, type, URL, or command.
axiomate mcp remove <name> Remove server configuration; remote OAuth tokens will also be cleaned up.
axiomate mcp reset-project-choices Clear the current project's approval/rejection records for .mcp.json servers.

Health checks will actually connect to servers. axiomate mcp list and axiomate mcp get will skip the workspace trust dialog and may start stdio servers from .mcp.json in order to check status. Only run these commands in directories you trust.

Authentication and Headers

Static Headers

HTTP, SSE, and WebSocket configurations can directly specify headers. It is recommended to use environment variables for sensitive values.

{
  "mcpServers": {
    "internal": {
      "type": "http",
      "url": "https://internal.example.com/mcp",
      "headers": {
        "Authorization": "Bearer ${INTERNAL_MCP_TOKEN}"
      }
    }
  }
}

Dynamic headersHelper

headersHelper runs a local command at connection time. The command must output a JSON object to stdout, and all values must be strings.

{
  "mcpServers": {
    "internal": {
      "type": "http",
      "url": "https://internal.example.com/mcp",
      "headersHelper": "node scripts/mcp-headers.js"
    }
  }
}
{
  "Authorization": "Bearer eyJ...",
  "X-Workspace": "payments"
}

The helper receives the AXIOMATE_CODE_MCP_SERVER_NAME and AXIOMATE_CODE_MCP_SERVER_URL environment variables at runtime, making it easy for one helper to serve multiple servers.

OAuth

If an HTTP/SSE server returns a response requiring OAuth, Axiomate will mark the server as needs-auth and provide an authentication option in /mcp. You can also specify OAuth parameters when adding a server:

axiomate mcp add --transport http work https://example.com/mcp \
  --client-id my-client-id \
  --client-secret \
  --callback-port 1455

--client-secret will prompt you for the client secret; in non-TTY environments, you can provide it via the MCP_CLIENT_SECRET environment variable.

Project Sharing with .mcp.json

Teams can write the MCP servers needed by the project into .mcp.json at the repository root. Axiomate searches upward from the current directory for .mcp.json, and configurations closer to the current working directory take higher priority.

{
  "mcpServers": {
    "repo-tools": {
      "type": "stdio",
      "command": "cmd",
      "args": ["/c", "npx", "-y", "@example/repo-mcp"]
    }
  }
}

Project shared configurations are not silently connected by default. The first time a project scope server is seen, Axiomate will ask you to approve or reject it. Approval records are saved in your local project configuration and do not need to be committed to the repository.

Do not commit secrets to .mcp.json. Team shared configurations should reference environment variables, such as ${MCP_TOKEN}, so each user can set their own values locally.

Advanced Options

add-json

Directly write a server JSON to the configuration, suitable for fields not covered by the current CLI add, such as headersHelper or ws.

--mcp-config

Load MCP servers temporarily from a JSON string or JSON file when starting Axiomate, with a scope of dynamic.

Plugin MCP

Plugins can declare mcpServers or a .mcp.json in the plugin directory. Axiomate will prefix plugin servers with plugin:<plugin>:<server>.

add-json Example

axiomate mcp add-json internal '{"type":"http","url":"https://example.com/mcp","headersHelper":"node scripts/headers.js"}'

--mcp-config Example

axiomate --mcp-config ./mcp.local.json
axiomate --mcp-config '{"mcpServers":{"api":{"type":"http","url":"https://example.com/mcp"}}}'

--strict-mcp-config will only use servers passed via --mcp-config, ignoring user, project, local, and plugin configurations. --bare will also skip auto-discovered MCP configurations, keeping only explicitly passed --mcp-config.

Security Model

MCP servers may execute local code, access the network, or read system resources. Axiomate applies several layers of constraints to MCP:

  • Project .mcp.json servers require user approval before connecting.
  • All MCP tool calls still go through Axiomate's tool permission and approval process.
  • The current project can disable a server; once disabled, it will not connect or expose tools.
  • Enterprise policies can restrict server names, commands, or URLs via allowedMcpServers / deniedMcpServers.
  • When an enterprise-managed MCP configuration exists, it can exclusively control all MCP servers.

Practical advice: only add servers you trust; do not run stdio commands from unknown sources; do not commit tokens to the repository; prefer environment variables and least-privilege tokens for team-shared configurations.

Troubleshooting

Symptom Checkpoint
/mcp shows no server First run axiomate mcp list; confirm the configuration was written to the correct scope; if it's .mcp.json, confirm the server has been approved.
URL server consistently fails Confirm that --transport http or --transport sse was specified when adding; check proxies, certificates, headers, and the server endpoint.
Shows needs-auth Go to /mcp and select the server to complete authentication; if the token has expired, re-authenticate or remove and re-add it.
Environment variable missing warning The configuration uses ${VAR} but the current shell has not set this variable. Set the variable and restart Axiomate, or use ${VAR:-default}.
npx server fails to start on Windows Change command to cmd and args to ["/c", "npx", ...].
Configuration changed but tools unchanged Reconnect in /mcp, or restart Axiomate. For plugin MCP changes, try reloading the plugin.

Debugging tip: Start with axiomate --debug and then open /mcp; connection errors for failed servers will be easier to identify.
