Goals — Persistent Goals Across Turns

Ordinary conversations are one question, one answer — you say something, the AI replies, done. Goal elevates your objective into a cross-turn "long-task anchor": you run /goal <description> to set a goal, the AI starts working, and after each turn a lightweight "judge model" automatically determines whether the goal has been achieved — if not, it feeds a continuation prompt to keep it going; if achieved, it stops and notifies you. It runs until completion / you stop it / the budget is exhausted. In the middle, you can send messages to preempt at any time, use /subgoal to add additional criteria, press Ctrl+C to pause — after /resume, --continue, or /goal resume, the goal state is fully preserved. It solves the problem of the AI forgetting the original goal partway through a long task.

1How It Works

Goal follows the classic "Ralph loop" paradigm — breaking a long goal into a set of automatically continued small turns, letting the AI build on the results of each turn. Axiomate's /goal writes the goal into the current session's persistent state; after each assistant turn, the judge evaluates it, and if not done, automatically queues the next continuation message. It stops when completed, paused, or the turn budget is exhausted.

What Happens Each Turn

Turn N The AI processes the previous user message normally (which may be the goal you set or the continuation prompt from the previous round), invoking tools, editing files, and generating a response.
End of Turn Axiomate extracts the last paragraph of the AI's response for this turn, combines it with your goal and all subgoals, and sends them to the judge model via the auxiliary.goalJudge route policy. Only when that policy is absent does it fall back to model.defaultRoute, asking: {"done": <bool>, "reason": "..."}.
Verdict: done The goal state flips to done, the screen shows ✓ Goal achieved: <reason>, and the loop ends.
Verdict: continue Turn count +1. If the budget hasn't been exceeded (default 20 turns), a continuation prompt [Continuing toward your standing goal] Goal: <original text> ... is pushed into the message queue; the REPL picks it up on the next idle tick as an ordinary user message and starts the next turn.
Budget Exhausted When turn count reaches the limit, auto-pause triggers with the message ⏸ Goal paused — 20/20 turns used. Use /goal resume to keep going, or /goal clear to stop.

Key Design Decisions

Ordinary User Messages

The continuation prompt goes through the exact same message queue as a manually typed message. It doesn't touch the system prompt or tool set — the prompt cache stays intact.

Judge Model as Auxiliary Task

Judging uses the auxiliary.goalJudge task policy: primary, fallback chain, timeout, failure disposition, and recovery gate. Only falls back to the main route when this policy is absent.

Persisted in JSONL

Goal state is written as type:'goal-state' entries in the session log. After /resume, it automatically recovers. Like file checkpoints, it uses an append-only + last-wins paradigm.

Fail-Open

Judge model errors (timeout, API failure, parse failure) always return continue — the loop won't get stuck because the judge fails. Only after N consecutive parse failures does it auto-pause (N defaults to 10, configurable via /config).

User First

When you type a new message mid-loop, it preempts the continuation prompt — this turn processes your message, and judging runs after your message is handled (and your message might just happen to complete the goal).

Clean Ctrl+C Exit

Interrupting a turn immediately pauses (no judging, no requeue), and the state is recorded as user-interrupted (Ctrl+C). /goal resume restores active; judging resumes after the next ordinary message is processed.

2UI Indicators

Here is where Goal appears on screen — these glyphs are the "identification codes" of this system, well worth remembering:

SymbolWhenMeaning
⊙ After /goal <text> / /goal status Goal is set / in progress. The status bar Goal line also shows ⊙ Goal 3/20: write fib.
↻ Start of each continue turn / continuation prompt in transcript The loop continues. Screen shows ↻ Continuing toward goal (3/20): <reason>; the auto-fed user message in the transcript is prefixed with ↻ so you can tell "this wasn't typed by me — it was auto-fed by the goal loop."
✓ Judge verdict done / you run /goal clear Goal achieved or manually concluded.
⏸ Budget exhausted / judge failed N rounds (default 10) / Ctrl+C / you run /goal pause Paused but state preserved. /goal resume restores active; judging resumes after the next ordinary message is processed. The status bar Goal line turns yellow.

Status Bar Goal Line

As long as a goal is active or paused, a dedicated Goal line is always visible below the input box: 

⊙ Goal 3/20: write a fibonacci function and verify with fib(10)
⏸ Goal paused: refactor auth — turn budget exhausted (20/20)

This line only displays the current goal state. Confirmation and prompts for commands like /goal and /subgoal go into the message stream and no longer occupy the status bar notification area.

After sending any ordinary message (e.g., continue), the goal loop will resume judging at the end of that turn. —

Tip: After /goal <description>, you do not need to press Enter again. The command itself automatically queues the description as the first user message. Command confirmation text goes into the message stream; the status bar Goal line only shows the current active / paused state.

4/subgoal — Adding Criteria Mid-Loop

You set /goal implement a fib function and after two turns realize the AI forgot unit tests. No need to clear and reset — use /subgoal to add criteria: 

/subgoal there must be a fib.test.ts covering edge cases after completion
/subgoal output the actual value of fib(10) to the console

The next turn's judging switches to the with-subgoals template: 

Goal:
<original goal>

Additional criteria the user added mid-loop (all must also be
satisfied for the goal to be DONE):
- 1. there must be a fib.test.ts covering edge cases after completion
- 2. output the actual value of fib(10) to the console

Agent's most recent response:
...

Decision: For each numbered criterion above, find concrete evidence
in the agent's response that the criterion is satisfied. ...

The judge is forced to find evidence for each subgoal individually — it won't let the AI get away with just shouting "all done."

FormEffect
/subgoalList the current goal state and all subgoals (with 1-based numbering like [1], [2]).
/subgoal list or /subgoal lsExplicit list aliases for /subgoal.
/subgoal <text>Append one criterion.
/subgoal remove <n>Remove one criterion by 1-based index.
/subgoal rm <n>Alias for /subgoal remove <n>.
/subgoal clearClear all subgoals (does not affect the main goal).

A subgoal does not trigger a turn on its own — it takes effect at the next judging after the current turn finishes. There is no "immediately notify the AI of the new criterion" semantics, because Axiomate does not interrupt an in-progress turn. So subgoals are suited for "plugging gaps," not "emergency stops." For emergency stops, use /goal pause.

5Judge Model Configuration

The judge model is the cost-sensitive part of the Goal system: it runs once per turn. Please configure it as the auxiliary.goalJudge auxiliary task to avoid using your main conversation model for every judging round.

PrioritySourceDescription
1 auxiliary.goalJudge.primary The model the judge tries first. Recommend a cheap, stable model that strictly follows JSON instructions.
2 auxiliary.goalJudge.fallbackChain Ordered fallback candidates, usable when the recovery decision allows switch_model.
3 model.defaultRoute Fallback only when auxiliary.goalJudge is absent; uses the main route chain.

Recommended Configuration (edit ~/.axiomate.json)

{
  "models": {
    "main": { "...": "..." },
    "judge": { "...": "..." }
  },
  "model": {
    "defaultRoute": "default",
    "routes": {
      "default": { "primary": "main", "fallbackChain": [] }
    }
  },
  "auxiliary": {
    "goalJudge": {
      "primary": "judge",
      "fallbackChain": ["main"],
      "recoveryProfile": "auxiliary-judge",
      "failure": "fail_open",
      "timeoutMs": 30000
    }
  },
  "goalsMaxTurns": 20,
  "goalsParseFailureLimit": 10,
  "goalsContinuationIncludeReason": true
}

Both goalsMaxTurns and goalsParseFailureLimit can be set via /config presets; goalsContinuationIncludeReason can also be toggled via /config and is enabled by default.

There is no legacy goal-specific model alias. Judge is one of Axiomate's auxiliary tasks. After configuring auxiliary.goalJudge, the recovery runner uses its model chain and policy gates.

When does it fall back to the main route: If auxiliary.goalJudge is absent, judging falls back to model.defaultRoute. It is recommended to configure auxiliary.goalJudge so that judging cost and behavior are more controllable.
Note on reasoning models: Models that output <think>...</think> before JSON can be used; Axiomate will strip the think block before parsing. However, such models are typically slow and token-heavy, making them unsuitable as judge models.

What the Judge Prompt Looks Like

The fixed system prompt sent to the judge model roughly looks like this: 

You are a strict judge evaluating whether an autonomous agent has
achieved a user's stated goal. ...

A goal is DONE only when:
- The response explicitly confirms the goal was completed, OR
- The response clearly shows the final deliverable was produced, OR
- The response explains the goal is unachievable / blocked / needs
  user input (treat this as DONE with reason describing the block).

Otherwise the goal is NOT done — CONTINUE.

Reply ONLY with a single JSON object on one line:
{"done": <true|false>, "reason": "<one-sentence rationale>"}

"Blocked / needs user input" also counts as DONE — this is a safeguard: the judge won't let the loop keep burning tokens while the AI is stuck waiting for your answer.

6Interaction With Other Systems

/resume — Automatic Recovery

The goal state is written in the JSONL session log. axiomate --continue, axiomate --resume, or /resume <sessionId> loads the previous session, and the goal automatically surfaces — as soon as you enter the session, the status bar Goal line shows ⊙ Goal 3/20: .... The state is exactly as you left it — active or paused.

/rewind — Goal Persists After Rewind

/rewind rewinds the conversation chain or file state; the goal state does not participate in the rewind chain (it is pointer-style metadata and does not enter buildConversationChain). In other words: even if you rewind to a conversation position several turns back, the goal remains the current goal — it follows the sessionId, not the conversation chain.

/branch / --fork-session — Goal Is Not Inherited After Fork

When forking a new session, the old session's goal-state entries are not copied to the new session — the fork filter only carries transcript messages, and the new session starts with "no goal." This is intentional: forking is usually for opening a parallel experiment, and carrying the old goal doesn't make sense.

Preemption — Your Messages vs. Continuation Prompts

What happens when you type and send a new message while the goal loop is running? Two cases:

  • The message is a slash command (e.g., /subgoal ...): Does not count as a "real message" and does not preempt — after the slash command is processed, the goal continues.
  • The message is ordinary text: Preempts. Axiomate's stopHook checks the message queue before judging; if it finds a non-slash real message, it defers — your message goes first, and judging runs again at the end of the next turn after it is processed.

Judge Verdicts Do Not Pollute Conversation Context

Prompts like ↻ Continuing toward goal (3/20): reason are marked with isMeta: true — you can see them, but they do not enter the model prompt for the next turn. This avoids the problem of "the judge's prose disrupting the prompt cache." However, the continuation prompt itself is a real user message and does enter the conversation history (otherwise the AI would have no idea it should continue).

7Failure Modes

The danger of a Ralph loop is "the judge model breaks and you don't feel it at all while tokens keep burning." Axiomate has several built-in brakes:

FailureBehaviorRecovery
Judge API timeout / network error Treated as continue, not counted as a parse failure. Loop continues. Auto-recovers, no action needed.
Judge returns non-JSON / empty reply Treated as continue, consecutiveParseFailures +1. Auto-pauses after N cumulative failures (see below, N defaults to 10). Process is transparent.
Judge returns garbage for N consecutive turns (N defaults to 10, /config presets: 3/5/10/20/50/0) State flips to paused, with message ⏸ Goal paused — the judge model (N turns) isn't returning the required JSON verdict. Set auxiliary.goalJudge in ~/.axiomate.json to a stricter model. Then /goal resume to continue. Edit ~/.axiomate.json to use an instruct model, /goal resume to restore, then send an ordinary message to continue.
Turn budget exhausted (default 20) State flips to paused, with message ⏸ Goal paused — 20/20 turns used. /goal resume to restore active, then send any ordinary message to continue; or /goal clear to wrap up.
User Ctrl+C interrupts a turn Immediately pauses (no judging, no requeue), state recorded as user-interrupted (Ctrl+C). /goal resume to restore, then send an ordinary message to continue.
AI produces empty response (no text at all in the final segment) Skip judging, not counted toward turn budget (to avoid being drained by API jitter). Auto-recovers, no action needed.
auxiliary.goalJudge absent, judging falls back to main route The loop still runs, but judging may be more expensive and less stable. Edit ~/.axiomate.json to add auxiliary.goalJudge.
The true hard failure — a runaway goal. If you set a semantically vague goal (e.g., /goal make this project better), the judge will forever say continue, and the AI will keep changing things. The turn budget is a safeguard, but 20 rounds of Opus calls aren't cheap either. Recommendation — write "completion criteria" into your goal description, e.g., /goal implement a fib function and pass all test cases in fib.test.ts rather than /goal write a fib function.

8FAQ

Is Goal enabled by default?

Goal consumes no resources proactively — if you don't type /goal, the entire system stays silent. No configuration toggle is needed.

Can I use it without configuring auxiliary.goalJudge?

Yes. If auxiliary.goalJudge is absent, judging falls back to the main route. It works, but configuring a dedicated judge route is usually cheaper and more predictable.

Can I do other things while a goal is running?

Yes. Axiomate's message queue is designed with "user first" — you can type and send messages at any time; the next turn processes your message, and goal judging runs after your turn is done. You can let a goal run while using read-only slash commands like /checkpoints status, /rewind, /goal status to check state.

How do I change the 20-turn budget if it's too low or too high?

In /config, select "Goal turn budget (per /goal)" — presets are 5 / 10 / 20 / 50 / 100 / 200 / 500 / 0. 0 = unlimited budget, relying only on the judge's done verdict, the parse-failure threshold, and manual pause as safeguards; the state line and status bar Goal line will show N/∞. You can also directly edit the goalsMaxTurns field in ~/.axiomate.json to any integer value.

Be cautious when increasing this number, and set auxiliary.goalJudge alongside it — Opus x 100 turns is real money burning (roughly ~$20+). When setting to 500 or 0, consider using a cheaper model for the main conversation too.

What counts as a turn?

A turn is not the same as a tool call. One turn = the AI finishes a complete response (possibly invoking 0-N tools in between) + final stop. The judging checkpoint is the stop hook, consistent with the work unit triggered when you type and send a message. Goal judging runs once at the end of each turn. So 500 turns is not 500 tool calls — it's the AI completing 500 independent pieces of work — orders of magnitude different.

Can the goal run automatically without appearing in the transcript?

Not currently. The continuation prompt is a real user message — it goes into the transcript, consumes tokens, and is brought back by /resume. This is intentional: so you can fully audit what the Ralph loop ran. Only the judge verdicts (↻ / ✓ / ⏸ etc.) are isMeta: true and do not enter the transcript chain.

Can /goal use a custom prompt template?

Not currently. The continuation prompt and judge prompt use fixed templates; Axiomate also defaults to including the previous round's judge reason in the next continuation prompt (toggleable via goalsContinuationIncludeReason). If you need domain-specific judging (e.g., "code must pass typecheck to count as done"), a better approach is to write that criterion into the goal description, letting the judge model evaluate by your words.

Is this the same as axiomate's /skills loop?

No. /loop runs a prompt on a schedule (cron-style) with no judging; Goal runs event-driven until a semantic condition is met. /loop suits "babysit a PR every 5 minutes"; Goal suits "finish one complete thing, then stop."

How is goal persistence implemented? Does it survive axiomate process restarts?

Yes. Goal state is written as type:'goal-state' entries append-only to the session JSONL (~/.axiomate/projects/<hash>/<sessionId>.jsonl), in the same file as your conversation history. Restarting axiomate or --continue automatically reads and restores the latest goal-state entry.

Is there a "global goal list" to see all goals across all projects?

Not at the moment. Goals are per-session with no cross-session index. To check goal state across all projects, you'll need to grep goal-state entries in all JSONL files under ~/.axiomate/projects/.
