Checkpoints — File Snapshots and Rewind

Axiomate automatically records file snapshots before the AI is about to modify files. File snapshots are stored in an isolated checkpoint git store and will never be written to your project git repository. To restore files, use /rewind's File tab; to inspect storage status, use /checkpoints.

Core Model

Before each AI tool turn is about to modify files, Axiomate attempts to capture a pre-change snapshot of the current working directory. If this snapshot is identical to the tip of the current checkpoint ref, no commit is written; if there are real file differences, a commit is written in the shadow git at ~/.axiomate/checkpoints/store.

FieldMeaning
commit hash The primary key for file rewind. File tab selection and restoration are both based on the hash.
messageId Auxiliary label used to link a checkpoint back to the user prompt that triggers the next file modification.
prompt preview Display text. When lost or missing, file restoration by hash is unaffected.
+x -y Semantics differ between interfaces; see the two sections below.

The checkpoint git is a shadow repository maintained by Axiomate itself. It is isolated from your project git via GIT_DIR, GIT_WORK_TREE, and GIT_INDEX_FILE. Your .git, commit history, staging area, stash, and hooks will never be modified by the checkpoint system.

/rewind File tab

The File tab is the file restoration interface. It lists historical commits on the current working directory's checkpoint ref, showing only rows that are meaningful for file restoration. Each row's restoration target is the commit hash.

The File tab's +x -y represents the file change consequences after pressing this row. It is not the commit's own historical diff.
RowWhat File tab stats compare againstQuestion answered
Latest checkpoint checkpoint tree -> current disk What changes on the current disk would be undone by reverting to the latest checkpoint?
Older checkpoint checkpoint tree -> next checkpoint tree Reverting to this historical point: what range would files be pushed back to?

This design has older-row stats derived from immutable commits, loading faster and avoiding contamination from ongoing disk changes; only the latest row needs comparison with the current disk.

The stats in the File tab list are a quick preview. After selecting a row with Enter, the confirmation page re-computes the diff for just that one restore hash against the current disk, so the +x -y and Restore file availability on the confirmation page reflect the disk state at selection time without reloading the entire list.

When rewind executes, it first creates a pre-rewind safety snapshot and then restores the target file tree. If restoration fails midway, the File tab will show a row like ↶ Before rewind, which you can select to restore the state before the rewind.

/checkpoints and CLI

/checkpoints is the read-only/maintenance entry point for the checkpoint store, not a file restoration selector. To actually restore files, use /rewind.

CommandPurpose
/checkpoints status
axiomate checkpoints status		 View checkpoint base, store size, project count, and recent projects.
/checkpoints list
axiomate checkpoints list		 Read-only listing of the current working directory's checkpoint commit history.
/checkpoints prune
axiomate checkpoints prune --force		 Clean up orphan, stale, over-count, and over-capacity checkpoint data.
/checkpoints clear
axiomate checkpoints clear --force		 Clear the entire checkpoint store. The CLI requires an explicit --force flag.
/checkpoints list +x -y semantics differ: it shows the checkpoint commit's own historical diff, i.e., parent(commit) -> commit. The same hash may display different numbers in /rewind because /rewind shows the restoration consequences after selecting that row.

/checkpoints list shows the actual commits on the checkpoint ref, including safety snapshots automatically created before rewinds and a small number of historical/external commits whose subject cannot be parsed into Axiomate's structured format. /rewind hides external commits by default because the file restoration interface requires interpretable restoration labels and consequences.

The TURN column does not display the raw git commit subject; it is formatted based on the checkpoint commit message:

  • Normal AI turn with prompt: <preview> body: displays Before "<preview>".
  • Normal AI turn without prompt body: displays Before <label> (<first 8 chars of messageId>).
  • Pre-rewind safety snapshot with target: <preview> body: displays Undo rewind to before "<preview>".
  • Pre-rewind safety snapshot without target body: displays Undo last rewind.
  • External/historical commit: displays the raw subject; empty subject displays (no subject).

The row count can be adjusted via --rows or the checkpointsStatusRows config option. Default is 50 rows, with a range of 1..500.

Nested Git and File Discovery

The checkpoint system tracks the current working directory as a whole filesystem snapshot rather than reusing your project git's staging area. To support nested git, submodules, and worktrees, Axiomate does not use the user repository's git add -A in production paths.

Current discovery and staging rules:

  • Regular files enter the snapshot as their current on-disk bytes.
  • VCS metadata such as .git, .hg, .svn are always excluded.
  • Regular files inside nested repos are included in the snapshot.
  • A nested repo's own staged/dirty/untracked state does not affect the checkpoint.
  • The .gitignore in the user project takes effect.
  • Parent directory ignore rules can prune an entire directory before entering a nested repo.

In testing, a fresh-index git add -A may be used as an oracle to verify normal tree semantics; this is only a test baseline, not the production implementation.

Default Limits

Config/LimitDefaultDescription
checkpointsStatusRows 50 Default row count displayed for status/list, range 1..500.
checkpointsMaxSnapshotsPerProject 1000 Maximum number of snapshots retained per project ref. Set to 0 for unlimited.
checkpointsMaxFiles 200000 Maximum number of working directory files scanned per snapshot. Set to 0 for unlimited.
Single file size 50 MB Files exceeding this size will not be included in the checkpoint managed tree.
Stale prune 30 days Project refs untouched for an extended period are automatically cleaned up.
Store size cap 1000 MB When exceeded, prune removes older commits, but at least one commit is kept per ref.

Failures and Recovery

SymptomMeaningAction
A certain AI change has no checkpoint Likely no real file differences, exceeded file count/size limits, git unavailable, or a transient error. Will retry on the next turn with file differences. For troubleshooting, check the debug log and /checkpoints status.
Rewind reports snapshot not found The target commit may have been removed by prune, clear, or external gc. Exit and reopen /rewind to refresh the list.
Rewind file restoration fails Failure during file restoration; disk may be partially modified. Detailed git/pathspec diagnostics are only written to the debug log. Reopen /rewind, switch to File tab, and select the latest restoration row; typically ↶ Before rewind.
Checkpoint store too large Excessive historical commits or large files. Run /checkpoints prune or adjust count/capacity config.

The checkpoint system as a whole is fail-open: snapshot failures do not block the AI tool from continuing. Rewind is the exception: if there is no safety snapshot or the target commit cannot be confirmed, Axiomate will halt before modifying disk.
