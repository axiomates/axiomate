OpenAI Responses Cache Compatibility Configuration Guide

This document explains the purpose, configuration, and applicable scenarios for promptCacheKey, promptCacheRewriteLimit, and codexTransportCompat. They only affect models with protocol: "openai-responses".

When to Configure

These fields are generally only needed with third-party OpenAI Responses compatible gateways, relay services, or self-hosted proxies. The official OpenAI endpoint typically does not require them.

Scenario Recommendation
Standard OpenAI Responses API Do not configure these fields; keep default behavior.
Third-party gateway requires prompt_cache_key in the body for reliable cache hits Configure promptCacheKey: true.
Third-party gateway reads x-codex-* headers like a Codex endpoint Configure codexTransportCompat: true; if it also requires a body key, configure promptCacheKey as well.
The server returns its own prompt_cache_key Keep the default promptCacheRewriteLimit: 3 to let Axiomate automatically reuse the server key.
The server rewrites the key every time, but you still want to always send the latest key Set promptCacheRewriteLimit: 0.

These fields do not guarantee that the provider will achieve KV cache hits. They only send stable request identifiers following common conventions of compatible gateways; whether cache hits actually occur still depends on how the gateway and model service implement caching.

Field Overview

Field Type Default Purpose
promptCacheKey boolean | string Not set Controls whether to send a dedicated prompt_cache_key in the Responses request body.
promptCacheRewriteLimit number 3 How many consecutive server rewrites of prompt_cache_key before stopping the dedicated body key. 0 means never disable because of this.
codexTransportCompat boolean Not set Controls whether to send Codex-style transport headers, such as originator, session_id, x-client-request-id, x-codex-window-id, and x-codex-turn-metadata.
userAgent string Not set Existing field. When codexTransportCompat is enabled, used as the User-Agent with priority; when not set, the Codex-compatible default UA is used.

How to Use promptCacheKey

Disabled

When not set or set to false, Axiomate will not send a dedicated body prompt_cache_key and will not record the server-returned key.

{
  "models": {
    "main": {
      "protocol": "openai-responses",
      "model": "gpt-5",
      "baseUrl": "https://api.example.com/v1",
      "apiKey": "sk-..."
    }
  }
}

Using the Default Key

When set to true, the request body will send a default key:

a:{projectHash}:{providerHash}:{sessionHash}

In an actual request, it renders to something like:

a:9f3a1c2d4e5f6780:4a7b8c9d0e1f2345:5c9d2a71e8b0f3c4
{
  "models": {
    "main": {
      "protocol": "openai-responses",
      "model": "gpt-5",
      "baseUrl": "https://api.example.com/v1",
      "apiKey": "sk-...",
      "promptCacheKey": true
    }
  }
}

The default key is bound to the session but uses a short hash format, length 52, suitable for Responses-compatible gateways that require prompt_cache_key to be no longer than 64 characters.

Custom Template

When set to a string, Axiomate treats it as a template for rendering. Supported variables:

  • {projectHash}: Hash of the current project path.
  • {providerHash}: Hash of the current Responses provider's baseUrl.
  • {sessionHash}: 16-character hash of the current Axiomate session ID, used by the default template.
{
  "models": {
    "main": {
      "protocol": "openai-responses",
      "model": "gpt-5",
      "baseUrl": "https://api.example.com/v1",
      "apiKey": "sk-...",
      "promptCacheKey": "axiomate:project:{projectHash}:{providerHash}"
    }
  }
}

This template above does not include {sessionHash}, which means different sessions under the same project and provider will share the same client key. This is only recommended when you explicitly want cross-session reuse.

Many gateways require prompt_cache_key to be at most 64 characters. Template variables only provide short hashes, not the full session ID. Custom templates should still avoid overly long fixed prefixes.

Relationship with extraParams.prompt_cache_key

extraParams.prompt_cache_key will still be sent as a regular passthrough field, but will not enable the server key state machine.

If both the dedicated promptCacheKey and extraParams.prompt_cache_key are configured, the dedicated promptCacheKey will override prompt_cache_key in the body.

Server Rewrite Behavior

Some Responses-compatible gateways return their own prompt_cache_key in the response. When the dedicated promptCacheKey is enabled, Axiomate will observe this return value.

  1. If the returned key matches the request key: considered stable, consecutive rewrite count resets to zero.
  2. If a different key is returned: save the server key, and use the server key for the next request.
  3. If the server returns a new key different from the request value multiple times in a row: once the count reaches promptCacheRewriteLimit, mark that key state as unsupported, and subsequently stop sending the dedicated body key.

The default for promptCacheRewriteLimit is 3.

Configuration Behavior
promptCacheRewriteLimit: 3 Default. The 1st and 2nd consecutive rewrites continue reusing the new server key; after the 3rd consecutive rewrite, stop sending the dedicated key.
promptCacheRewriteLimit: 1 Stop sending the dedicated key on the first rewrite.
promptCacheRewriteLimit: 0 Never disable due to rewrite; whenever the server returns a new key, save it and continue sending the latest key on the next request.

State is saved in the prompt-cache-state directory under the Axiomate configuration directory. The default key includes {sessionHash}, so default state is also session-level isolated; when a custom template does not include {sessionHash}, state is reused across sessions.

How to Use codexTransportCompat

When set to true, Axiomate will send Codex-style transport metadata in the headers of OpenAI SDK request options.

{
  "models": {
    "main": {
      "protocol": "openai-responses",
      "model": "gpt-5",
      "baseUrl": "https://proxy.example.com/v1",
      "apiKey": "sk-...",
      "codexTransportCompat": true
    }
  }
}

When only codexTransportCompat is enabled, the body prompt_cache_key is not automatically sent. These two configurations are orthogonal.

When only headers are enabled, the header tokens use the real Axiomate session ID. The headers sent include:

  • originator: codex_exec
  • session_id: <sessionId>
  • x-client-request-id: <sessionId>
  • x-codex-window-id: <sessionId>:0
  • x-codex-turn-metadata, containing session_id, turn_id, and sandbox: "none".
  • User-Agent. If the model has userAgent configured, it is used with priority; otherwise, the Codex-compatible default UA is used.

Enabling Both promptCacheKey and codexTransportCompat

In this case, the behavior closely matches the Codex/Hermes compatibility patch: the body prompt_cache_key and Codex headers use the same selected key.

{
  "models": {
    "main": {
      "protocol": "openai-responses",
      "model": "gpt-5",
      "baseUrl": "https://proxy.example.com/v1",
      "apiKey": "sk-...",
      "promptCacheKey": true,
      "promptCacheRewriteLimit": 3,
      "codexTransportCompat": true
    }
  }
}

If the server returns a new prompt_cache_key, subsequent body and headers will all switch to the server key. If the rewrite limit is reached and marked unsupported, both the body key and these compat headers will stop being sent.

Body Cache Key Only

{
  "models": {
    "responses-proxy": {
      "name": "Responses Proxy",
      "protocol": "openai-responses",
      "model": "gpt-5",
      "baseUrl": "https://proxy.example.com/v1",
      "apiKey": "sk-...",
      "promptCacheKey": true
    }
  }
}

Codex-Style Relay

{
  "models": {
    "codex-like-proxy": {
      "name": "Codex-like Responses Proxy",
      "protocol": "openai-responses",
      "model": "gpt-5",
      "baseUrl": "https://proxy.example.com/v1",
      "apiKey": "sk-...",
      "promptCacheKey": true,
      "promptCacheRewriteLimit": 3,
      "codexTransportCompat": true
    }
  }
}

Relay Requiring a Custom User-Agent

{
  "models": {
    "codex-like-proxy": {
      "protocol": "openai-responses",
      "model": "gpt-5",
      "baseUrl": "https://proxy.example.com/v1",
      "apiKey": "sk-...",
      "promptCacheKey": true,
      "codexTransportCompat": true,
      "userAgent": "codex_exec/0.120.0 (Ubuntu 22.4.0; x86_64) gnome-terminal (codex_exec; 0.120.0)"
    }
  }
}

If there are no special requirements, you do not need to manually set userAgent. When codexTransportCompat is enabled and userAgent is not configured, Axiomate will use the Codex-compatible default UA.

Cross-Session Client Key Reuse

{
  "models": {
    "responses-proxy": {
      "protocol": "openai-responses",
      "model": "gpt-5",
      "baseUrl": "https://proxy.example.com/v1",
      "apiKey": "sk-...",
      "promptCacheKey": "axiomate:project:{projectHash}:{providerHash}"
    }
  }
}

This template does not include {sessionHash}, suitable when you explicitly want multiple sessions in the same project and provider to share the same cache namespace.

FAQ

Symptom Check
No improvement in cache hits after configuration Verify the model uses the openai-responses protocol; confirm the relay actually reads prompt_cache_key or Codex headers; confirm that the system prompt, tool list, cwd, model, and provider are not frequently changing.
Multiple Axiomate instances interfere with each other Use the default promptCacheKey: true, which includes {sessionHash}. Do not use a custom template without session.
Want to reuse cache across sessions Set promptCacheKey to a template without {sessionHash}, for example axiomate:project:{projectHash}:{providerHash}.
Relay returns a different key every time Keeping the default rewrite limit will stop sending the dedicated key after 3 consecutive rewrites; if you want to always follow the server key, set promptCacheRewriteLimit: 0.
Only want to mimic Codex headers, not send body key Only set codexTransportCompat: true, do not set promptCacheKey.
Only want to send body key, not Codex headers Only set promptCacheKey, do not set codexTransportCompat.
