Checkpoints — File Snapshots & Conversation Replay

Axiomate's /rewind operates on two independent axes. File axis: before each file-modifying action, a snapshot of the working directory is taken in the background and stored in a separate git shadow repo; you can restore the disk to any historical turn. Conversation axis: built on top of JSONL session logs, lets you jump back to any Q/A pair, with the input box pre-filled so you can "redo" that turn. The two axes are 100% decoupled -- files don't touch conversations, conversations don't touch files. The entire system is nearly transparent to you -- you barely notice it day to day, but it has your back when it matters.

1How It Works

Checkpoints is an independent git repo living at ~/.axiomate/checkpoints/store/, completely separate from your project's own git repository. Before Axiomate modifies any files, it writes a commit in this shadow repo to record the current state.

Key Design

Shadow Repo

Doesn't pollute your project's own .git/. Your git log and git status will never see these snapshots.

One Ref Per Project

Different working directories write to different refs/axiomate/<hash>, mutually isolated.

Structured Commit Subject

Format axiomate:<messageId>:<label>, traceable back to the conversation turn.

Respects .gitignore

When traversing the working directory, ignored directories like node_modules/ are not included in snapshots.

Advantages Over Similar Systems

Axiomate's file checkpointing draws from hermes-agent's pre-tool workdir snapshot model -- a battle-tested foundation. On top of that, axiomate does several things hermes doesn't:

Dual-Axis Rewind + Dual Cursors

Hermes only does file rollback. Axiomate adds an independent conversation replay axis -- press Tab in the picker to switch to the Conversation tab, jump back to any Q/A pair, and the input box is automatically pre-filled with the next prompt so you can redo it. File and Conversation are fully decoupled. The Selection cursor (▶) and Head cursor ((current)) are conceptually separate -- your visual cursor is not session state. More explicit than hermes's merged "cursor = state" model.

Persistent Conversation Rewind + Future Nodes Visible

Conversation rewind head state is written into the JSONL (a type:"head" entry). /exit and then resume the same session via the UI -- your position matches where you rewound to, unlike traditional in-memory truncation that's lost on exit. After rewinding, turns newer than head still appear in the picker rendered in dim gray (the data is still in the JSONL, not deleted), ready to be selected again at any time. "Rewind, then redo" is axiomate's core mental model.

Pre-Rewind Safety Net on Every Rewind

Hermes's rewind is a one-way operation -- once you go back, later state is lost. Axiomate takes a safety-net snapshot of the current workdir before every rewind; if you regret it, pick the ↶ row in the picker to undo the rewind itself. Infinitely nestable -- rewind on rewind never loses data.

Nested Rewind Shown as Hash Chain

After several consecutive rewinds, the picker renders these ↶ rows as ↶ Rewind to <target-hash> [own-hash] -- you can see at a glance "which rewind this one branched off of." The hierarchy is explicit in text, not held in memory.

Event-Aligned CHANGES Column (Git Log Mental Model)

The CHANGES column in the picker, list, and status describes what this row's turn wrote (anchor[i] vs anchor[i-1]), matching how you read git log. Hermes's original commit-vs-parent model has an off-by-one: a row's CHANGES actually belongs to the next row's turn, which is initially confusing.

Aggressive Defaults (Tuned for Modern Dev Machines)

Default caps: 5000 snapshots (configurable via /config) / 60-day retention / 5 GB pool / 50 MB per file / 200,000 files per workdir / 5,000 rows max. Hermes defaults to 7-day retention, which easily loses data across sprint boundaries; axiomate's 60 days + 5 GB pool fits modern development cycles including vacations and cross-sprint context switches.

UX Edge: Newest-First + Adaptive Timestamps

The picker defaults to newest at the top (hermes has no equivalent), so your most recent operation is immediately visible when you open it. Timestamps are adaptive: just now → just now, minutes ago → 5m ago, crossing a day → absolute date.

Full /checkpoints CLI

Beyond list, axiomate provides status / prune / clear subcommands plus a command-line version. The --keep-orphans safety valve prevents snapshot loss when an external drive is disconnected.

2Automatic Behavior No Setup Needed

When Snapshots Are Taken Automatically

Before every model call to a file-modifying tool (Write / Edit / NotebookEdit), Axiomate writes a commit in the shadow repo. All of this is enabled by default with no configuration required.

When Snapshots Are Skipped Automatically

ConditionBehavior
Current directory has more than 200,000 filesSkip this snapshot (too expensive)
A single file exceeds 50 MBThat file is excluded from the index; other files are snapshotted normally
Current directory is / / ~ / drive rootEntire snapshot is refused (safety guard)
No changes since last snapshotNo empty commit is written
git is not installed on the systemThe entire checkpoint subsystem becomes a no-op, no error thrown

When Automatic Cleanup Runs

Axiomate runs a background prune when you've been idle for >= 1 minute and the process has been running for at least 10 minutes. The cleanup covers:

  • Orphans: project refs whose working directories have been manually deleted from disk are removed.
  • Stale: project refs untouched for more than 60 days.
  • Size-cap: when all snapshots exceed 5 GB total, oldest ones are trimmed first.
  • Session anchor (keep-ref): if a ref about to be deleted still has commits referenced by sessions within the last 28 days, it's anchored into refs/axiomate/_keep/ and preserved until the corresponding session also expires.

Cleanup runs at most once every 24 hours (throttled by a .last_prune marker) and won't repeat. All prune operations are fail-soft -- errors are logged but never surfaced.

3User Commands Manual Invocation

/rewind — Go Back to Before a Turn

Type /rewind in the REPL to open an interactive picker listing turns from this session along with historical snapshots. Each row's label reads Before "<prompt>" -- selecting a row means restoring state to right before that turn ran.

Two Tabs: File vs Conversation

The picker opens on the File tab by default. Press Tab to switch to the Conversation tab. The two tabs are independent subsystems: different list sources, different row semantics, different chooser actions, different persistence. Each tab gets its own section below.

File Tab — File Rollback

Mental Model: Each Anchor Is a Pre-Tool Snapshot

Axiomate takes snapshots before each tool call (same as hermes-agent). So an anchor's tree describes "what the disk looked like before that turn ran." Picker rows explicitly say Before "<prompt>" to avoid mistaking an anchor for a "git log commit." Stats on the right side of each row (file +X -Y) describe what changed on disk between the previous anchor and this anchor, matching git log's commit-vs-parent mental model. Data comes from the shadow repo (~/.axiomate/checkpoints/store/), independent of JSONL session logs.

Only turns that affected disk files are shown (turns where the AI called Edit / Write / NotebookEdit / modified files via Bash). Pure chat or read-only Q&A turns do not appear.

Sorting: Newest-First

The picker, like /checkpoints list, shows newest at the top: 

(current)
↶ Rewind to 041da07 [71c90d4]    test.txt +1 -1   just now
↶ Rewind to before "v2 to v3" [041da07]   test.txt +1 -1   just now
Before v2 to v3                test.txt +1 -1   2m ago
Before v1 to v2                test.txt +1 -1   3m ago
Before create test.txt with v1    test.txt +1 -0   4m ago

The default cursor lands on the second row (the nearest rewindable target). (current) is not selectable -- it's a layout anchor indicating "now." g jumps to the first selectable row (skipping (current)), G jumps to the end. j / k move up and down.

Chooser: What You Can Do After Selecting a Row

  • Has diff: Restore file / Never mind. Restores the disk to before this anchor's state. Chat is unaffected.
  • No diff: only Never mind (anchor.tree matches current disk, so restore would be a no-op).

Pre-Rewind Safety Net

Before every Restore file action, Axiomate automatically takes a pre-rewind safety-net snapshot (subject format: axiomate:<synth-uuid>:pre-rewind:<hash>). Its tree captures the disk state just before the rewind operation executes. If a rewind overwrites important changes you made, just open /rewind again, pick that ↶ Rewind to before "<X>" row → Restore file, and your original state comes back. Infinitely nestable: undoing a rewind itself also creates another safety-net snapshot.

Picker Row Variants

Every ↶ row ends with a [7-char short hash] -- this is that anchor's own git short id. Regular conversation turn rows have no hash (the user's prompt itself serves as the row's identifier). All anchor-bearing rows display stats + time on the right.

  • Before "create test.txt with v1" -- anchor of a regular conversation turn. In the File tab, the Before prefix reminds you this is a snapshot from "before <X> ran."
  • ↶ Before "<prompt>" [<hash>] -- anchor of a conversation turn not on the current chain. E.g., after conversation has been rewound away, the file state left behind by that turn is still restorable.
  • ↶ Rewind to before "<target>" [<hash>] -- the pre-rewind safety net from a rewind operation. Selecting it undoes that rewind. The quoted text is the prompt of the turn that rewind was targeting.
  • ↶ Rewind to <target-hash> [<own-hash>] -- safety net from a nested rewind (undoing a prior rewind operation itself). The bare 7-char hash is the short id of the target it was restoring to; the bracketed hash is this row's own short id. Visually, bare = "who I reference," bracketed = "who I am," letting you follow hashes upward to trace the entire rewind chain.
  • ↶ Off-branch anchor [<hash>] -- an anchor with no conversation context (from a legacy session or an externally committed snapshot in the store).
  • (current) -- current disk state placeholder, not selectable.

Timestamps on Each Row

The time column to the right of stats follows the same rules as /checkpoints list: just now (<1 min) / 5m ago (<1 hour) / 3h ago (<24 hours) / 2026-05-24 09:42 (≥24 hours).

What the CHANGES Column Means

Each row's stats describe what that row's turn itself wrote (not the previous row). Algorithm:

  • The newest row: anchor.tree vs current disk ("what the disk currently looks like after the latest turn ran"). If you manually changed files outside axiomate, this row's stats will reflect that -- it shows the full diff from anchor.tree to current disk.
  • All other rows: anchor.tree vs the previous (newer) anchor's tree. Equivalent to "the state produced after this row's turn ran, feeding into the next anchor," frozen and immutable.
  • file +X -Y / N files +X -Y for single/multi-file changes. (no diff) means this turn didn't change files (e.g., pure chat / read-only Q&A).

Feedback Messages

After Restore file completes, the transcript includes a line: 

● File rewound to "v2 to v3". Use /rewind → "↶ Rewind" row to undo.

Restore conversation produces no feedback -- the truncated conversation chain is self-evident (trimmed turns disappear from the screen, the cut point's prompt returns to the input box).

Empty Directory Behavior

When you start Axiomate in a brand-new empty directory and let it create the first file, that turn's anchor is a synthetic root snapshot (empty tree). Restoring to this anchor will delete all files Axiomate created afterward.

Slash Commands Don't Appear in the Picker

/checkpoints status, /help, /clear, and similar are not conversation turns -- they are tool invocations you ran in the REPL, and the picker automatically filters them out. "Rewind to before /checkpoints status" is the same state as rewinding to the real turn before it.

Conversation Tab — Conversation Replay

Mental Model: Each Row = One Q/A Pair

Shows all real user prompts on the current chain. Each row = one complete Q&A pair (user prompt + AI response). The (current) tag marks which pair you're standing on -- i.e., the turn that "head" points to. Data comes from the JSONL session log (~/.axiomate/projects/.../*.jsonl), independent of disk files.

Example rendering (sent hi → ok → good, no rewind yet): 

good (current)
ok
hi

Reverse-chrono order, newest Q/A at the top. Cursor auto-lands on the (current) row; ↑/↓ move up and down. No Before "<prompt>" prefix like the File tab -- here the row text is the prompt itself.

Past / Future: Future Rows Are Dimmed

Rewind moves head to an earlier turn. Turns newer than head remain visible in the picker (they still exist in the JSONL), but are rendered in dim gray -- a reminder that they are "future" nodes, disappeared from the chat area but still selectable to redo. 

good          ← dim gray (future)
ok (current)  ← default color (current)
hi            ← default color (history)

As long as you haven't run /clear or /branch within the same session, future nodes are never lost no matter how many times you rewind.

Two Cursors: Selection and Head

The Conversation tab has two cursors simultaneously -- don't confuse them:

  • Selection cursor (left-side ▶) -- the visual cursor you move with ↑/↓, determining which row gets selected when you press Enter. Auto-lands on the head row when the picker opens.
  • Head cursor (row-end (current) tag) -- the session's current state pointer, "which Q/A pair am I standing on." Head does not move with ↑/↓; it only changes position after you perform Restore conversation.

The two cursors can overlap: when the picker first opens, selection also sits on the head row, so that row shows both ▶ and (current). Move with ↑/↓ and selection goes elsewhere while head stays on the (current) row -- the two pieces of information are independent.

Rewind Semantics: Truncate to This Pair + Pre-fill the Next Prompt

Select a row X in the picker and press Enter; the chooser offers Restore conversation. On execution:

  • Chat area: truncated to the end of pair X (including X's user prompt and AI response); turns after X disappear from the screen (data remains in JSONL).
  • Input box: automatically pre-filled with the next user prompt after X -- so you can "revise this Q&A" and resubmit. If X is already the chain tail (no next prompt), the input box is cleared.
  • Head record: a line is written to JSONL marking that you're now standing on pair X.

Selecting the (current) row itself: no-op (you're already there, redoing is meaningless). Enter does nothing; the picker stays open.

Persistence: /exit Then Resume -- Still in Place

Conversation rewind head is written to JSONL as a type: "head" entry. /exit + restart + UI resume the session, and you land exactly where you rewound. Open /rewind again and the (current) tag is still on row X -- state is not lost.

What the Picker Does Not Show

The following are automatically filtered out to reduce noise:

  • tool_use / tool_result (in JSONL type=user but content is tool execution output, not a real prompt)
  • Continue from where you left off. continuation line injected during resume (isMeta=true)
  • Input/output wrappers for slash commands like /clear / /help
  • Compact summary placeholders, transcript-only metadata
  • Pure attachment / read-only "non-prompt" messages

Command Line (CLI)

The same subcommands can be invoked from axiomate's command line without entering the TUI -- useful for scripting, CI integration, or a quick status/cleanup without launching the full REPL. 

axiomate checkpoints status
axiomate checkpoints status --rows 50
axiomate checkpoints list
axiomate checkpoints list --rows 50
axiomate checkpoints prune --force
axiomate checkpoints prune --retention-days 90 --max-size-mb 10000
axiomate checkpoints prune --keep-orphans --force
axiomate checkpoints clear --force

Each subcommand rejects excess positional arguments (emitting too many arguments), never silently ignoring them. clear on the CLI requires an explicit --force, since it bypasses the interactive confirmation in the slash-command version.

Controlling status / list Row Count

status and list tables default to 100 rows max. To see more (or fewer) rows, pass an integer in the 1..5000 range. Slash commands use positional arguments (/checkpoints status 50, /checkpoints list 50); CLI uses --rows N, consistent with other CLI flags (axiomate checkpoints status --rows 50). Non-integers or out-of-range values are rejected.

To make the override stick long-term, write checkpointsStatusRows in Settings (labeled Checkpoints rows shown, cycles through 30/50/100/200/500/1000/2000/5000), or use /config checkpointsStatusRows N, or directly edit ~/.axiomate.json. Per-invocation positional arguments (slash command) or --rows N (CLI) always take precedence over the config value.

--keep-orphans — Preserve Temporarily Missing Workdir Refs

By default, prune deletes project refs whose working directories have vanished from disk. But if you've temporarily disconnected an external drive, renamed a path intending to rename it back, or plan to re-clone to the original path later, add --keep-orphans (CLI) or keep-orphans (slash command): this prune skips the orphan pass while still running stale and size-cap as usual. /checkpoints prune keep-orphans is also supported in the REPL. The number of refs protected by this safety valve appears in the output as Orphan refs skipped: N.

Column meanings:

  • WHEN -- anchor timestamp, adaptive format (<1m = just now, <1h = Nm ago, <24h = Nh ago, ≥24h = 2026-05-24 09:42).
  • ID -- git short hash (7 characters). Can be fed directly to external git commands, e.g., git --git-dir=~/.axiomate/checkpoints/store show 8da1461. Full hash + message UUID available in --debug logs.

All anchors are shown. To restore, select the corresponding row in /rewind → Restore file.

5Failure Modes Safety Net

The checkpoint subsystem's core contract is fail-open -- any individual snapshot failure must never block the main workflow. This means:

  • git not installed on the system → entire subsystem is a no-op, no error raised.
  • Transient git command failure (EBUSY, timeout) → skipped this time, retried automatically next turn.
  • Working directory with more than 200,000 files → skipped to avoid git add hanging.
  • Disk full / permission denied → error goes to logForDebugging, conversation continues.

Cross-Workdir Edge Cases

If you /resume an old session but the current working directory differs from the original, before /rewind Axiomate will show a system message indicating one of:

  • reachable: snapshot is in the current ref, rewinding works normally.
  • reachable-other-worktree: snapshot exists but belongs to a different working directory. You'll be prompted to cd there.
  • unreachable: snapshot has been pruned, rewind will fail.

Orphan Notice During Cleanup

The tail of /checkpoints status output may include a line like: 

Note: 3 snapshots from 1 orphan workdir will be discarded on next prune.

If you see this line and that workdir's commits are still useful (e.g., you just unplugged an external drive), restore the workdir to its original path and the next prune will recognize it as live again.

6FAQ

Will it pollute my git repo?

No. All commits are written to ~/.axiomate/checkpoints/store/, completely independent of your project's own .git/. git status, git log, git push -- none of them see these commits.

How much disk space does it use?

Hard cap of 5 GB (configurable). git itself performs object deduplication and zlib compression, so editing the same file 100 times only stores a few deltas on top. Typical daily usage is in the hundreds of MB to 1-2 GB range.

Can I sync across machines?

No. Checkpoints are local machine snapshots, not designed for cross-machine sync, encryption, or signing. Switching machines means starting fresh with no history.

Can I export / import?

There is no checkpoints export / import yet. The shadow repo itself is a standard bare git repo; git-savvy users can inspect it directly with git --git-dir=~/.axiomate/checkpoints/store ....

How do I turn it off completely?

Set fileCheckpointingEnabled to false in Settings, or use the environment variable AXIOMATE_CODE_DISABLE_FILE_CHECKPOINTING=1. Previously accumulated snapshots are not auto-cleaned; run /checkpoints clear manually if desired.

/rewind says commit not found -- what do I do?

Most likely that turn's commit has already been pruned (older than 60 days and not referenced by the current session). This is expected behavior. For critical milestones, keep a separate copy of the relevant workdir output; don't rely solely on checkpoints.
