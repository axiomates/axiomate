Checkpoints — 文件快照与 rewind

Axiomate 会在 AI 准备改文件之前自动记录文件快照。文件快照存放在独立的 checkpoint git store 中，不会写入你的项目 git 仓库。需要回退文件时， 使用 /rewind 的 File tab；需要查看存储状态时，使用 /checkpoints。

核心模型

每次 AI 工具回合准备修改文件之前，Axiomate 会尝试对当前工作目录做一次 pre-change snapshot。如果这次快照和当前 checkpoint ref 顶端完全相同， 不会写 commit；如果有真实文件差异，就会在 ~/.axiomate/checkpoints/store 的 shadow git 中写入一个 commit。

字段含义
commit hash 文件 rewind 的主键。File tab 选择和恢复都以 hash 为准。
messageId 辅助标签，用来把 checkpoint 关联回触发下一次文件修改的用户 prompt。
prompt preview 展示用文本。丢失或缺失时，不影响按 hash 恢复文件。
+x -y 不同界面语义不同，见下面两节。

checkpoint git 是 Axiomate 自己维护的影子仓库。它通过 GIT_DIR、GIT_WORK_TREE 和 GIT_INDEX_FILE 与你的项目 git 隔离。你的 .git、commit 历史、暂存区、stash、hooks 都不会被 checkpoint 系统修改。

/rewind File tab

File tab 是文件回退界面。它列出当前工作目录 checkpoint ref 上的历史 commit， 只显示有文件回退意义的行。每一行的恢复目标都是 commit hash。

File tab 的 +x -y 表示按下这行后的文件变化后果。 它不是这个 commit 自己的历史 diff。
行File tab 统计的比较对象回答的问题
最新 checkpoint checkpoint tree -> 当前磁盘 回到最新 checkpoint 会撤销当前磁盘上的哪些变化？
更旧 checkpoint checkpoint tree -> 下一个 checkpoint tree 回到这个历史点，会把文件推回到哪个区间？

这个设计让旧行的统计来自不可变 commit，加载更快且不会因为当前磁盘继续变化 而污染；只有最新行需要和当前磁盘比较。

rewind 执行时会先创建一个 pre-rewind safety snapshot，然后再恢复目标文件树。 如果恢复中途失败，File tab 会出现类似 ↶ Before rewind 的行， 你可以选择它恢复到 rewind 之前的状态。

/checkpoints 与 CLI

/checkpoints 是 checkpoint store 的只读/维护入口，不是文件回退 selector。实际回退文件请使用 /rewind。

命令作用
/checkpoints status
axiomate checkpoints status		 查看 checkpoint base、store 大小、项目数量、最近项目。
/checkpoints list
axiomate checkpoints list		 只读列出当前工作目录的 checkpoint commit 历史。
/checkpoints prune
axiomate checkpoints prune --force		 清理 orphan、stale、超数量和超容量的 checkpoint 数据。
/checkpoints clear
axiomate checkpoints clear --force		 清空 checkpoint store。CLI 必须显式传 --force。
/checkpoints list 的 +x -y 语义不同： 它展示 checkpoint commit 自身的历史 diff，即 parent(commit) -> commit。同一个 hash 在 /rewind 中可能显示不同数字，因为 /rewind 显示的是 选择该行后的回退后果。

行数可以通过 --rows 或配置项 checkpointsStatusRows 调整。默认显示 50 行，范围是 1..500。

Nested git 与文件发现

checkpoint 系统把当前工作目录当作一个大的 filesystem snapshot 来追踪， 而不是复用你的项目 git 暂存区。为了支持 nested git、submodule 和 worktree， Axiomate 不在生产路径使用用户仓库的 git add -A。

当前发现与 staging 规则：

  • 普通文件按当前磁盘字节进入 snapshot。
  • .git、.hg、.svn 等 VCS metadata 永远排除。
  • nested repo 里的普通文件会被纳入 snapshot。
  • nested repo 自己的 staged/dirty/untracked 状态不会影响 checkpoint。
  • 用户项目里的 .gitignore 会生效。
  • 父目录 ignore 可以在进入 nested repo 前剪掉整个目录。

测试中可能用 fresh-index git add -A 作为 oracle 验证普通树语义； 这只是测试基准，不是生产实现。

默认限制

配置/限制默认值说明
checkpointsStatusRows 50 status / list 默认显示行数，范围 1..500。
checkpointsMaxSnapshotsPerProject 1000 每个项目 ref 最多保留的 snapshot 数。设为 0 表示不限制。
checkpointsMaxFiles 200000 单次 snapshot 允许扫描的工作目录文件数。设为 0 表示不限制。
单文件大小 50 MB 超过该大小的文件不会进入 checkpoint managed tree。
stale prune 30 天 长时间未触碰的项目 ref 会被自动清理。
store size cap 1000 MB 超过上限时 prune 会删除较旧 commit，但每个 ref 至少保留一个 commit。

失败与恢复

现象含义处理
某次 AI 改动没有 checkpoint 可能是没有真实文件差异、超过文件数/大小限制、git 不可用或临时错误。 下一次有文件差异时会再次尝试。需要排查时看 debug log 和 /checkpoints status。
rewind 报 snapshot 不存在 目标 commit 可能被 prune、clear 或外部 gc 移除。 退出并重新打开 /rewind 刷新列表。
rewind failed mid-way 恢复文件过程中失败，磁盘可能部分修改。 重新打开 /rewind，选择 ↶ Before rewind 行恢复。
checkpoint store 太大 历史 commit 或大文件过多。 运行 /checkpoints prune 或调整数量/容量配置。

checkpoint 系统整体是 fail-open 的：快照失败不会阻止 AI 工具继续执行。 rewind 是例外：如果没有安全 snapshot 或无法确认目标 commit，Axiomate 会在 修改磁盘前停止。
