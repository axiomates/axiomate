Checkpoints — 文件快照与对话回放

Axiomate 的 /rewind 在两条独立轴上工作。File 轴： 每次准备改文件之前在后台拍一张工作目录的快照，放进独立的 git 影子仓库； 你可以把磁盘还原到任何一个历史回合。Conversation 轴： 在 JSONL 会话日志的基础上，让你回到某一组 Q/A 对话的位置，连同输入框 的回填一起送你"重做"那个回合。两条轴 100% 脱钩 — 文件不动对话、对话 不动文件。整个系统对你近乎透明 — 平时几乎感觉不到，关键时刻它替你兜底。

1它怎么工作

Checkpoints 是一个独立的 git 仓库，住在 ~/.axiomate/checkpoints/store/，跟你项目本身的 git 仓库完全分开。Axiomate 每次准备改文件之前，会在这个 影子仓库里写一个 commit 来记录当前状态。

关键设计

影子仓库

不污染你项目自己的 .git/。你的 git log、git status 都看不到这些快照。

每个项目一条 ref

不同的工作目录写到不同的 refs/axiomate/<hash>，互不干扰。

结构化 commit subject

格式 axiomate:<messageId>:<label>，能反查到对话回合。

遵守 .gitignore

遍历工作目录时，像 node_modules/ 这种被忽略的目录不会被拍进快照。

2自动行为 无需手动

什么时候会自动拍快照

在每一次模型即将调用文件修改工具（Write / Edit / NotebookEdit）之前，Axiomate 会在影子仓库里写一个 commit。这一切默认开启，无需配置。

什么时候会自动跳过

条件行为
当前目录里文件数超过 200,000跳过这一次快照（成本太高）
单个文件大于 50 MB该文件不进 index，其他文件正常拍
当前目录是 / / ~ / 盘符根整个快照拒绝（防误伤）
跟上一次相比没有变化不写空 commit
系统里没装 git整个 checkpoint 子系统 no-op，不报错

什么时候会自动清理

Axiomate 会在你空闲 ≥1 分钟、且启动满 10 分钟之后， 在后台跑一次 prune。清理的内容包括：

  • 孤儿（orphan）：工作目录已经被你手动删掉的项目， 它们的 ref 会被删掉。
  • 过期（stale）：超过 60 天没碰过的项目 ref。
  • 超容量（size-cap）：所有快照加起来超过 5 GB 时，从最旧的开始裁。
  • 会话锚定（keep-ref）：如果某条即将被删的 ref 上 还有最近 28 天内 session 引用的 commit，会把它锚到 refs/axiomate/_keep/ 里，保住到对应 session 也过期为止。

清理至多 24 小时跑一次（用 .last_prune 标记控制），不会 重复执行。所有 prune 都是 fail-soft 的，遇到错误只记录、不抛出。

3用户命令 主动调用

/rewind — 回到某个回合之前

在 REPL 里输入 /rewind，弹出一个交互式 picker，列出本次 会话的回合 + 历史快照。每行的 label 形如 Before "<prompt>" —— 选中一行表示把状态还原到那一回合还没运行之前。

心智模型：每个 anchor 是 pre-tool 快照

Axiomate 在每个工具调用之前拍快照（跟 hermes-agent 一致）。所以 anchor 自身的 tree 描述的是"那一回合还没跑时磁盘 长什么样"。Picker 行显式说 Before "<prompt>" 避免把 anchor 误当成"git log 一条 commit"。每行右侧的 stats （file +X -Y）描述的是从前一个 anchor 到这个 anchor 之间磁盘发生了什么，跟 git log 的 commit-vs-parent 心智一致。

排序：newest-first

Picker 跟 /checkpoints list 一样，新的在最上面： 

(current)
↶ Rewind to 041da07 [71c90d4]    test.txt +1 -1   just now
↶ Rewind to before "v2 改成 v3" [041da07]   test.txt +1 -1   just now
Before v2 改成 v3                test.txt +1 -1   2m ago
Before v1 改成 v2                test.txt +1 -0   3m ago
Before 创建 test.txt 内容 v1    (no diff)        4m ago

默认光标停在第二行（最近的可还原目标）。(current) 不可选中 —— 它只是排版定位用的"现在"。g 跳到第一个可选行（跳过 (current)），G 跳到末尾。j / k 上下走。

两个标签：File vs Conversation

Picker 默认进入 File 标签。按 Tab 切到 Conversation 标签。两个标签是独立的子系统： 列表来源不同、行语义不同、chooser 动作不同、持久化存储不同。下面分别说。

File 标签：每行 = pre-tool 文件快照

只显示影响过磁盘文件的回合（AI 调用 Edit / Write / NotebookEdit / 经过 Bash 改过文件的回合）。每行 label 形如 Before "<prompt>"，选中后能真正还原文件。每行右侧的 stats（file +X -Y）描述从前一个 anchor 到这个 anchor 之间磁盘发生了什么，跟 git log 的 commit-vs-parent 心智一致。 数据来源是影子仓库（~/.axiomate/checkpoints/store/），跟 JSONL 会话日志无关。

Conversation 标签：每行 = 一组 Q/A 对话

显示当前 chain 上所有真实的 user prompt。心智模型："每 一行 = 一组完整的问答（user prompt + AI 回复）"。(current) tag 标在你正站在哪一组之上 —— 即"head" 指向的那一回合。 数据来源是 JSONL 会话日志（~/.axiomate/projects/.../*.jsonl）， 跟磁盘文件无关。

行渲染示例（发了 hi → ok → good，没 rewind 过的状态）： 

good (current)
ok
hi

reverse-chrono 顺序，最新的 Q/A 在最上面。光标自动落在 (current) 行，↑/↓ 向上向下走。没 有 File 标签那种 Before "<prompt>" 前缀 —— 这里行 文本就是 prompt 自身。

历史 / 未来：未来行 dim

Rewind 把 head 移到了某个早一点的 turn 之后，比 head 更新的回合在 picker 里仍然可见（它们在 JSONL 里还存在），但渲染成 dim 灰色 —— 提醒你它们是"未来"的、已经从 chat 区消失但还能选回去重做的节点。 

good          ← dim 灰色（未来）
ok (current)  ← 默认色（当前）
hi            ← 默认色（历史）

只要你在同一个 session 里没有跑 /clear 或 /branch， 不管 rewind 多少次，未来节点都不会丢。

Rewind 的语义：截到这一组 + 输入框回填下一条

在 picker 里选某一行 X 之后按 Enter，chooser 提供 Restore conversation。执行后：

  • chat 区：截到 X 这组对话结束为止（含 X 的 user prompt 和 AI 回复）；X 之后的 turn 从屏幕消失（数据还在 JSONL）。
  • 输入框：自动回填 X 下一条 user prompt 的内容 —— 让你"修改这次问答"再发一次。如果 X 已经是 chain tail （没有下一条），输入框清空。
  • head record：写一条到 JSONL，标记你现在站在 X 这组。

选 (current) 行本身：no-op（你已经在这了，重做无意义）。 Enter 不响应，picker 保持开着。

持久化：/exit 后再 resume 仍在原地

Conversation rewind 的 head 写到 JSONL 一条 type: "head" entry。/exit + 重启 + UI resume 这个 session 之后，落点跟你 上次 rewind 的位置一致。再次打开 /rewind，(current) tag 仍在 X 行 —— 状态没丢。

Picker 不显示什么

下列内容会被自动过滤掉，避免噪音：

  • tool_use / tool_result（在 JSONL 里 type=user 但 content 是 工具执行结果，不是真实 prompt）
  • resume 时注入的 Continue from where you left off. 续行（isMeta=true）
  • /clear / /help 等 slash 命令的输入和输出包装
  • Compact summary 占位、transcript-only 元数据
  • 纯 attachment / readonly 的"非 prompt"消息

已知限制：mid-turn 输入

AI 还在跑 turn 时你输入的 prompt（"插队"输入）不会出现 在 picker 里。这种输入被 axiomate 编码成 queue-operation audit log 写入 JSONL，但不挂在 conversation chain 上， 所以 picker 看不见。它们随父 turn 一起去留 —— 也就是说 rewind 父 turn 时 mid-turn 输入也会一起回滚（attachment 形式嵌在父 turn 里）。 未来版本可能在 picker 里显示 mid-turn 输入作为 dim 子行。

Chooser：选了行后能做什么

Picker 里选中一行按 Enter 进 chooser。chooser 根据当前标签 + 行状态 给出动作。File 标签只给 file 动作，Conversation 标签只给 conversation 动作 —— 两条轴完全脱钩。

  • File 标签 + 有 diff: Restore file / Never mind。把磁盘还原到这个 anchor 之前的状态， chat 不动。
  • File 标签 + 无 diff: 仅 Never mind （anchor.tree 跟当前磁盘一致，还原是 no-op）。
  • Conversation 标签: Restore conversation / Summarize / Never mind。仅截 chat 链 + 回填输入框，磁盘不动。

没有 "Restore file and conversation" 选项 —— 早期版本 有这个组合，但 file 是持久化（可撤）、conversation 是内存截断（不可 撤）的混合操作，半边可撤、半边不可撤造成混乱。要"两个都还原"分两步： 先 File 标签 → Restore file，再 /rewind → Tab → Conversation 标签 → Restore conversation。两条轴解耦后，conversation 也是持久化的（写 head record），不再是"不可撤"的内存截断。

Pre-rewind 安全网

每次 Restore file 之前，Axiomate 会自动拍一张 pre-rewind 安全网快照（subject 形如 axiomate:<synth-uuid>:pre-rewind:<hash>）。 它的 tree 是 rewind 操作执行之前的磁盘状态。如果 rewind 把你 重要的改动覆盖了，再开 /rewind，挑那条 ↶ Rewind to before "<X>" 行 → Restore file， 原状态就回来了。无限嵌套：撤销 rewind 操作本身也会再拍一张安全网。

Picker 行的几种形式

每个 ↶ 行末尾带一个 [7-字符短 hash] —— 这是该 anchor 自己的 git short id。普通对话回合的行不带 hash（用户的 prompt 本身就是行的标识）。所有有 anchor 的行右侧都显示 stats + 时间。

  • Before "创建 test.txt 内容 v1" —— 普通对话回合的 anchor。在 File 标签中，Before 前缀提醒你这是 "<X> 还没运行之前"的快照。
  • ↶ Before "<prompt>" [<hash>] —— 不在当前 chain 上的对话回合的 anchor。比如 conversation 已经 rewind 走了， 但当年那一回合留下的文件状态仍可还原。
  • ↶ Rewind to before "<target>" [<hash>] —— 一次 rewind 操作的 pre-rewind 安全网。选它就是撤销那次 rewind。 引号包的是这次 rewind 想还原回去的那个 turn 的 prompt。
  • ↶ Rewind to <target-hash> [<own-hash>] —— 嵌套 rewind 的安全网（撤销了一次 rewind 操作本身）。 裸 7-字符 hash 是它要还原的目标的 short id； 方括号包的 hash 是这一行自己的 short id。视觉上 bare = "我引用谁"，bracketed = "我是谁"，可以顺着 hash 一路向上 追溯整条 rewind 链。
  • ↶ Off-branch anchor [<hash>] —— 没有对话上下文 的 anchor（来自 legacy session 或外部直接 commit 进 store）。
  • (current) —— 当前 disk 状态占位，不可选中。

每行的时间

stats 列右侧统一显示时间，规则跟 /checkpoints list 一致： just now（<1 分钟）/ 5m ago（<1 小时） / 3h ago（<24 小时）/ 2026-05-24 09:42 （≥24 小时）。

每行的 CHANGES 列含义

  • file +X -Y 或 N files +X -Y —— 这个 anchor 跟它前一个 anchor 之间磁盘改动了多少（commit-vs-parent， 跟 git log 同心智）。表示"上一个回合在磁盘上写了多少"。
  • (no diff) —— anchor 跟前一个 anchor tree 一样， 即那一段时间磁盘没动。
  • 最旧那一行的 stats 描述的是"打开 axiomate 时整个 repo 长啥样" （root commit vs ∅）。这个值正确但通常很大，不要紧张 —— 选了 它是 Restore file 会把整个 repo 还原到那一刻。

反馈消息

Restore file 完成后，transcript 里会有一行： 

● File rewound to "v2 改成 v3". Use /rewind → "↶ Rewind" row to undo.

Restore conversation 不发反馈 —— 截断后的对话链已经 自显（剪掉的回合从屏幕消失，剪点的 prompt 回到输入框）。

空目录行为

在全新空目录里启动 Axiomate、让它创建第一个文件，那一回合的 anchor 是合成的根快照（空 tree）。还原到这个 anchor，会把 Axiomate 之后创建的文件全部删掉。

Slash 命令在 picker 里不出现

/checkpoints status、/help、/clear 这些不属于对话回合 —— 它们是你在 REPL 跑的工具调用，picker 自动 过滤掉。"rewind 到 /checkpoints status 之前" 和 rewind 到再前一个真实回合是同一个状态。

命令作用

命令行版本

同样的子命令也可以从 axiomate 的命令行调用，不进 TUI： 

axiomate checkpoints status
axiomate checkpoints status --rows 50
axiomate checkpoints list
axiomate checkpoints list --rows 50
axiomate checkpoints prune --force
axiomate checkpoints prune --retention-days 90 --max-size-mb 10000
axiomate checkpoints prune --keep-orphans --force
axiomate checkpoints clear --force

每个子命令都拒绝多余的位置参数（输出 too many arguments）， 不会静默忽略。clear 在命令行端必须显式 --force， 因为它绕过了斜杠版本里的交互确认。

控制 status / list 显示几行

status 和 list 渲染的表格默认最多 100 行。想多看（或少看）几行，传一个 1..5000 之间的整数。斜杠命令直接用位置参数 （/checkpoints status 50、/checkpoints list 50）； 命令行用 --rows N，跟其他 CLI flag 风格一致 （axiomate checkpoints status --rows 50）。 非整数或超界会被拒绝。

想让覆盖值长期生效，把 checkpointsStatusRows 写到 Settings 里（叫 Checkpoints rows shown，在 30/50/100/200/500/1000/2000/5000 之间循环），或者 /config checkpointsStatusRows N，再或者直接编辑 ~/.axiomate.json。每次调用的位置参数（斜杠命令） 或 --rows N（CLI）永远优先于 config 里的值。

--keep-orphans — 保留临时缺失的工作目录 ref

默认情况下，prune 会把工作目录已经从磁盘上消失的项目 ref 直接删掉。 但如果你只是临时拔了外置盘、改了路径名打算再改回来、或者计划晚点 重新 clone 到原路径，可以加 --keep-orphans（CLI）或 keep-orphans（斜杠命令）：这次 prune 跳过孤儿那一遍， stale 和 size-cap 照常跑。/checkpoints prune keep-orphans 在 REPL 里也支持。命中这个安全阀的 ref 数量会以 Orphan refs skipped: N 的形式出现在输出里。

典型输出

/checkpoints status

Checkpoint base: C:\Users\you\.axiomate\checkpoints
Total size:      284.3 MB
  store/         284.3 MB
Projects:        7

  WORKDIR                                                        COMMITS    LAST TOUCH  STATE
  C:\public\workspace\axiomate                                       142          5m ago  live
  C:\public\workspace\hermes-agent                                    87          2h ago  live
  C:\Users\you\Desktop\old-prototype                                   3         18d ago  orphan

  ok 96, no-changes 3, skipped 0, failed 1
  duration p50 142ms, p95 380ms

WORKDIR / COMMITS / LAST TOUCH / STATE 各列含义：项目工作目录、shadow store 里这个项目的 anchor 数、最近一次 anchor 的时间（自适应：<24h 显示 5m ago / 3h ago，>24h 显示绝对日期）、状态（live / orphan，orphan 表示 workdir 不存在了）。

Snapshot metrics 块是最近最多 100 次 snapshot 尝试的滚动窗口 （last N 那个数字会随快照累计涨到 100）。窗口数据写在 ~/.axiomate/checkpoints/metrics.jsonl 里，重启不丢。 第一次 snapshot 之前这块整段不显示； p50 / p95 至少要 2 个 ok 样本才会 展示数字，否则显示 —。

/checkpoints list

Checkpoints for C:\public\test:

  WHEN     ID       CHANGES         TURN
  5m ago   8da1461  test.txt +1 -1  Before "v2 改成 v3"
  4m ago   f72c710  test.txt +1 -0  Before "v1 改成 v2"
  3m ago   c143625  (no diff)       Before "创建 test.txt 内容 v1"

Roll back to a turn with /rewind (interactive selector).

列含义：

  • WHEN —— anchor 的时间戳，自适应格式（<1m = just now，<1h = Nm ago，<24h = Nh ago，≥24h = 2026-05-24 09:42）。
  • ID —— git short hash（7 字符）。可以直接喂给外部 git 命令，例如 git --git-dir=~/.axiomate/checkpoints/store show 8da1461。 完整 hash + message UUID 在 --debug 日志里。
  • CHANGES —— 这个 anchor 跟它前一个 anchor 之间磁盘 发生了什么（commit-vs-parent，跟 git log 同心智）。 file +X -Y 单文件变更，N files +X -Y 多 文件，(no diff) 跟前一 anchor tree 一致 / 没有 parent。
  • TURN —— 这一行的 label，说明是哪个回合的 pre-tool 快照。形如 Before "<prompt>" 或 Undo rewind to before "<target>"（pre-rewind 安全网）。

所有 anchor 都会显示，包括跟当前 disk 一致的（(no diff)）— 历史回执是真的，不该被隐藏。要还原选 /rewind 里相应的行 → Restore file。

5失败模式 兜底

Checkpoint 子系统的核心契约是fail-open — 任何一次 快照失败都不能阻塞主流程。这意味着：

  • 系统里没装 git → 整个子系统 no-op，不报错。
  • git 命令瞬时失败（EBUSY、超时） → 这一次跳过，下一回合自动重试。
  • 工作目录有超过 200,000 个文件 → 跳过，避免 git add 卡死。
  • 磁盘满 / 权限不足 → 错误进 logForDebugging，对话继续。

跨工作目录的边界情况

如果你 /resume 了一个旧 session，但当前工作目录跟当时 不一样，/rewind 之前 Axiomate 会用一行系统消息提示你：

  • reachable：快照在当前 ref 里，可以正常 rewind。
  • reachable-other-worktree：快照存在，但属于另一个 工作目录。提示你 cd 过去。
  • unreachable：快照已被 prune 删掉，rewind 会失败。

清理时的孤儿提醒

/checkpoints status 末尾可能会附一行类似： 

Note: 3 snapshots from 1 orphan workdir will be discarded on next prune.

如果你看到这行、并且那个工作目录的 commit 还有用（比如刚把外置硬盘 拔了），可以把工作目录复原到原路径，下一次 prune 会重新认它为 live。

6FAQ

会污染我的 git 仓库吗？

不会。所有 commit 都写到 ~/.axiomate/checkpoints/store/， 跟你项目自己的 .git/ 完全独立。git status、 git log、git push 全都看不到这些 commit。

占多少磁盘？

硬上限 5 GB（可配）。git 自己会做对象去重和 zlib 压缩，所以同一个 文件改 100 次，实际只多存了几次 delta。日常使用一般在几百 MB 到 1-2 GB。

能跨机器同步吗？

不能。Checkpoints 是本机快照，没设计跨机器的同步、加密、 签名机制。换机器就当回到没历史的状态。

能导出 / 导入吗？

暂时没有 checkpoints export / import。 影子仓库本身是一个标准的 bare git repo，懂 git 的用户可以直接用 git --git-dir=~/.axiomate/checkpoints/store ... 检查它。

怎么彻底关掉？

在 Settings 里把 fileCheckpointingEnabled 设成 false， 或者环境变量 AXIOMATE_CODE_DISABLE_FILE_CHECKPOINTING=1。 之前已经攒下的快照不会被自动清理，需要手动 /checkpoints clear。

/rewind 报错说 commit 找不到，怎么办？

多半是那个回合的 commit 已经被 prune 掉了（超过 60 天 + 当前 session 没引用）。这是预期行为。如果是关键节点，请另行保留一份相关工作目录 的输出，不要只依赖 checkpoint。
