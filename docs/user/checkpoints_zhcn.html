Checkpoints — 文件快照与对话回放

Axiomate 的 /rewind 在两条独立轴上工作。File 轴： 每次准备改文件之前在后台拍一张工作目录的快照，放进独立的 git 影子仓库； 你可以把磁盘还原到任何一个历史回合。Conversation 轴： 在 JSONL 会话日志的基础上，让你回到某一组 Q/A 对话的位置，连同输入框 的回填一起送你"重做"那个回合。两条轴 100% 脱钩 — 文件不动对话、对话 不动文件。整个系统对你近乎透明 — 平时几乎感觉不到，关键时刻它替你兜底。

1它怎么工作

Checkpoints 是一个独立的 git 仓库，住在 ~/.axiomate/checkpoints/store/，跟你项目本身的 git 仓库完全分开。Axiomate 每次准备改文件之前，会在这个 影子仓库里写一个 commit 来记录当前状态。

关键设计

影子仓库

不污染你项目自己的 .git/。你的 git log、git status 都看不到这些快照。

每个项目一条 ref

不同的工作目录写到不同的 refs/axiomate/<hash>，互不干扰。

结构化 commit subject

格式 axiomate:<messageId>:<label>，能反查到对话回合。

遵守 .gitignore

遍历工作目录时，用户 .gitignore 会生效；Axiomate 只拍最终未被忽略的普通文件。

支持嵌套 Git 仓库

子目录里有自己的 .git 时，普通文件按当前磁盘内容保存，.git/ 元数据本身不保存。

跟同类系统比的优势

Axiomate 的 file checkpointing 借鉴了 hermes-agent 的 pre-tool workdir 快照模型 —— 这个底层已经很稳。在它之上，axiomate 做了几 个 hermes 没有的事：

双轴 rewind + 双游标

Hermes 只做文件回滚。Axiomate 加了独立的对话回放轴 —— picker 里按 Tab 切到 Conversation 标签，回到任意 一组 Q/A 之上，输入框自动回填下一组 prompt 让你重做。File 和 Conversation 完全脱钩。Selection 游标 (▶) 与 Head 游标 ((current)) 概念分离，视觉光标不等于会话 状态，比 hermes "光标=状态" 的合并模型更明确。

对话 rewind 持久化 + 未来节点可见

Conversation rewind 的 head 状态写进 JSONL（一条 type:"head" entry）。/exit 后再用 UI resume 进同一个 session，落点跟你 rewind 时一致 —— 不像传统的 内存截断那样退出就丢。Rewind 之后比 head 更新的对话回合在 picker 里仍以 dim 灰色显示（JSONL 没删），随时可再选回去。 "rewind 之后能 redo" 是 axiomate 的心智核心。

每次 rewind 自动拍 pre-rewind 安全网

Hermes 的 rewind 是单向操作 —— 一旦回退就丢失更晚的状态。 Axiomate 在每次 rewind 之前先把当前工作区拍一张安全网快照 再回退；后悔了点 picker 里那条 ↶ 行就能撤销 rewind 自身。 可无限嵌套，rewind 套 rewind 也不会丢。

嵌套 rewind 用 hash chain 显示

连续做了几次 rewind 之后，picker 里这些 ↶ 行会以 ↶ Rewind to <target-hash> [own-hash] 的形式串起来 —— 你能一眼看出"这条 rewind 是从哪条又 rewind 回去的"。层级关系在文本里明示，不靠记忆。

CHANGES 列跟 git log 同心智（event-aligned）

Picker / list / status 三处的 CHANGES 列描述本回合 写了什么（anchor[i] vs anchor[i-1]），跟你看 git log 一致。Hermes 原始的 commit-vs-parent 模型存在 off-by-one： 某行的 CHANGES 实际是下一行那一回合改的， 初看心智上对不上。

更激进的默认值（适配现代开发机）

默认快照上限 1000（可通过 /config 调整） / 保留 30 天 / 1GB 池 / 单文件 50MB / 工作目 录最多 200,000 个文件（可通过 /config 调整） / 列表最多 500 行。Hermes 默认 7 天 保留，跨 sprint 项目易丢；axiomate 30 天保留 + 1GB 池优先给近期工作 保留可回滚空间，同时避免默认磁盘预算过重。

UX 反超：newest-first + 自适应时间

Picker 默认最新在顶（hermes 无对照），刚操作的 回合一打开就在视线里。时间格式自适应：刚才 → just now，几分钟前 → 5m ago，跨天 → 绝对日期。

完整的 /checkpoints CLI

除了 list，axiomate 还有 status / prune / clear 子命令 + 命令行版本。--keep-orphans 安全阀允许 外置盘断开时不丢快照。

2自动行为 无需手动

什么时候会自动拍快照

在每一次模型即将调用文件修改工具（Write / Edit / NotebookEdit）之前，Axiomate 会在影子仓库里写一个 commit。这一切默认开启，无需配置。

什么时候会自动跳过

条件行为
当前目录里文件数超过 checkpointsMaxFiles（默认 200,000）跳过这一次快照（成本太高）
单个文件大于 50 MB该文件不进 index，其他文件正常拍
当前目录是 / / ~ / 盘符根整个快照拒绝（防误伤）
跟上一次相比没有变化不写空 commit
系统里没装 git整个 checkpoint 子系统 no-op，不报错

什么时候会自动清理

Axiomate 会在你空闲 ≥1 分钟、且启动满 10 分钟之后， 在后台跑一次 prune。清理的内容包括：

  • 孤儿（orphan）：工作目录已经被你手动删掉的项目， 它们的 ref 会被删掉。
  • 过期（stale）：超过 30 天没碰过的项目 ref。
  • 超容量（size-cap）：所有快照加起来超过 1 GB 时，从最旧的开始裁。
  • 会话锚定（keep-ref）：如果某条即将被删的 ref 上 还有最近 28 天内 session 引用的 commit，会把它锚到 refs/axiomate/_keep/ 里，保住到对应 session 也过期为止。

清理至多 24 小时跑一次（用 .last_prune 标记控制），不会 重复执行。所有 prune 都是 fail-soft 的，遇到错误只记录、不抛出。

3用户命令 主动调用

/rewind — 回到某个回合之前

在 REPL 里输入 /rewind，弹出一个交互式 picker，列出本次 会话的回合 + 历史快照。每行的 label 形如 Before "<prompt>" —— 选中一行表示把状态还原到那一回合还没运行之前。

两个标签：File vs Conversation

Picker 默认进入 File 标签。按 Tab 切到 Conversation 标签。两个标签是独立的子系统： 列表来源不同、行语义不同、chooser 动作不同、持久化存储不同。下面 每个标签自成一节，独立讲述。

File 标签 — 文件回滚

心智模型：每个 anchor 是 pre-tool 快照

Axiomate 在每个工具调用之前拍快照（跟 hermes-agent 一致）。所以 anchor 自身的 tree 描述的是"那一回合还没跑时磁盘 长什么样"。Picker 行显式说 Before "<prompt>" 避免把 anchor 误当成"git log 一条 commit"。每行右侧的 stats （file +X -Y）描述的是从前一个 anchor 到这个 anchor 之间磁盘发生了什么，跟 git log 的 commit-vs-parent 心智一致。 数据来源是影子仓库（~/.axiomate/checkpoints/store/）， 跟 JSONL 会话日志无关。

只显示影响过磁盘文件的回合（AI 调用 Edit / Write / NotebookEdit / 经过 Bash 改过文件的回合），纯聊天 / 只读问答不出现。

排序：newest-first

Picker 跟 /checkpoints list 一样，新的在最上面： 

(current)
↶ Rewind to 041da07 [71c90d4]    test.txt +1 -1   just now
↶ Rewind to before "v2 改成 v3" [041da07]   test.txt +1 -1   just now
Before v2 改成 v3                test.txt +1 -1   2m ago
Before v1 改成 v2                test.txt +1 -1   3m ago
Before 创建 test.txt 内容 v1    test.txt +1 -0   4m ago

默认光标停在第二行（最近的可还原目标）。(current) 不可选中 —— 它只是排版定位用的"现在"。g 跳到第一个可选行（跳过 (current)），G 跳到末尾。j / k 上下走。

Chooser：选了行后能做什么

  • 有 diff: Restore file / Never mind。把磁盘还原到这个 anchor 之前的状态， chat 不动。
  • 无 diff: 仅 Never mind （anchor.tree 跟当前磁盘一致，还原是 no-op）。

Pre-rewind 安全网

每次 Restore file 之前，Axiomate 会自动拍一张 pre-rewind 安全网快照（subject 形如 axiomate:<synth-uuid>:pre-rewind:<hash>）。 它的 tree 是 rewind 操作执行之前的磁盘状态。如果 rewind 把你 重要的改动覆盖了，再开 /rewind，挑那条 ↶ Rewind to before "<X>" 行 → Restore file， 原状态就回来了。无限嵌套：撤销 rewind 操作本身也会再拍一张安全网。

Picker 行的几种形式

每个 ↶ 行末尾带一个 [7-字符短 hash] —— 这是该 anchor 自己的 git short id。普通对话回合的行不带 hash（用户的 prompt 本身就是行的标识）。所有有 anchor 的行右侧都显示 stats + 时间。

  • Before "创建 test.txt 内容 v1" —— 普通对话回合的 anchor。在 File 标签中，Before 前缀提醒你这是 "<X> 还没运行之前"的快照。
  • ↶ Before "<prompt>" [<hash>] —— 不在当前 chain 上的对话回合的 anchor。比如 conversation 已经 rewind 走了， 但当年那一回合留下的文件状态仍可还原。
  • ↶ Rewind to before "<target>" [<hash>] —— 一次 rewind 操作的 pre-rewind 安全网。选它就是撤销那次 rewind。 引号包的是这次 rewind 想还原回去的那个 turn 的 prompt。
  • ↶ Rewind to <target-hash> [<own-hash>] —— 嵌套 rewind 的安全网（撤销了一次 rewind 操作本身）。 裸 7-字符 hash 是它要还原的目标的 short id； 方括号包的 hash 是这一行自己的 short id。视觉上 bare = "我引用谁"，bracketed = "我是谁"，可以顺着 hash 一路向上 追溯整条 rewind 链。
  • ↶ Off-branch anchor [<hash>] —— 没有对话上下文 的 anchor（来自 legacy session 或外部直接 commit 进 store）。
  • (current) —— 当前 disk 状态占位，不可选中。

每行的时间

stats 列右侧统一显示时间，规则跟 /checkpoints list 一致： just now（<1 分钟）/ 5m ago（<1 小时） / 3h ago（<24 小时）/ 2026-05-24 09:42 （≥24 小时）。

每行的 CHANGES 列含义

每行的 stats 描述这一行 turn 自己写了什么（不是 上一行）。算法：

  • 最新那一行：anchor.tree vs 当前磁盘（"最新 turn 跑完后磁盘当前长这样"）。如果你在 axiomate 之外手动改了 文件，这一行的 stats 会跟着变 —— 它反映 anchor.tree 到当前 disk 的全部差异。
  • 其它行：anchor.tree vs 上一个 anchor 的 tree（newer 那个）。等价于"这一行的 turn 跑完后产生了下一个 anchor 的状态"，frozen 不变。
  • file +X -Y / N files +X -Y 单/多 文件变更。(no diff) 这一行 turn 没改文件 （比如纯聊天 / 只读问答）。

反馈消息

Restore file 完成后，transcript 里会有一行： 

● File rewound to "v2 改成 v3". Use /rewind → "↶ Rewind" row to undo.

Restore conversation 不发反馈 —— 截断后的对话链已经 自显（剪掉的回合从屏幕消失，剪点的 prompt 回到输入框）。

空目录行为

在全新空目录里启动 Axiomate、让它创建第一个文件，那一回合的 anchor 是合成的根快照（空 tree）。还原到这个 anchor，会把 Axiomate 之后创建的文件全部删掉。

Slash 命令在 picker 里不出现

/checkpoints status、/help、/clear 这些不属于对话回合 —— 它们是你在 REPL 跑的工具调用，picker 自动 过滤掉。"rewind 到 /checkpoints status 之前" 和 rewind 到再前一个真实回合是同一个状态。

Conversation 标签 — 对话回放

心智模型：每行 = 一组 Q/A 对话

显示当前 chain 上所有真实的 user prompt。每行 = 一组 完整的问答（user prompt + AI 回复）。(current) tag 标在你 正站在哪一组之上 —— 即"head" 指向的那一回合。 数据来源是 JSONL 会话日志（~/.axiomate/projects/.../*.jsonl）， 跟磁盘文件无关。

行渲染示例（发了 hi → ok → good，没 rewind 过的状态）： 

good (current)
ok
hi

reverse-chrono 顺序，最新的 Q/A 在最上面。光标自动落在 (current) 行，↑/↓ 向上向下走。没 有 File 标签那种 Before "<prompt>" 前缀 —— 这里行 文本就是 prompt 自身。

历史 / 未来：未来行 dim

Rewind 把 head 移到了某个早一点的 turn 之后，比 head 更新的回合在 picker 里仍然可见（它们在 JSONL 里还存在），但渲染成 dim 灰色 —— 提醒你它们是"未来"的、已经从 chat 区消失但还能选回去重做的节点。 

good          ← dim 灰色（未来）
ok (current)  ← 默认色（当前）
hi            ← 默认色（历史）

只要你在同一个 session 里没有跑 /clear 或 /branch， 不管 rewind 多少次，未来节点都不会丢。

两种游标：selection 和 head

Conversation 标签里同时有两种游标，别混淆：

  • Selection 游标（左侧 ▶）—— 你 ↑/↓ 移动的视觉光标，决定按 Enter 时选中 哪一行。打开 picker 时自动落在 head 行。
  • Head 游标（行末 (current) tag）—— 会话当前的状态指针，"我现在站在哪一组对话之上"。head 不会随 ↑/↓ 移动，只在你做 Restore conversation 之后才换位置。

两种游标可以重叠：刚打开 picker 时 selection 也在 head 行，那一行 同时有 ▶ 和 (current)。↑/↓ 移动后 selection 跑去别处，head 仍留在 (current) 行 —— 这两条信息是独立的。

Rewind 的语义：截到这一组 + 输入框回填下一条

在 picker 里选某一行 X 之后按 Enter，chooser 提供 Restore conversation。执行后：

  • chat 区：截到 X 这组对话结束为止（含 X 的 user prompt 和 AI 回复）；X 之后的 turn 从屏幕消失（数据还在 JSONL）。
  • 输入框：自动回填 X 下一条 user prompt 的内容 —— 让你"修改这次问答"再发一次。如果 X 已经是 chain tail （没有下一条），输入框清空。
  • head record：写一条到 JSONL，标记你现在站在 X 这组。

选 (current) 行本身：no-op（你已经在这了，重做无意义）。 Enter 不响应，picker 保持开着。

持久化：/exit 后再 resume 仍在原地

Conversation rewind 的 head 写到 JSONL 一条 type: "head" entry。/exit + 重启 + UI resume 这个 session 之后，落点跟你 上次 rewind 的位置一致。再次打开 /rewind，(current) tag 仍在 X 行 —— 状态没丢。

Picker 不显示什么

下列内容会被自动过滤掉，避免噪音：

  • tool_use / tool_result（在 JSONL 里 type=user 但 content 是 工具执行结果，不是真实 prompt）
  • resume 时注入的 Continue from where you left off. 续行（isMeta=true）
  • /clear / /help 等 slash 命令的输入和输出包装
  • Compact summary 占位、transcript-only 元数据
  • 纯 attachment / readonly 的"非 prompt"消息

TUI 斜杠命令 (/checkpoints)

在 axiomate REPL 里直接输入。子命令清单：

命令作用
/checkpoints
/checkpoints status
/checkpoints status N		 显示影子仓库总大小、每个项目的快照数和最近活动时间。可选位置参数 N 控制最多显示几行（默认 50，范围 1..500）。
/checkpoints list
/checkpoints list N		 列出当前项目的快照，按时间倒序。可选位置参数 N 控制最多显示几行（默认 50，范围 1..500）。
/checkpoints prune
/checkpoints prune force
/checkpoints prune keep-orphans		 立即跑一次清理。默认带 24h 节流；加 force 绕过。加 keep-orphans 跳过孤儿（工作目录消失的）那一遍，stale 和 size-cap 仍然跑。斜杠命令的参数是位置子词（不带 --）；拼错的词（如 frce）会被拒绝并报用法错误。命令行端保留 POSIX 风格 flag — 见下方。
/checkpoints clear 清空整个影子仓库（所有项目的所有快照）。不可恢复。

命令行 (CLI)

同样的子命令也可以从 axiomate 的命令行调用，不进 TUI —— 用于脚本、CI 集成、或快速查看 / 清理时不想拉起整个 REPL。 

axiomate checkpoints status
axiomate checkpoints status --rows 50
axiomate checkpoints list
axiomate checkpoints list --rows 50
axiomate checkpoints prune --force
axiomate checkpoints prune --retention-days 90 --max-size-mb 10000
axiomate checkpoints prune --keep-orphans --force
axiomate checkpoints clear --force

每个子命令都拒绝多余的位置参数（输出 too many arguments）， 不会静默忽略。clear 在命令行端必须显式 --force， 因为它绕过了斜杠版本里的交互确认。

控制 status / list 显示几行

status 和 list 渲染的表格默认最多 50 行。想多看（或少看）几行，传一个 1..500 之间的整数。斜杠命令直接用位置参数 （/checkpoints status 50、/checkpoints list 50）； 命令行用 --rows N，跟其他 CLI flag 风格一致 （axiomate checkpoints status --rows 50）。 非整数或超界会被拒绝。

想让覆盖值长期生效，把 checkpointsStatusRows 写到 Settings 里（叫 Checkpoints rows shown，在 10/20/50/100/200/500 之间循环），或者 /config checkpointsStatusRows N，再或者直接编辑 ~/.axiomate.json。每次调用的位置参数（斜杠命令） 或 --rows N（CLI）永远优先于 config 里的值。

--keep-orphans — 保留临时缺失的工作目录 ref

默认情况下，prune 会把工作目录已经从磁盘上消失的项目 ref 直接删掉。 但如果你只是临时拔了外置盘、改了路径名打算再改回来、或者计划晚点 重新 clone 到原路径，可以加 --keep-orphans（CLI）或 keep-orphans（斜杠命令）：这次 prune 跳过孤儿那一遍， stale 和 size-cap 照常跑。/checkpoints prune keep-orphans 在 REPL 里也支持。命中这个安全阀的 ref 数量会以 Orphan refs skipped: N 的形式出现在输出里。

典型输出

/checkpoints status

Checkpoint base: C:\Users\you\.axiomate\checkpoints
Total size:      284.3 MB
  store/         284.3 MB
Projects:        7

  WORKDIR                                                        COMMITS    LAST TOUCH  STATE
  C:\public\workspace\axiomate                                       142          5m ago  live
  C:\public\workspace\hermes-agent                                    87          2h ago  live
  C:\Users\you\Desktop\old-prototype                                   3         18d ago  orphan

Note: 3 snapshots from 1 orphan workdir will be discarded on next prune.

Snapshot metrics (last 100):
  ok 96, no-changes 3, skipped 0, failed 1
  duration p50 142ms, p95 380ms

WORKDIR / COMMITS / LAST TOUCH / STATE 各列含义：项目工作目录、shadow store 里这个项目的 anchor 数、最近一次 anchor 的时间（自适应：<24h 显示 5m ago / 3h ago，>24h 显示绝对日期）、状态（live / orphan，orphan 表示 workdir 不存在了）。

Snapshot metrics 块是最近最多 100 次 snapshot 尝试的滚动窗口 （last N 那个数字会随快照累计涨到 100）。窗口数据写在 ~/.axiomate/checkpoints/metrics.jsonl 里，重启不丢。 第一次 snapshot 之前这块整段不显示； p50 / p95 至少要 2 个 ok 样本才会 展示数字，否则显示 —。

/checkpoints list

Checkpoints for C:\public\test:

  WHEN      ID       CHANGES         TURN
  just now  bd10d75  test.txt +1 -1  Before "v1 改成 v2"
  just now  49f18dc  test.txt +1 -0  Before "创建 test.txt，内容 v1"

Roll back to a turn with /rewind (interactive selector).

列含义：

  • WHEN —— anchor 的时间戳，自适应格式（<1m = just now，<1h = Nm ago，<24h = Nh ago，≥24h = 2026-05-24 09:42）。
  • ID —— git short hash（7 字符）。可以直接喂给外部 git 命令，例如 git --git-dir=~/.axiomate/checkpoints/store show 8da1461。 完整 hash + message UUID 在 --debug 日志里。
  • CHANGES —— 跟 picker 同源："这一行 turn 写了 什么"。最新那一行用 anchor-vs-disk，其它行用相邻 anchor 间的 diff。file +X -Y / N files +X -Y 单/多 文件变更，(no diff) 这次 turn 没改文件。
  • TURN —— 这一行的 label，说明是哪个回合的 pre-tool 快照。形如 Before "<prompt>" 或 Undo rewind to before "<target>"（pre-rewind 安全网）。

所有 anchor 都会显示。要还原选 /rewind 里相应的行 → Restore file。

5失败模式 兜底

Checkpoint 子系统的核心契约是fail-open — 任何一次 快照失败都不能阻塞主流程。这意味着：

  • 系统里没装 git → 整个子系统 no-op，不报错。
  • git 命令失败（EBUSY、超时、权限问题、文件在扫描和写入 index 之间变化等） → 这一次跳过，下一回合自动重试。
  • 工作目录文件数超过 checkpointsMaxFiles（默认 200,000） → 跳过，避免快照扫描成本过高；当前运行内会缓存这个结果，配置变化或重启后重新检测。
  • 磁盘满 / 权限不足 → 错误进 logForDebugging，对话继续。

transient-error 是什么

transient-error 表示 Axiomate 尝试创建快照时，某一步 git 命令失败了。它不会让本轮工具调用失败，只是本次没有留下可 rewind 的文件快照。下一次写文件前会再试。

如果使用 axiomate --debug 启动，可以在 debug log 里搜索 [checkpoint-diagnostics]。这类日志只在 debug 模式下输出， 正常启动时不会做额外的大量诊断扫描。常见失败点包括目录边界判断用的 check-ignore、文件发现用的 ls-files、 写入快照 index 的 update-index、write-tree 和 commit-tree。这些错误通常表示文件系统正在变化、文件被锁、 权限不足、磁盘不足，或 Git 写入 shadow store 超时。

嵌套 Git 仓库怎么处理

简化心智模型：Axiomate 把嵌套 Git 仓库当成“可进入的普通文件夹”来拍， 只跳过里面的 VCS 元数据。子仓库当前磁盘上的普通文件会保存；进入子仓库后， .gitignore 范围从子仓库自己重新开始。

处理规则

  • 不会保存 .git/、.git、.hg/、.svn/ 这些 VCS 元数据。
  • 会保存子仓库里的普通文件，包括 dirty、staged、untracked 文件的当前磁盘内容。
  • 父级 .gitignore 只负责“能不能走到子仓库目录入口”；一旦进入子仓库，内部文件只看子仓库范围的 .gitignore。

父级 .gitignore 的边界例子

  • child/：忽略 child 目录本身，整个子仓库跳过。
  • child/* 且 child/.git 存在：child 目录入口没被忽略，会进入子仓库；进入后按子仓库自己的 .gitignore 重新判断文件。
  • vendor/* 且子仓库在 vendor/child/.git：vendor/child 这个入口已经被父级 .gitignore 忽略，所以跳过。
  • 子仓库没有自己的 .gitignore：不会继承父级 .gitignore，只受 Axiomate 的极小默认排除影响。

实现细节

  • Axiomate 不从父目录执行 git add -A，所以嵌套仓库不会被保存成 gitlink。
  • 目录入口用 Git 的 check-ignore 判断；最终文件用独立空 index 下的 ls-files --others --exclude-standard 枚举。
  • 写入 checkpoint shadow index 时使用 update-index，不读取用户仓库自己的 index 状态。

性能对比

这套算法比直接跑父仓库 git add -A 多做了“寻找嵌套仓库边界”和 “把子仓库当普通文件拍”的工作。Windows warm median 实测如下：

场景 Axiomate 对照 回退
普通 5k 文件 2122ms git add -A 1991ms 1.07x，多 131ms
大量被 .gitignore 忽略的文件/目录 2292ms git add -A 2088ms 1.10x，多 204ms
8 个嵌套仓库，每个 500 文件 3022ms 每个仓库分别 fresh-index git add -A 后合并，1037ms 2.92x，多 1985ms

读法：比例只说明相对回退，绝对时间更能说明体感。普通项目和大量 ignored 文件场景只多约 0.1-0.2 秒；嵌套仓库场景多约 2 秒，因为 Git 的父仓库语义 倾向于把子仓库当 gitlink，而 Axiomate 必须进入子仓库并保存里面的普通文件。 这是为了保证 rewind 能恢复嵌套仓库里的当前磁盘内容。

默认排除只有 Axiomate 自己管理的一小组：.git/、 .hg/、.svn/、node_modules/、 .DS_Store、Thumbs.db、desktop.ini。 .env、.env.local、.env.*.local、 *.log、.next/、build/、 bin/、obj/ 和 .vs/ 不会被默认排除， 除非项目自己的 .gitignore 明确忽略。项目仓库自己的 .git/info/exclude 和全局 gitignore 不作为 checkpoint 的用户规则； Axiomate 只使用 shadow store 的 info/exclude 放这组极小默认值。

too-many-files 和文件数看起来不一致

too-many-files 检查的是本次 checkpoint 的工作目录， 不一定是你眼前打开的那个子文件夹。例如从 D:\Temp 或 D:\AI 启动 Axiomate，即使你只让它改其中一个 30 个文件的 子目录，checkpoint 仍可能先按启动工作目录做保护性文件数检查。 看到这类问题时，用 --debug 搜索 [checkpoint-diagnostics] 或查看 /checkpoints status 里的 WORKDIR，先确认实际被扫描的是哪个目录。

跨工作目录的边界情况

如果你 /resume 了一个旧 session，但当前工作目录跟当时 不一样，/rewind 之前 Axiomate 会用一行系统消息提示你：

  • reachable：快照在当前 ref 里，可以正常 rewind。
  • reachable-other-worktree：快照存在，但属于另一个 工作目录。提示你 cd 过去。
  • unreachable：快照已被 prune 删掉，rewind 会失败。

清理时的孤儿提醒

/checkpoints status 末尾可能会附一行类似： 

Note: 3 snapshots from 1 orphan workdir will be discarded on next prune.

如果你看到这行、并且那个工作目录的 commit 还有用（比如刚把外置硬盘 拔了），可以把工作目录复原到原路径，下一次 prune 会重新认它为 live。

6FAQ

会污染我的 git 仓库吗？

不会。所有 commit 都写到 ~/.axiomate/checkpoints/store/， 跟你项目自己的 .git/ 完全独立。git status、 git log、git push 全都看不到这些 commit。

占多少磁盘？

硬上限 1 GB（prune 命令可临时覆盖）。git 自己会做对象去重和 zlib 压缩，所以同一个 文件改 100 次，实际只多存了几次 delta。日常使用一般在几百 MB 到 1-2 GB。

能跨机器同步吗？

不能。Checkpoints 是本机快照，没设计跨机器的同步、加密、 签名机制。换机器就当回到没历史的状态。

能导出 / 导入吗？

暂时没有 checkpoints export / import。 影子仓库本身是一个标准的 bare git repo，懂 git 的用户可以直接用 git --git-dir=~/.axiomate/checkpoints/store ... 检查它。

怎么彻底关掉？

在 Settings 里把 fileCheckpointingEnabled 设成 false， 或者环境变量 AXIOMATE_CODE_DISABLE_FILE_CHECKPOINTING=1。 之前已经攒下的快照不会被自动清理，需要手动 /checkpoints clear。

/rewind 报错说 commit 找不到，怎么办？

多半是那个回合的 commit 已经被 prune 掉了（超过 30 天 + 当前 session 没引用）。这是预期行为。如果是关键节点，请另行保留一份相关工作目录 的输出，不要只依赖 checkpoint。

父级 .gitignore 会影响子仓库吗？

只影响“能不能到达子仓库目录”这个边界，不影响已经进入后的子仓库内部文件。 进入带 .git 的子目录后，Axiomate 会从该子目录重新评估 ignore 规则，行为等价于在子仓库里单独看它自己的 .gitignore。 child/ 会让 child 整体跳过； child/* 不会挡住 child/.git 这个边界； vendor/* 会挡住 vendor/child/.git 这个更深一层的边界。
