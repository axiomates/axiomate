Context / Memory Gap Report

Claude Code Context/Memory Gap Report: 2.1.88 -> 2.1.158

本报告只聚焦 Claude Code 与 Axiomate 的 context / memory 系统。按照当前维护判断，Axiomate 在这一层可视为 Claude Code CLI 2.1.88 基线：fork 之后基本没有重写 memory 架构，主要是品牌、配置、provider 和局部 UI/稳定性改动。

结论：下一版本 Axiomate 的首要任务不是再引入一套 Hermes 式 RAG memory provider，而是补齐 Claude Code 在 2.1.88 之后公开产品化出来的上下文生命周期：可解释、可审计、可压缩、可分层、可按路径懒加载、可隔离到 subagent。

Feature Playbook

这些功能到底是什么、怎么用、运行时发生什么

下面按“用户要解决的问题 -> 操作入口 -> 运行时机制 -> Axiomate 落地层”写。读这部分的目标不是背 Claude 的功能名，而是明确下一轮 Axiomate 应该补哪些可见行为、状态机和审计面。

1. Path-scoped rules：把指令从全局噪音变成按文件触发

官方 Claude Code 用 .claude/rules/ 承载项目规则；带 paths: frontmatter 的规则只在匹配文件被读取时进入 context。

P0
解决什么问题单个 CLAUDE.md 越写越大后，每次启动都把测试、前端、API、迁移、安全规则全部塞进 context。模型会被无关规则干扰，启动 token 也被浪费。Path-scoped rules 的核心价值是“只在相关路径工作时加载相关规则”。
用户怎么用把规则拆到 .claude/rules/testing.md、.claude/rules/frontend/react.md、.claude/rules/api.md。全局适用的文件不写 paths；局部规则写 paths: ["src/api/**/*.ts"] 或测试 glob。
什么时候用monorepo、前后端规范不同、某些测试命令只适用于子目录、生成代码/迁移代码有特殊约束、团队规则互相冲突时。
运行时发生什么：启动时扫描规则目录；无 paths 的规则进入启动 context；有 paths 的规则登记在 matcher 中但暂不占用 context。Claude 调用 Read 读取匹配文件时，loader 把规则内容注入当前消息历史，并发出 loaded notice。compact 后这类规则会随消息历史被总结掉，直到下次再读匹配文件才回来。
Axiomate 当前基础有 AXIOMATE.md loader、memory file 统计、Read/Edit 工具链和 analyzeContext 基础，但没有产品化的 .axiomate/rules/、frontmatter glob、loaded reason 和 compact survival 标记。
Axiomate 要实现.axiomate/rules/ 递归扫描、YAML 解析、glob matcher、Read 触发注入、InstructionsLoaded debug 事件、/memory 展示“为何加载”、/context 展示 token 与 survival。
验收例子用户读取 src/api/user.ts 后，界面显示“Loaded .axiomate/rules/api.md because path matched src/api/**/*.ts”。运行 /compact 前能看到该规则属于 lazy reload，而不是永久保留。

2. CLAUDE.md 层级治理：把组织、团队、个人和子目录说明分清

Claude Code 文档把 memory 明确分成用户写的指令文件和 Claude 自动写的 auto memory；二者都只是 context，不是强制配置。

P0
解决什么问题同一个项目里会同时存在企业安全要求、用户个人偏好、团队 build/test 规范、当前机器的私有 URL、子目录专用说明。如果 loader 不说明来源和优先级，用户无法判断模型到底看到了哪份指令。
用户怎么用组织策略放 managed policy 位置；个人偏好放 ~/.claude/CLAUDE.md；团队规范放 ./CLAUDE.md 或 ./.claude/CLAUDE.md；本机私有说明放 CLAUDE.local.md；共享片段用 @path import。
什么时候用团队需要共享规范但不共享个人配置；monorepo 的祖先目录有无关 CLAUDE.md；多个 worktree 需要同一项目规范；本地测试 URL/账号不能提交。
运行时发生什么：启动按 scope 顺序加载上层目录中的 CLAUDE.md / local 文件；子目录文件不是启动全量加载，而是 Claude 读取子目录文件时再按需加载。@path import 会在启动时展开，因此它帮助组织文件，但不节省启动 context。claudeMdExcludes 用于跳过不相关祖先或团队文件。
Axiomate 当前基础有 agent/src/utils/axiomatemd.ts 与 memory file 展示，但报告按 2.1.88 基线看待，缺的是新版文档中更清晰的 source/scope/exclude/import 审计。
Axiomate 要实现AXIOMATE.md 等价层级、local 文件、import 深度限制、exclude glob、trust-aware custom directory、/memory source tree、冲突检测和“指令只是 context，不是 guardrail”的提示。
验收例子/memory 能列出 user/project/local/subdirectory/imported，每项显示路径、scope、是否从启动加载、是否懒加载、token 估算和最近修改时间。

3. Auto memory：模型自己写的长期项目笔记，不是用户手写规则

官方文档强调 auto memory 默认开启，按 git repo 共享；MEMORY.md 是索引，topic files 是详细内容。

P0
解决什么问题用户不断纠正“这个项目用 pnpm”“API 测试要 Redis”“不要再改 generated 文件”，这些事实如果只留在 transcript，下次 session 会丢。Auto memory 让模型把未来可复用的经验写成 markdown。
用户怎么用直接对模型说“remember that ...”；用 /memory 打开 memory folder 查看/编辑；用 autoMemoryEnabled 或环境变量关闭；用 autoMemoryDirectory 改存储位置。
什么时候用长期项目、跨 session 调试经验、常见测试命令、用户风格偏好、模型第二次犯同一个错误时。
运行时发生什么：每个项目有独立 memory 目录；同一 git repo 的 worktree 和子目录共享。启动时只加载 MEMORY.md 的前 200 行或 25KB，以先到者为准；topic files 不启动加载，而是在模型需要细节时用普通文件工具读取。模型可在会话中读写 memory 文件，界面会显示 writing/recalled memory。
Axiomate 当前基础agent/src/memdir/*、extractMemories、SessionMemory 已有 2.1.88 风格 filesystem memory。问题不是没有存储，而是缺少写入 provenance、大小阈值 UI、撤回和 stale/contradiction 标记。
Axiomate 要实现明确 200 lines / 25KB 行为、topic split 建议、memory 写入来源、session id、agent id、风险标签、memory doctor、手动/自动写入开关和按路径打开编辑。
验收例子当 MEMORY.md 接近阈值，/memory 提醒“索引过长，建议 dream/prune”；/context 显示 auto memory startup token 与 topic files 未加载状态。

4. /memory audit：证明模型到底看到了哪些记忆和规则

新版 Claude 文档把 /memory 写成排查入口，而不只是“打开文件”的快捷方式。

P0
解决什么问题用户说“我明明写了规则，模型为什么没遵守”。这时最需要的不是再提示模型，而是可视化确认规则是否加载、来自哪个路径、是否被 exclude、是否只在读匹配文件后才加载。
用户怎么用运行 /memory，查看当前 session 加载的 CLAUDE.md、CLAUDE.local.md、rules 和 auto memory；选中文件打开编辑；切换 auto memory。
什么时候用模型不听规则、怀疑 memory 过期、想知道 auto memory 写了什么、调试 path-scoped rule 触发时。
运行时发生什么：loader 维护 loaded source registry，记录 source path、scope、加载时机、触发原因、token 估算。/memory 读取这个 registry，而不是重新猜测文件系统。与 InstructionsLoaded hook 配合时，可以把“何时加载、为何加载”写入 debug log。
Axiomate 当前基础已有 memory selector 和 auto-dream 状态的部分 UI，但缺少完整 loaded-source registry 和 reason 字段。
Axiomate 要实现/memory 列表分组：instructions、rules、auto memory index、loaded topic files、subagent memory、dream outputs；每项支持 open/edit、copy path、disable/exclude、查看来源。
验收例子用户能在一个屏幕回答：这条规则是启动加载、按路径加载、import 进来、还是根本没加载。

5. /context doctor：上下文为什么满、哪些内容最贵

这是下一轮 Axiomate 最基础的仪表盘：没有它，rules、memory、tool search、compact 的效果都不可解释。

P0
解决什么问题长会话变慢、auto compact 反复触发、MCP 很多、skills 很多、文件读太多时，用户需要知道 token 被谁占了，而不是只看到“context 快满了”。
用户怎么用运行 /context，看 system prompt、memory、rules、skills、MCP tools、conversation、file reads、tool results、attachments 等 bucket 的估算与建议。
什么时候用开始大任务前、compact 前、模型变慢/贵、路由模型 contextWindow 或 maxOutputTokens 配置变化后。
运行时发生什么：系统用 token counter 或估算器对 system prompt、工具 schema、loaded memory、agent definitions、skill frontmatter、消息历史和工具结果分桶计数。它还结合 route 的 contextWindow / maxOutputTokens 计算有效窗口，并标注哪些 bucket 会在 compact 后重注入、哪些只是历史的一部分。
Axiomate 当前基础agent/src/utils/analyzeContext.ts 已能分桶 system/tools/MCP/agents/memory/skills/messages；ContextVisualization.tsx 已有可视化。但 DEV-only 细节多、用户级建议不足、缺 survival/cache/source 语义。
Axiomate 要实现生产级 /context：source provenance、loaded reason、compact survival、cache impact、route limits、top tool results、top file reads、优化建议和 link 到 /memory//mcp//skills。
验收例子用户看到“MCP tools deferred saved 38k tokens”“tool result auth.log 占 22k，建议 compact 或 move to file reference”“path rule will reload only after next Read”。

6. Compaction survival map：compact 后哪些内容还在

Claude 文档已经把 compact survival 明确成用户模型；Axiomate 应该在 compact 前后显示这张地图。

P0
解决什么问题用户 compact 后经常以为“规则丢了”或“模型忘了”。真实情况是不同来源的 context 生命周期不同：有些重注入，有些被总结，有些需要下次触发。
用户怎么用在 /context 或 /compact 确认页查看每类 context 的 survival：re-injected、summarized、lazy reload、dropped。

Axiomate 依赖glob/policy deny、read reason logging、LSP/code map、context doctor 的 file-read bucket。

13. Hooks 作为 context producer，需要预算和审计

Hook 输出进入 context 的路径必须显式化，否则会成为不可见 token 泄漏。

P1
是什么hook JSON 的 hookSpecificOutput.additionalContext 进入 Claude context；plain stdout exit 0 只进 debug log。
怎么用只把必要事实作为 additionalContext 输出，例如 formatter 结果、test failure 摘要、InstructionsLoaded 调试信息。
什么时候用tool 后需要告诉模型环境变化、session start 需要 reload skills/title、pre-compact 需要记录边界信息。
Axiomate 依赖hook parser、additionalContext token cap、source attribution、debug log link、security/trust gate。

14. Dream / memory consolidation 是独立重点

Dream 不是普通 summary，也不是 RAG 检索；它是长期 memory 的离线整理、冲突修复和可审计晋升机制。

P0
是什么官方 Managed Agents dreams 是异步 job：读取现有 memory store 和 1-100 个 session transcripts，生成独立 output memory store，合并重复、替换矛盾/过期项、提炼新模式。
怎么用指定 input store、sessions、model 和 instructions；跟踪 pending/running/completed/failed/canceled；完成后 review output store，再 attach、archive 或 discard。
什么时候用memory index 膨胀、topic files 重复、旧事实被新代码推翻、用户纠正反复出现、多个 agent/session 暴露同一类经验时。
Axiomate 依赖非破坏式 output store、session batch selector、diff/report、memory version/audit、manual promote、rollback/redact、usage/cost 显示。
Dream Deep Dive

Dream 不是附属功能，而是 memory 系统的整理层

官方语义： Claude Managed Agents 的 dream 是 research preview。它读取一个既有 memory store 和 1 到 100 个历史 sessions，异步运行整理管线，并产出一个新的 output memory store。input store 不被原地修改，因此开发者可以先 review，再决定是否用新 store 替换旧 store、并行 attach，或直接丢弃。
Axiomate 语义转换： Axiomate 不需要照搬云端 Managed Agents API，但应该照搬 dream 的产品边界：输入不可变、输出独立、运行可观察、失败可检查、晋升需要 review。也就是说，本地实现仍然是 markdown/filesystem store，但状态机和审计方式要接近官方 dream。

Dream 解决的问题

  • 重复：同一事实在多个 topic files 或多次 session 中被反复写入。
  • 矛盾：旧调试结论、旧路径、旧命令被新代码或新配置推翻。
  • 过期：相对日期、临时 workaround、一次性失败原因在后续 session 中不再成立。
  • 信号稀释：MEMORY.md 变成长列表，topic files 变成日志 dump。
  • 跨 session 模式：单次 session 看不出来的用户偏好、项目惯例、常见错误，需要从一批 transcript 里提炼。

官方管线要点

  • 输入：一个 pre-existing memory store，加 1-100 个 sessions。
  • 输出：另一个 memory store，独立于 input。
  • 状态：pending、running、completed、failed、canceled。
  • 可观察：running 后有底层 session_id，可以 stream events 看它读写了什么。
  • 失败/取消：partial output store 会保留，供检查后清理。

Axiomate 当前基础

  • agent/src/services/autoDream/autoDream.ts 已有后台触发、lock、progress watcher、cache usage log。
  • consolidationPrompt.ts 已有四阶段：Orient、Gather recent signal、Consolidate、Prune and index。
  • DreamTask.ts 和 background task dialog 已能展示 dream task、turns、files touched、kill。
  • /memory UI 已有 Auto-dream toggle 和 last ran/running 状态。
  • 这说明 Axiomate 不是从零开始；缺的是官方 dream 的产品化边界。

Missing Layers

Axiomate autoDream 目前缺的关键层

缺口 为什么重要 Claude / Managed Agents 参考语义 Axiomate 应该怎么补
非破坏式 output store 当前 dream 如果直接改 live memory，错误整理会污染之后所有 session；用户很难知道哪些事实被悄悄改掉。 官方 dream 永不改 input store，而是生成独立 output memory store。 本地创建 memory/.dreams/<dream-id>/output/ 或 sibling store；所有整理先写候选目录，不直接覆盖 live memory。
Review / promote / discard Dream 的价值在整理，但风险在误删、误合并、误判断 contradiction；必须有人类可审计入口。 官方建议 review output store 后 attach 或 discard。 新增 /dream review：展示文件级 diff、index diff、删除/合并/新增原因；支持 promote、archive、discard。
Memory version/audit 长期 memory 最危险的是不可追溯：谁写的、来自哪个 session、为什么改、能否回滚。 Managed Agents memory 每次 mutation 都有 immutable memory version，可审计和 point-in-time recovery。 为 markdown store 增加 .memory-versions/ 或 manifest：session id、agent id、source transcript、operation、old/new sha、timestamp、reason。
Session batch selection 梦到什么取决于读了哪些 transcript。全量扫会贵、慢、噪音大；扫少了又看不出跨 session 模式。 官方 dream 显式接收 1-100 个 sessions。 按 last dream timestamp、touched memory files、goal tags、project branch、error keywords 选 session batch，并在 report 中列出选择理由。
Partial output preservation 长 dream 失败不应只留下“失败”；部分输出可能包含有价值发现，也可能暴露 bug。 官方 failed/canceled 会保留 output store 的已写内容。 失败时保留 .dreams/<id>/output/、transcript、usage、last phase、错误类型；UI 提供 inspect / delete。
Cost / cache / token accounting Dream 是后台任务，用户更需要知道它为什么跑、读了多少、花了多少、是否值得。 官方 dream resource 有 usage；成本随 sessions 数量和长度近似线性增长。 Dream report 显示 input tokens、cache read/create、output tokens、读取文件数、session 数、wall time、是否命中节流。
Safety and trust 如果 transcript 里有 prompt injection 或 untrusted web/tool output，dream 可能把恶意内容升级成长期 memory。 Managed Agents memory 支持 read_only/read_write store，并提醒 untrusted input 可污染 writable memory。 对候选 memory 打标签：untrusted-source、user-confirmed、tool-output-derived、secret-like、policy-sensitive；promote 前提醒。
Recommended Local Design

Axiomate 本地 dream 设计建议

目录与数据模型

  • memory/：当前 live store，仍是 markdown，不引入默认 RAG。
  • memory/.dreams/<dream-id>/input-manifest.json：记录 input store sha、sessions、instructions、model、trigger。
  • memory/.dreams/<dream-id>/output/：候选整理后的完整 store。
  • memory/.dreams/<dream-id>/report.md：新增/修改/删除/合并/矛盾修复列表。
  • memory/.dreams/<dream-id>/audit.jsonl：每个文件操作的 old sha、new sha、source transcript、reason。

运行流程

  • Plan：选择 session batch，读取 MEMORY.md 与 topic files，生成 dream plan。
  • Consolidate：写入 output store，而不是 live store。
  • Validate：检查 index 行数/大小、死链、重复标题、过长 topic、矛盾标记、secret-like 内容。
  • Report：生成 human-readable diff 和 machine-readable audit。
  • Promote：用户确认后原子替换 live store，并保留 rollback snapshot。

候选 report 字段

  • summary：这次整理的总判断。
  • inputs：memory root、base sha、session ids、选择理由。
  • changes：added/modified/deleted/merged/replaced。
  • evidence：每项改动引用 transcript id 和 message/time。
  • risk：secret-like、untrusted、low-confidence。
  • context_effect：启动 index token 增减、topic files 数量变化。

失败/取消处理

  • 失败不回滚 live memory，因为 live memory 根本不应被写。
  • 保留 output store、partial report、usage、错误类型和最后阶段。
  • 用户可以 inspect partial output，再决定 delete/archive。
  • 如果输入 session 或 memory 在运行中消失，dream 标记 failed，report 说明 unavailable input。
  • 取消 running dream 后进入 canceled，已写 output 保留供检查。

自动调度边界

  • Auto-dream 只能创建候选 output store，不能直接 promote。
  • 默认按时间 + session 数 gate 触发，继续保留用户可关闭开关。
  • 触发原因要写进 report：超过阈值、memory 太大、重复度高、上次 dream 后 session 数达到阈值。
  • 高风险结果只提示 review，不自动晋升。
  • 后台任务必须显示 usage，避免用户不知道后台花费。

/dream 命令面

  • /dream run [--since last|7d|N sessions]
  • /dream status
  • /dream review <id>
  • /dream promote <id>
  • /dream discard <id>
  • /dream archive <id>

/memory 集成

  • Auto-dream toggle 保留。
  • 显示 last dream、running dream、pending review 数量。
  • Memory file selector 增加 “open latest dream report”。
  • 当 MEMORY.md 超过阈值时提示运行 dream。

/context 集成

  • 显示 dream 是否会读取 transcripts。
  • 显示 memory index 是否过大、topic 是否过碎。
  • 显示哪些 memory 是 dream 产物。
  • 显示 promoted dream 对 context startup token 的影响。
优先级修正： Dream 不应排到 P2。正确顺序是：/memory audit、memory version/audit、dream 非破坏式 output store、dream review/promote/discard 同属 P0；自动调度、复杂评分、跨 agent shared dream 可以放到 P1/P2。
